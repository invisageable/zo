# zo-binder.

Turn a C-ABI Rust shim crate into the `provider/<lib>/<lib>.zo` foreign-function (FFI) declarations the zo compiler consumes. Write the Rust functions once; zo-binder writes the matching `pub ffi` declarations, so the two never drift.

  ```
  Rust shim (src/lib.rs)            zo-binder              provider/<lib>/<lib>.zo
  ─────────────────────   ───────────────────────────   ───────────────────────
  #[no_mangle]            parse  → extract C-ABI fns     #link { ... }
  pub extern "C" fn  ───► tymap  → Rust types → zo  ───► pub ffi <name>(...) -> ...;
    open(...) -> i64      emit   → render the .zo
                          bind   → orchestrate
  ```

## why.

zo drives an AArch64 foreign call straight from a `pub ffi` declaration's type signature, so adding a C function costs one line of zo and no compiler changes (see the design docs below). Writing those declarations by hand, and keeping them matched to the Rust shim, was the last manual step. zo-binder removes it.

zo-binder runs ahead of time, outside `zo run`. Our manifesto require that:

  - **Velocity.** No `cargo build`, no toolchain probe, no hashing on the compile-to-run path. You run the generator when you choose.
  - **Pragmatism / no magic.** The generated `.zo` is a committed, reviewable file, the same as any other source file. No hidden cache, no in-compiler generation.

## design & spec.

No single paper backs zo-binder. It implements **signature-driven binding generation**: read a foreign function's signature, map its types, emit a native declaration. Odin (`foreign import`) and Zig (`@cImport`) take the same approach.

Three specs carry the weight:

  - **AArch64 AAPCS64**, the procedure-call application binary interface (ABI). zo-binder writes the `pub ffi` declarations; the ABI-driven codegen makes the call. zo-binder never touches the ABI itself.
  - **`PLAN_AAPCS_FFI_FROM_SIGNATURE.md`**, the consumption half (shipped). Codegen drives the call from the signature; the linker routes the dylib from `#link` metadata.
  - **`PLAN_ZO_BINDER.md`**, the generation half: scope, type table, and phase plan.

Both plans live in `crates/compiler/zo-notes/personal/architecture`.

## how it works.

zo-binder is a small pure pipeline (a library) plus a thin command-line interface:

| Module      | Responsibility                                                                        |
| :---------- | :------------------------------------------------------------------------------------ |
| `parse.rs`  | `syn`-parse the shim; keep `extern "C"` + `#[no_mangle]` fns; resolve `type` aliases. |
| `tymap.rs`  | Map each Rust type to a zo type; reject an unsupported type by its spelling.          |
| `emit.rs`   | Render the `.zo` text (`Emitter` over `zo_buffer::Buffer`).                           |
| `bind.rs`   | Orchestrate `parse → tymap → emit`.                                                   |
| `main.rs`   | Locate the shim, render, then write only when the output changed.                     |

Three rules govern the output:

  - **Name passthrough.** The zo `pub ffi` name equals the exported C symbol, character for character — no prefix, no stripping. The shim author prevents cross-dylib symbol clashes (for example, a `<lib>_` name prefix on each function).
  - **Alias resolution.** zo-binder follows `type ZoHandle = i64;` transitively, with a depth guard, so the type mapper sees concrete types.
  - **Width-faithful.** A Rust `i64` maps to `s64`, never to `int` (s32), because the generator cannot assume a value fits in 32 bits.

## usage.

### 1. Write a C-ABI Rust shim.

Wrap the crate you want in a `crates/compiler/zo-provider-<lib>` cdylib (a C dynamic library) that exports plain C functions:

  ```rust
  // crates/compiler/zo-provider-undoredo/src/lib.rs

  type ZoHandle = i64;

  /// Create a new undo stack.
  #[unsafe(no_mangle)]
  pub extern "C" fn undo_stack_new() -> ZoHandle {
    /* ... */
  }

  #[unsafe(no_mangle)]
  pub extern "C" fn undo_stack_push(stack: ZoHandle, value: i64) {
    /* ... */
  }
  ```

### 2. generate the bindings.

  ```sh
  just bind undoredo
  # OR
  cargo run -p zo-binder -- undoredo
  ```

zo-binder reads `zo-provider-undoredo/src/lib.rs` and writes `crates/compiler-lib/provider/undoredo/undoredo.zo`:

  ```zo
  -! undoredo bindings. Generated by zo-binder — do not edit
  -! by hand; edit the shim and re-run `just bind undoredo`.

  #link {
    macos: "@executable_path/libzo_provider_undoredo.dylib",
    linux: "@executable_path/libzo_provider_undoredo.so",
  }

  -! Create a new undo stack.
  pub ffi undo_stack_new() -> s64;

  pub ffi undo_stack_push(stack: s64, value: s64);
  ```

Re-running without a Rust change rewrites nothing.

### 3. use it from zo.

  ```zo
  load provider::undoredo::*;

  fun main() {
    imu stack: s64 = undo_stack_new();
    undo_stack_push(stack, 42);
  }
  ```

Build the shim's cdylib so the linker resolves the symbols. The compiler stages `libzo_provider_undoredo.dylib` next to your binary.

## type mapping.

| Rust (in the shim)            | zo                             | Notes                                  |
| :---------------------------- | :----------------------------- | :------------------------------------- |
| `i8` / `i16` / `i32` / `i64`  | `s8` / `s16` / `int` / `s64`   | `int` is s32                           |
| `isize`                       | `s64`                          | pointer-width signed                   |
| `u8` / `u16` / `u32` / `u64`  | `u8` / `u16` / `uint` / `u64`  | `uint` is u32                          |
| `usize`                       | `u64`                          |                                        |
| `f32` / `f64`                 | `float` / `f64`                | `float` is f32                         |
| `bool`                        | `bool`                         |                                        |
| `()` / no return              | (omit)                         | no `-> T` clause                       |
| `*const c_char`               | `CStr`                         | adds `load core::c::*;`                |
| `*const T` / `*mut T`         | `s64`                          | opaque pointer / handle                |

zo-binder rejects anything outside the table (`&str`, `String`, `Vec<_>`, references, a struct passed by value) and names the offending Rust type.

## limitations.

  - **Rust shims only.** Parsing raw C headers (system raylib, say) is a separate, deferred front-end on the same back-end.
  - **No struct generation.** A `#[repr(C)]` struct passed by value maps to a zo `struct` that must already exist in `provider/<lib>/`.
  - **C-string returns.** A bare `const char *` return carries no length, so it maps to `s64`; wrap it by hand (`CBytes` / `.to_str()`).
  - **Docs pass through unchanged.** Rust `///` comments become `-!` lines as written, Rust-specific sections included.
