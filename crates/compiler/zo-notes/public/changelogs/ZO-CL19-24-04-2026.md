# ZO-CL19 ‚Äî 2026-04-24

## PLAN_CONCURRENCY_CODEGEN follow-up.

CL18 landed the PLAN_PREHISTORY runtime + the first
end-to-end spawn + nursery path. CL19 closes the gap
between SIR-level concurrency and a compiled binary
that actually runs channels: producer / consumer
programs compile through `zo build` and execute
correctly from the binary.

`PLAN_CONCURRENCY_CODEGEN.md` lays out 4 phases.
Phases 1 and 2 land in this CL.

---

## Phase 1 ‚Äî channel primitives end-to-end.

A program that `channel()`s, spawns a producer that
`.send(v)`s, spawns a consumer that `.recv()`s, and
prints the received value now runs from the compiled
binary.

### what lands

- `size_of_ty(TyId) -> u32` helper in codegen ‚Äî
  resolves any TyId to its ABI byte width (int=4,
  bool=1, pointer-backed=8, etc.) from the
  tychecker's canonical registration order.
- `ChannelCreate` ‚Äî single-dst refactor. Old
  `{ tx, rx, ... }` produced two aliased ValueIds
  but reg-alloc tracks one dst per insn, so `rx`
  had no register and `TupleLiteral` wrote
  garbage. Now `{ dst, ... }`; the executor's
  subsequent `TupleLiteral` aliases the one
  handle into the `(Tx<T>, Rx<T>)` pair.
- Codegen channel arg marshaling:
  - `ChannelCreate`: X0 = elem_sz, X1 = capacity,
    `BL _zo_chan_new`.
  - `ChannelSend`: value spilled to a 16-byte
    per-function scratch slot; X0 = chan,
    X1 = scratch address, `BL _zo_chan_send`.
  - `ChannelRecv`: runtime writes to scratch slot,
    codegen loads the result back into the dst
    register.
- `FunctionInfo::chan_scratch_size` + matching
  prologue / epilogue math in codegen reserve the
  16-byte scratch slot in the frame of any function
  using channels.
- `has_calls` recognition extended to cover every
  concurrency insn that emits a `BL`
  (`ChannelCreate / Send / Recv`, `TaskSpawn`,
  `TaskAwait`, `NurseryEnd`, `SelectWait`).
  Without this, concurrency-using functions got
  leaf prologues ‚Äî caller-save STRs clobbered the
  return address and SIGBUS'd on exit.
- Codegen frame epilogue includes
  `chan_scratch_size` alongside the other frame
  components so SP restores correctly.
- N-ary `TaskSpawn` ‚Äî runtime exports
  `_zo_task_spawn_1 / _2 / _3` plus matching
  `task_shim_1 / 2 / 3` that unpack
  `user_arg0..2` from the `ZoTask` before jumping
  to `user_entry`. Codegen selects the variant
  from `args.len()`; callee address lands in X0,
  args in X1..X3.
- Liveness `insn_uses` + `compute_value_ids`
  enumerate every concurrency operand and dst so
  the defining `TupleIndex` / `Load` / etc. survive
  DCE. Without this, `ch.0` loads were pruned and
  the runtime read garbage.
- DCE `is_impure` marks all concurrency insns as
  side-effecting. `collect_calls_in_range` treats
  `TaskSpawn.callee` as reachable so the spawned
  function survives DCE all the way to codegen's
  `function_addr_fixups`.
- SIR `emit()` returns `dst` for every concurrency
  value-producing insn (ChannelCreate, ChannelRecv,
  TaskSpawn, TaskAwait).

### test

- `concurrency-channel-roundtrip.zo` ‚Äî two green
  tasks spawned under a nursery exchange `42`
  through a `channel()`, consumer prints `got 42`.
  Runs end-to-end in `just zo_test_runner`.

## Phase 2 ‚Äî `tx.close()` / `rx.close()` surface.

Either half of a channel pair can close it; runtime
wakes every parked sender and receiver so they
observe the closed state on their next loop.

### what lands

- `Insn::ChannelClose { channel }` ‚Äî single-operand,
  non-value-producing SIR insn.
- SIR visitor wiring: `visit_value_ids_mut` walks
  the channel operand; `visit_ty_ids_mut` no-op
  (no ty); `emit()` returns sentinel;
  `compute_value_ids` returns None.
- Liveness `insn_uses` reports the channel operand
  so the defining Load / TupleIndex stays alive.
- DCE `is_impure` marks `ChannelClose`.
- Validator ‚Äî `ChannelClose` joins the no-ty-id
  arm.
- Pretty-printer ‚Äî `chan.close %N`.
- Executor:
  - `is_dot_method_call` recognizes `close` on
    `ChannelTx` / `ChannelRx` so the Dot handler
    doesn't treat it as field access.
  - `execute_potential_call` routes `.close()` to
    new `execute_channel_close`, which pops the
    receiver from the three stacks and emits the
    `ChannelClose` insn.
- Codegen-arm ‚Äî `Insn::ChannelClose` arm, X0 =
  channel handle, `BL _zo_chan_close`.
- Register-allocation `has_calls` and compiler's
  runtime-staging detection include `ChannelClose`.

### test

- `concurrency-channel-close.zo` ‚Äî producer sends
  1 + 2 + closes; consumer receives both and
  prints `got 1` / `got 2`.

### deferred

`rx.recv_timeout(ms)` surface syntax. Needs
`Option<T>` wrapping (scope creep) or a bool-return
MVP that doesn't fit the current method-dispatch
shape without wider changes. The runtime primitive
(`_zo_chan_recv_timeout`) stays available; surface
syntax lands when a concrete use case demands it.

## Phase 3 ‚Äî `select { arm, arm }` end-to-end.

`select { chan_expr => fn(p: T) => body, ... }` now
compiles to a binary that atomically waits on N
channels, routes control to the fired arm, and binds
the recv'd value to the closure parameter.

### what lands

- `Insn::SelectRecv { dst, which, ty_id, chans_len }`
  ‚Äî companion to `SelectWait`. Split out so the
  allocator sees a single-dst insn per SIR entry;
  `SelectWait` produces `out_which`, `SelectRecv`
  produces the received value by LDRing the scratch
  buffer the runtime wrote into. `which` anchors the
  uses ‚áí the preceding `SelectWait` can't be
  reordered or DCE'd.
- `FunctionInfo::select_scratch_size` + matching
  prologue / epilogue math ‚Äî `nchans * 8` bytes of
  `*mut ZoChan` pointers immediately followed by an
  `elem_sz` output buffer, sized at the worst-case
  select in the function and 16-byte aligned.
- Codegen `SelectWait` arm ‚Äî spills each chan
  operand into the on-stack array, zeros the out
  buffer (so LDR of untouched upper bytes reads
  zero), loads X0 = chans_ptr, X1 = nchans,
  X2 = out_ptr, X3 = elem_sz, `BL _zo_select_wait`.
- Codegen `SelectRecv` arm ‚Äî LDRs the received
  value from `select_scratch_base + chans_len * 8`
  into the allocator-assigned `dst` register.
- Executor `execute_select` rewrite ‚Äî manually walks
  the select body (bypassing main-loop arm scan),
  splits arms by top-level `FatArrow` / `Comma`,
  evaluates each chan expression through
  `execute_node` to harvest its `ValueId`, emits
  `SelectWait` + `SelectRecv`, then per arm emits
  `ConstInt(i) + BinOp Eq + BranchIfNot + body +
  Jump(end) + Label(next)`. Derives `elem_ty` from
  the first `Rx<T>` / `Tx<T>` it meets.
- `execute_select_arm_body` ‚Äî closure-arm form
  (`fn(p: T) => expr`) is the established surface;
  the executor extracts the param name from the
  `Fn` subtree, emits `Store { name, value: out_value,
  ty_id }` to allocate a frame slot mirror of the
  select's out buffer, and registers a `Local` so the
  body's `Ident(param)` emits a `Load` that lowers
  through codegen's `mutable_slots` table. A direct-
  expression fallback (no `fn` wrapper) is also
  accepted.
- `_zo_select_wait` runtime ‚Äî when called from non-
  task code (e.g. `main` before the nursery drain)
  now drains one ready task per iteration via
  `scheduler::pop_local` + `run_one_external`.
  Without this, producers spawned inside `main`'s
  nursery never run before `select` polled them, and
  the select loop deadlocked on an empty queue.
- Liveness `compute_value_ids` / `insn_uses` wired
  for both `SelectWait` and `SelectRecv`. DCE
  `is_impure` marks both.
- SIR pp / validator cover the new variant.

### task return-value plumbing (unblocks
`concurrency-spawn-await` regression).

The test added in `f930fd62` was checked in without
the runtime half: `_zo_task_await` returned void and
the task's shims transmuted the user callee as
`fn()` (discarding the real X0 result). `await task`
therefore returned whatever junk sat in X0 after the
call, and `imu result = await task` bound to garbage.

Landed here:

- `ZoTask::ret_value: u64` ‚Äî filled by the shim on
  the `Completed` transition, read by
  `_zo_task_await` before the `Box` reclaim.
- `task_shim / _1 / _2 / _3` transmute the callee as
  `fn(..) -> u64`; on arm64 reading X0 as `u64` is
  safe for every return width the ABI emits.
- `_zo_task_await(task) -> u64` ‚Äî returns the
  captured value through X0; codegen's existing
  `emit_mov_reg(dst, X0)` after the `BL` stages it
  into the allocator-assigned `dst`.

### test

- `concurrency-select-pick.zo` ‚Äî two channels, two
  producers, `select` picks the first to fire and
  prints `a: 10`. Runs end-to-end.

## Phase 4 ‚Äî `t.cancel()` / `t.cancelled()` on `Task<T>`.

Cooperative cancellation surface. Mirrors the `Tx/Rx`
method shape ‚Äî the handle carries the type, methods
attach to it.

### design note.

`PLAN_CONCURRENCY_CODEGEN.md` originally spec'd
`task::cancelled()` as a magic identifier with a
`_zo_task_current() -> *mut ZoTask` runtime hop. That
broke the established pattern (`.send/.recv/.close`
attach to `Tx<T>/Rx<T>`, not to a global `channel`
keyword). Reworked to `t.cancel()` / `t.cancelled()`
methods on `Task<T>` ‚Äî consistent with every other
handle in the language, no reserved `task` keyword,
one fewer runtime call. Dropped the
`_zo_task_current` export entirely.

### what lands

- `Insn::TaskCancelled { dst, task, ty_id }` ‚Äî reader.
  Returns `bool`. Lowered to `BL _zo_task_is_cancelled`
  with the task handle in X0.
- `Insn::TaskCancel { task }` ‚Äî writer. No dst.
  Lowered to `BL _zo_task_cancel` with the handle in
  X0.
- `is_dot_method_call` recognizes `cancel` / `cancelled`
  on `Ty::Task(_)` so the Dot handler routes through
  the method-call path instead of field access.
- Executor `execute_task_cancel` / `execute_task_cancelled`
  pop the receiver from the three stacks and emit the
  matching insn (plus the bool result for
  `cancelled`). Dispatched from the postfix dot-call
  branch in `execute_potential_call` ‚Äî
  `[recv, method, ., (, )]` ‚Äî next to the existing
  `send` / `recv` cases.
- Liveness / DCE / pp / validator / codegen arms for
  both variants. Register allocator's `has_calls`
  detection includes both.
- Runtime: no new exports. Both
  `_zo_task_cancel(task)` and
  `_zo_task_is_cancelled(task)` were already present
  for internal use; Phase 4 just exposes them to the
  surface.

### test

- `concurrency-task-cancel.zo` ‚Äî parent cancels a
  spawned worker, then reads back the flag. Prints
  `cancelled: true`.
