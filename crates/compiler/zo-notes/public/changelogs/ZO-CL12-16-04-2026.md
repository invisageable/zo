# ZO-CL12 — 16-04-2026.

## feature — abstract definition + apply abstract for type.

`abstract` defines a contract (method signatures, no bodies).
`apply Abstract for Type { ... }` implements the contract.

```zo
abstract Show {
  fun show(self) -> str;
}

apply Show for Point {
  fun show(self) -> str {
    return "point";
  }
}
```

### parser (zo-parser).

- handle `Token::Abstract` — emit node, push Introducer,
  close at RBrace (added to Enum/Struct/Apply close list)
- `fun` inside abstract treated as `Ext` (no body, ends
  with `;`) — detected via introducer stack
- `Token::For` inside apply context emits as plain token
  (not for-loop) — detected via introducer stack

### executor (zo-executor).

- `execute_abstract` — parses method signatures (name,
  params with self, return type), registers in
  `abstract_defs: HashMap<Symbol, AbstractDef>`
- `execute_apply` — detects `for` pattern by scanning for
  `Token::For` between first Ident and LBrace. when present:
  first Ident = abstract name, Ident after for = target type.
  `apply_context` set to target type (methods mangle as
  `Type::method`, not `Abstract::method`)
- registers impl in `abstract_impls: HashMap<(Symbol, Symbol),
  Vec<Symbol>>`

### method dispatch.

already works — `p.show()` resolves to `Point::show(p)`
via mangled name lookup in `self.funs`. no vtable, no
dynamic dispatch, compile-time resolution.

### blocker — struct methods returning str.

`apply Point { fun show(self) -> str { return "point"; } }`
returns garbage instead of the string. this is a pre-existing
codegen bug affecting ALL struct methods that return str,
not specific to abstract. `struct-stateful.zo` only tests
int returns. needs investigation before built-in Show works.

### files changed.

| file | change |
|------|--------|
| `zo-parser/src/parser.rs` | Token::Abstract, For-in-apply, fun-as-Ext in abstract, RBrace close |
| `zo-executor/src/executor.rs` | AbstractDef, AbstractMethod, execute_abstract, apply for detection, abstract_impls |

### verified with.

- `abstract Show {}` + `fun main() { showln("hello"); }` ✓
- `abstract Show { fun show(self) -> str; }` + main ✓
- `apply Show for Point { fun show(self) -> str { ... } }` ✓
  (method registered, dispatch works, string return blocked)
- 260/260 zo_test pass, zero regressions

## bugfix — apply for tree ordering.

`apply Show for Point { ... }` silently failed: `execute_apply`
returned early because the parser placed `Token::For` before
`Ident(Show)` in the tree (expression buffer not flushed).

**root cause:** parser's `Token::For` handler inside Apply
context emitted `For` directly without flushing the expression
buffer. `Show` was still buffered, so tree order was
`[Apply, For, Show, Point]` instead of `[Apply, Show, For, Point]`.

**fix (zo-parser):** add `self.flush_expr()` before emitting
`Token::For` in the Apply context.

**fix (zo-executor):** `execute_apply` now scans children for
the first `Token::Ident` instead of assuming `start_idx + 1`.
also scans past LBrace for `Token::For` since the parser may
place it inside the brace children.

## bugfix — method name shadowing by built-in functions.

`p.show()` called the built-in `show` instead of
`Point::show` when the method name matched a preloaded
function.

**root cause:** the dot-call detection at `execute_potential_call`
checked `fun_idx - 1 == Dot`, but the parser's tree order for
known function names is `[receiver, method, ., (]` where
`fun_idx - 1 = receiver` (Ident), not Dot. the check failed
and the call fell through to the bare built-in function.

**fix (zo-executor):** added a second dot-call detection path
using `lparen_idx - 1 == Dot` (tree-order independent). added
`resolve_dot_call_with_receiver` that takes the receiver index
directly (at `fun_idx - 1`) instead of computing `method_idx - 2`.

### files changed.

| file | change |
|------|--------|
| `zo-executor/src/executor.rs` | lparen-1 Dot detection, resolve_dot_call_with_receiver |

### verified with.

- `p.show()` returns `"point"` (not garbage) ✓
- `apply Show for Point { fun show(self) }` works ✓
- struct-stateful.zo still passes ✓
- 261/261 zo_test pass

## feature — built-in Show abstract + auto-dispatch.

`showln(struct_value)` auto-dispatches to `Type::show(value)`
when the struct implements `Show`.

```zo
apply Show for Point {
  fun show(self) -> str { return "Point"; }
}

showln(p);  -- calls Point::show(p), prints "Point"
```

### std/preload.zo.

added `Show` and `Eq` abstract definitions:
```zo
pub abstract Show { fun show(self) -> str; }
pub abstract Eq { fun eq(self, other: Self) -> bool; }
```

### executor (zo-executor).

in `execute_call`, when the call is `show`/`showln` and the
single argument has a struct type: look up `abstract_impls`
for `(Show, TypeName)`. if found, emit `Call Type::show(arg)`
first, then pass the returned string to showln.

| `zo-executor/src/executor.rs` | Show auto-dispatch in execute_call |

## feature — Eq abstract operator overloading.

`==` and `!=` on structs auto-dispatch to `Type::eq` when
the struct implements `Eq`.

```zo
apply Eq for Point {
  fun eq(self, other: Point) -> bool {
    return self.x == other.x;
  }
}

check(a == b);  -- calls Point::eq(a, b)
check(a != c);  -- calls !Point::eq(a, c)
```

### executor (zo-executor).

in `execute_binop`, when `BinOp::Eq` or `BinOp::Neq` and
the operand type is a struct with an Eq impl: emit
`Call Type::eq(lhs, rhs)` instead of `BinOp::Eq`. for
`Neq`, wrap the result with `UnOp::Not`.

### files changed.

| file | change |
|------|--------|
| `zo-executor/src/executor.rs` | Eq dispatch in execute_binop |
| `zo-tests/programming/abstract-eq.zo` | integration test |

## known issue — SIR duplication in monomorphization.

the `fn monomorphize` pass (executor.rs line 9196) clones
SIR instruction ranges for each generic instantiation. this
means N instantiations of `fun f<$T>` produce N copies of
the body in the SIR.

**impact:**
- code size grows linearly with instantiation count
- ValueIds in cloned bodies may collide (not offset)
- inner Call/BinOp instructions not rewritten (fixed for
  abstract methods in phase 6, but general case remains)

**proper fix:** either re-execute the function body per
instantiation (Carbon approach — aligns with zo's execution-
based compilation) or use indirect dispatch with a single
SIR body. both require architectural changes.

**status:** tracked. does not affect correctness for current
test suite. will become a problem at scale.

## feature — generic constraints (phase 6). DONE.

`fun are_equal<$T: Eq>(a: $T, b: $T) -> bool { a.eq(b) }`
— generic functions with abstract constraints. method calls
on generic types resolved at monomorphization time via
`__abstract::method` placeholder rewriting.

### parser (zo-parser).

- `$T: Colon Ident` parsed in generic params
- `fun` inside abstract closes on `;` (check any ancestor
  is Abstract, not just parent — the stack is
  `[Abstract, LBrace, Fun]`)

### executor (zo-executor).

- `is_dot_method_call`: `Ty::Infer` receivers check
  `abstract_defs` for matching method names
- `resolve_dot_call` + `resolve_dot_call_with_receiver`:
  emit `__abstract::method` placeholder for Infer types
- `execute_dot_method_call`: placeholder path pops explicit
  args first, then receiver (Bug 3 fix — was swapped)
- struct/enum type names in monomorphization mangling
  (was `"unknown"` — now resolves through ty_table)
- mangling uses substituted fresh vars (unified with
  concrete types), not original inference vars
- monomorphize: `offset_value_ids` on cloned SIR to avoid
  ValueId collisions
- monomorphize: rewrites `__abstract::method` → `Type::method`
  using type suffix from mangled function name

### abstract_defs propagation (Bug 1 fix — root cause).

preload's `abstract Show/Eq` definitions were not exported
to the user's Executor. `abstract_defs` was empty at
user compile time, causing `is_dot_method_call` to reject
generic method calls (Ty::Infer → no abstract match).

**fix:** propagated `abstract_defs` through the full chain:
- `Executor::execute()` returns `HashMap<Symbol, AbstractDef>`
- `SemanticResult` carries `abstract_defs`
- `ImportedSymbols` carries `abstract_defs`
- `Executor::with_imports` accepts and merges `abstract_defs`
- compiler collects from `mod_sem.abstract_defs` during preload
