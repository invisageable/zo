# ZO-CL12 — 16-04-2026.

## feature — abstract definition + apply abstract for type.

`abstract` defines a contract (method signatures, no bodies).
`apply Abstract for Type { ... }` implements the contract.

```zo
abstract Show {
  fun show(self) -> str;
}

apply Show for Point {
  fun show(self) -> str {
    return "point";
  }
}
```

### parser (zo-parser).

- handle `Token::Abstract` — emit node, push Introducer,
  close at RBrace (added to Enum/Struct/Apply close list)
- `fun` inside abstract treated as `Ext` (no body, ends
  with `;`) — detected via introducer stack
- `Token::For` inside apply context emits as plain token
  (not for-loop) — detected via introducer stack

### executor (zo-executor).

- `execute_abstract` — parses method signatures (name,
  params with self, return type), registers in
  `abstract_defs: HashMap<Symbol, AbstractDef>`
- `execute_apply` — detects `for` pattern by scanning for
  `Token::For` between first Ident and LBrace. when present:
  first Ident = abstract name, Ident after for = target type.
  `apply_context` set to target type (methods mangle as
  `Type::method`, not `Abstract::method`)
- registers impl in `abstract_impls: HashMap<(Symbol, Symbol),
  Vec<Symbol>>`

### method dispatch.

already works — `p.show()` resolves to `Point::show(p)`
via mangled name lookup in `self.funs`. no vtable, no
dynamic dispatch, compile-time resolution.

### blocker — struct methods returning str.

`apply Point { fun show(self) -> str { return "point"; } }`
returns garbage instead of the string. this is a pre-existing
codegen bug affecting ALL struct methods that return str,
not specific to abstract. `struct-stateful.zo` only tests
int returns. needs investigation before built-in Show works.

### files changed.

| file | change |
|------|--------|
| `zo-parser/src/parser.rs` | Token::Abstract, For-in-apply, fun-as-Ext in abstract, RBrace close |
| `zo-executor/src/executor.rs` | AbstractDef, AbstractMethod, execute_abstract, apply for detection, abstract_impls |

### verified with.

- `abstract Show {}` + `fun main() { showln("hello"); }` ✓
- `abstract Show { fun show(self) -> str; }` + main ✓
- `apply Show for Point { fun show(self) -> str { ... } }` ✓
  (method registered, dispatch works, string return blocked)
- 260/260 zo_test pass, zero regressions

## bugfix — apply for tree ordering.

`apply Show for Point { ... }` silently failed: `execute_apply`
returned early because the parser placed `Token::For` before
`Ident(Show)` in the tree (expression buffer not flushed).

**root cause:** parser's `Token::For` handler inside Apply
context emitted `For` directly without flushing the expression
buffer. `Show` was still buffered, so tree order was
`[Apply, For, Show, Point]` instead of `[Apply, Show, For, Point]`.

**fix (zo-parser):** add `self.flush_expr()` before emitting
`Token::For` in the Apply context.

**fix (zo-executor):** `execute_apply` now scans children for
the first `Token::Ident` instead of assuming `start_idx + 1`.
also scans past LBrace for `Token::For` since the parser may
place it inside the brace children.

## bugfix — method name shadowing by built-in functions.

`p.show()` called the built-in `show` instead of
`Point::show` when the method name matched a preloaded
function.

**root cause:** the dot-call detection at `execute_potential_call`
checked `fun_idx - 1 == Dot`, but the parser's tree order for
known function names is `[receiver, method, ., (]` where
`fun_idx - 1 = receiver` (Ident), not Dot. the check failed
and the call fell through to the bare built-in function.

**fix (zo-executor):** added a second dot-call detection path
using `lparen_idx - 1 == Dot` (tree-order independent). added
`resolve_dot_call_with_receiver` that takes the receiver index
directly (at `fun_idx - 1`) instead of computing `method_idx - 2`.

### files changed.

| file | change |
|------|--------|
| `zo-executor/src/executor.rs` | lparen-1 Dot detection, resolve_dot_call_with_receiver |

### verified with.

- `p.show()` returns `"point"` (not garbage) ✓
- `apply Show for Point { fun show(self) }` works ✓
- struct-stateful.zo still passes ✓

## feature — Eq abstract operator overloading.

`==` and `!=` on structs auto-dispatch to `Type::eq` when
the struct implements `Eq`.

```zo
apply Eq for Point {
  fun eq(self, other: Point) -> bool {
    return self.x == other.x;
  }
}

check(a == b);  -- calls Point::eq(a, b)
check(a != c);  -- calls !Point::eq(a, c)
```

### executor (zo-executor).

in `execute_binop`, when `BinOp::Eq` or `BinOp::Neq` and
the operand type is a struct with an Eq impl: emit
`Call Type::eq(lhs, rhs)` instead of `BinOp::Eq`. for
`Neq`, wrap the result with `UnOp::Not`.

### files changed.

| file | change |
|------|--------|
| `zo-executor/src/executor.rs` | Eq dispatch in execute_binop |
| `zo-tests/programming/abstract-eq.zo` | integration test |

## known issue — SIR duplication in monomorphization.

the `fn monomorphize` pass (executor.rs line 9196) clones
SIR instruction ranges for each generic instantiation. this
means N instantiations of `fun f<$T>` produce N copies of
the body in the SIR.

**impact:**
- code size grows linearly with instantiation count
- ValueIds in cloned bodies may collide (not offset)
- inner Call/BinOp instructions not rewritten (fixed for
  abstract methods in phase 6, but general case remains)

**proper fix:** either re-execute the function body per
instantiation (Carbon approach — aligns with zo's execution-
based compilation) or use indirect dispatch with a single
SIR body. both require architectural changes.

**status:** tracked. does not affect correctness for current
test suite. will become a problem at scale.

## feature — generic constraints (phase 6). DONE.

`fun are_equal<$T: Eq>(a: $T, b: $T) -> bool { a.eq(b) }`
— generic functions with abstract constraints. method calls
on generic types resolved at monomorphization time via
`__abstract::method` placeholder rewriting.

### parser (zo-parser).

- `$T: Colon Ident` parsed in generic params
- `fun` inside abstract closes on `;` (check any ancestor
  is Abstract, not just parent — the stack is
  `[Abstract, LBrace, Fun]`)

### executor (zo-executor).

- `is_dot_method_call`: `Ty::Infer` receivers check
  `abstract_defs` for matching method names
- `resolve_dot_call` + `resolve_dot_call_with_receiver`:
  emit `__abstract::method` placeholder for Infer types
- `execute_dot_method_call`: placeholder path pops explicit
  args first, then receiver (Bug 3 fix — was swapped)
- struct/enum type names in monomorphization mangling
  (was `"unknown"` — now resolves through ty_table)
- mangling uses substituted fresh vars (unified with
  concrete types), not original inference vars
- monomorphize: `offset_value_ids` on cloned SIR to avoid
  ValueId collisions
- monomorphize: rewrites `__abstract::method` → `Type::method`
  using type suffix from mangled function name

### abstract_defs propagation (Bug 1 fix — root cause).

preload's `abstract Show/Eq` definitions were not exported

**fix:** propagated `abstract_defs` through the full chain:
- `Executor::execute()` returns `HashMap<Symbol, AbstractDef>`
- `SemanticResult` carries `abstract_defs`
- `ImportedSymbols` carries `abstract_defs`
- `Executor::with_imports` accepts and merges `abstract_defs`
- compiler collects from `mod_sem.abstract_defs` during preload

## bugfix — render only `#dom`-targeted templates.

`065-zsx-nested-component.zo` rendered `"lorem ipsum."`
twice: once for the `nested` component declaration and
once for the `paragraph` that consumes it via `<nested />`.
declarations should never render — only `#dom <name>` acts.

**root cause (two-sided).**

1. executor (`execute_directive`): `#dom paragraph` ran the
   ident through the normal expression path, emitting an
   `Insn::Load` whose fresh `dst` `ValueId` landed in
   `Insn::Directive.value` — not the template's own id.
2. driver (`cmd/run.rs`): with no usable link from directive
   to template, the driver flattened **every** `Insn::Template`
   in the SIR into one render stream and only used the
   directive as a boolean flag (`has_dom_directive`). the
   `zsx-nested-component.zo` test had the same bug silently
   — its two identical renders just looked like one.

**fix (zo-executor).**

in `execute_directive`, short-circuit `#dom <ident>`: find
the Ident child, `lookup_local`, and store `local.value_id`
(which IS the template's `Insn::Template { id, .. }` id)
directly into `Insn::Directive.value`. no spurious `Load`,
no fresh runtime id — the directive now references the
template it actually targets.

**fix (zo-driver).**

two-pass SIR scan:
1. collect `Directive { name: "dom", value }` ids into
   `dom_targets: Vec<ValueId>`.
2. extract commands from `Insn::Template { id, commands, .. }`
   only when `dom_targets.contains(id)`.

### files changed.

| file | change |
|------|--------|
| `zo-executor/src/executor.rs` | `execute_directive` special-cases `#dom` — resolves ident to local.value_id, no Load |
| `zo-driver/src/cmd/run.rs` | two-pass extraction — `Directive` targets first, matching `Template` commands second |

### verified with.

- 065-zsx-nested-component.zo: 10 → 6 commands (only
  `paragraph`; `nested` inlined, no longer rendered on its
  own).
- zsx-nested-component.zo: 8 → 5 commands (silent duplicate
  gone).
- 264/264 zo_test pass.

## feature — `uint` type support.

`imu x: uint = 10` — `uint` is the default unsigned integer
type (= `u32`), parallel to `int` (= `s32`). Integer literals
are parsed as the default `int`; when used in a context that
expects a different concrete integer type (annotation or
typed comparison), the literal's SIR `ConstInt` is rewritten
to match — Rust-style polymorphic integer literals, but
locally scoped.

### tokenizer + tychecker.

already in place — `uint` keyword tokenizes to
`Token::UintType`, resolves to `Ty::Int { signed: false,
width: U32 }` via `resolve_ty_symbol` and `resolve_type_token`.

### executor (zo-executor).

added `narrow_int_literal(sir_val, src_ty, target_ty)`
helper. rewrites a ConstInt's `ty_id` to `target_ty` when:
- `src_ty` is the default int (`s32`),
- `target_ty` is a different concrete integer type,
- `sir_val` points to an `Insn::ConstInt`.

called from:
- `finalize_pending_decl` — `imu x: uint = 10`.
- `execute_check_modifier` — `check@eq(x, 10)` where `x: uint`.

scope: direct literals only. expressions like `10 + 5`
produce a `BinOp` whose result is s32; retyping through
expressions is a broader change (polymorphic literal
propagation) not attempted here.

### files changed.

| file | change |
|------|--------|
| `zo-executor/src/executor.rs` | `narrow_int_literal` helper, applied at decl and check@eq |
| `zo-tests/programming/lit-uint.zo` | promoted from `wip/wip-pass` |

### verified with.

- `imu x: uint = 10; check@eq(x, 10);` ✓

## bugfix — register aliasing in `BinOp::Eq` str codegen.

`check@eq(template, str_literal)` silently failed at runtime
for `061-zsx-hello.zo` and `064-zsx-html-directive.zo`. Both
strings were correctly emitted with identical bytes, but the
comparison always returned false. These failures pre-dated
today's work — they surfaced only through a full
`zo-test-runner` pass (`zo run` with `#dom` short-circuits to
the UI runtime, bypassing the check in the binary).

**root cause.** the str equality path emitted:

```
add X0, l, 8    ; ptr1
add X1, r, 8    ; ptr2
mov X2, X16     ; len
bl  _memcmp
```

When the register allocator placed `r` in X0 (the destination
of the first `add`), the first instruction clobbered the rhs
pointer before the second instruction read it. `memcmp` then
compared `lhs_data` against `lhs_data + 8`, which differ in
the first 8 bytes — equal strings reported as unequal.

**fix (zo-codegen-arm).** detect the aliasing and pick a
safe ordering:
- if `r == X0` and `l == X1` (both alias): stash rhs in X9,
  then `add X0, l, 8; add X1, X9, 8`.
- if `r == X0`: emit rhs add first, then lhs add.
- otherwise: keep the original order.

### files changed.

| file | change |
|------|--------|
| `zo-codegen-arm/src/codegen.rs` | aliasing-aware ordering in str BinOp::Eq/Neq path, X9 scratch import |

### verified with.

- `061-zsx-hello.zo` — template `check@eq` passes. ✓
- `064-zsx-html-directive.zo` — `{#html}` splice `check@eq`
  passes. ✓
- 265/265 zo_test pass (was 263/265).

## feature — compile-time string slicing.

`s[lo..hi]` and `s[lo..=hi]` lower to a fresh
`Insn::ConstString` containing the sliced bytes. runtime
slicing needs a view layout or heap allocator — neither
exists today — so v1 is compile-time-only. both bounds and
the receiver must be compile-time constants; anything else
errors at analysis time.

```zo
fun main() {
  imu s: str = "hello, world!";
  imu hello: str = s[0..5];
  imu world: str = s[7..=11];

  check@eq(hello, "hello");
  check@eq(world, "world");
}
```

### detection.

the parser emits postorder, so `s[0..5]` is laid out
`Ident(s), LBracket, Int(0), Int(5), DotDot, RBracket` —
`DotDot`/`DotDotEq` sits at `idx - 1` when the index is a
range. the `Token::RBracket` handler checks that one slot
and forks into the slice path, no extra state on the
executor.

### executor (zo-executor).

- `execute_str_slice_const` — pops `hi`, `lo`, receiver;
  validates compile-time-ness, UTF-8 boundary, range order,
  bounds; interns the sliced bytes; emits `ConstString`.
- `resolve_const_str_sym` — traces a SIR `ValueId` back to
  its source string: either a direct `Insn::ConstString`
  (chained slices like `s[0..10][2..5]`) or an
  `Insn::Load { src: LoadSource::Local(sym), .. }` referring
  to a `Value::String` local.
- `value_as_const_int` — unwraps a `Value::Int` on the value
  stack to its underlying `u64`. used for both slice bounds.

inclusive ranges (`..=`) adjust the upper bound by +1
before bounds check; utf-8 boundary violations surface as
`StrSliceOutOfBounds` (same effect for the user — the
slice can't land there).

### errors (zo-error, zo-reporter).

four new `ErrorKind` variants, all compile-time:

| kind | trigger |
|------|---------|
| `StrSliceRequiresConstBounds` | `lo` or `hi` isn't a compile-time int |
| `StrSliceRequiresStr` | receiver doesn't trace to a const `str` |
| `StrSliceOutOfBounds` | `hi > src.len()` or slice mid-UTF-8 |
| `StrSliceInvalidRange` | `lo > hi` |

### tests.

|-------|-------|
| `zo-parser` | `test_str_slice_tree_shape` — locks tree contract |
| `zo-executor` | 5 new in `tests/str_slicing.rs` — happy paths + all 4 error paths |
| `zo-tests` | `lit-str-slice.zo` — integration |

| file | change |
|------|--------|

## bugfix — generics: ty_id rewrite in monomorphization.

the monomorphize pass cloned a generic's SIR body and
rewrote its FunDef name + `__abstract::` placeholder calls,
but left every inner `ty_id` field pointing at the
generic's original `Ty::Infer(..)` inference vars. BinOp,
TupleIndex, FieldStore, Cast — any instruction with a
`ty_id` — kept generic types in the monomorphized output.

today no codegen path trusts those ty_ids for correctness
(the str-comparison bug earlier in CL12 was register
aliasing, not type confusion), so the tests never caught
it — but any future pass that reads `insn.ty_id` for a
real decision would get `Infer(old_var)` and misbehave.

**fix.** at each instantiation, install
`old_generic_var → fresh_var_unified_with_concrete`
substitutions, walk every `ty_id` in the cloned body via
`visit_ty_ids_mut`, and rewrite through `resolve_id`. the
fresh vars are already unified with the concrete argument
types at call time, so `old_var → fresh_var → concrete`
chains to a concrete TyId. substitutions are cleared after
each instantiation so different types (`sum__int`,
`sum__float`) don't interfere.

### the correctness invariant.

`test_mono_body_has_no_infer_ty_ids` in
`zo-executor/src/tests/generics.rs` asserts the invariant:
every instruction inside a monomorphized body's
FunDef..Return range must have every `ty_id` field resolve
to a non-`Ty::Infer(..)` kind. this is invariant 3 from
`PLAN_STR_SLICING_AND_GENERICS.md`:

> ty_ids are concrete at instantiation boundaries. for
> every `Insn` in a mangled-fn body, `kind_of(ty_id)` never
> returns `Ty::Infer`.

### what changed.

- `zo-sir/src/sir.rs` — new `Insn::visit_ty_ids_mut`
  walking every `ty_id` field (every `Const*` / `Load` /
  `Store` / `Call` / `BinOp` / `UnOp` / `Array*` /
  `Tuple*` / `Struct*` / `EnumConstruct` / `FieldStore` /
  `Template` / `VarDef` / `ConstDef` / `Directive` /
  `Return`, plus `Cast.to_ty`). kinds without a type slot
  (`FunDef`, `Jump`, `BranchIfNot`, `Label`, `Nop`, etc.)
  are no-ops.
- `zo-ty-checker/src/tychecker.rs` —
  `install_substitution(var_ty, target) -> Option<TyId>`
  and `clear_substitution(var_ty, prev)`. scoped pair that
  extracts the `InferVarId` from an infer-var TyId and
  inserts/removes the substitution, returning the previous
  mapping for reversible binding.
- `zo-executor/src/executor.rs`:
  - call site records `(mangled, generic_name, subs)` in a
    new `pending_instantiations` side table where
    `subs: Vec<(TyId, TyId)>` captures
    `(old_var_ty, fresh_var_ty)` per type parameter.
  - `monomorphize` drains the queue into a `HashMap`, then
    per-instantiation: installs the substitutions, walks
    the cloned body with `visit_ty_ids_mut` + `resolve_id`,
    clears the substitutions.
  - also lands the scaffolding (`generic_tree_ranges`,
    `mono_name_override`) for a future re-execution
    refactor (phase 2.c+). not yet active — current
    approach keeps the clone pass, just makes it correct.
