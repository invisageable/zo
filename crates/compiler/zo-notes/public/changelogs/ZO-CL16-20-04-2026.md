### infrastructure — `zo-test-runner --target`.

The test-runner used to hard-code the
default target (arm64-apple-darwin on
this host). Added a `--target / -t`
flag that threads into every
`zo build` invocation in the
runner, so the same 296-program
suite can be measured against
the CLIF path by passing
`--target x86_64-apple-darwin`.

#### `build_cmd` helper.

`zo build` gets pre-applied with the
optional target flag via a shared
helper so `run_test` / `run_project`
don't each reimplement the conditional
arg injection:

```rust
fn build_cmd(zo: &Path, target: Option<&str>) -> Command {
  let mut cmd = Command::new(zo);

  cmd.arg("build");

  if let Some(t) = target {
    cmd.args(["-t", t]);
  }

  cmd
}
```

#### `run_with_timeout` — kill-without-wait.

The first CLIF-target run hung
indefinitely — programs on Rosetta
were stuck at 0% CPU on specific
insn patterns, and rayon's
`par_iter` waited for every
worker before printing a
conclusion.

Added a 10-second per-test
timeout via `try_wait` polling.

A subtlety that burned one
iteration: on timeout the helper
calls `child.kill()` but
intentionally SKIPS the follow-up
`child.wait()`. Some Rosetta-
translated processes ignore
SIGKILL outright; `wait` would
then block the rayon worker
forever. A zombie leak is
acceptable (the OS reaps on
parent exit), a deadlocked
worker is not:

```rust
if start.elapsed() > timeout {
  let _ = child.kill();

  return Err(io::Error::new(
    io::ErrorKind::TimedOut,
    "runtime timeout",
  ));
}
```

### key code changes (files touched).

#### `zo-codegen-clif/src/translate.rs`

- **`emit_exit_1` replaces every raw `trap` emission** (except the unreachable post-`exit` terminator inside `emit_exit_1` itself) via a bulk sed swap. Rosetta hangs on `ud2` but handles `exit(1)` cleanly; the one-liner `abort()` intermediate was rejected because macOS' crash reporter blocks the process awaiting diagnostic collection.
- **`coerce_int_width(builder, v, from, to)`** — shared helper that `uextend`s widening and `ireduce`s narrowing between integer CLIF types. Used by both the Call arm (per-arg against the callee's declared signature) and the BinOp arm (widest-wins on operands). Same-type / non-int pairs pass through unchanged.
- **`Store` auto-declare path** — if `ctx.vars.get(name)` misses, declare the `Variable` with the store value's actual CLIF type (not the SIR-declared `ty_id`, which can be a generic alias whose width doesn't match).
- **`VarDef` init-driven sizing** — when an initializer is present, the `Variable`'s CLIF type follows the init value's type rather than the SIR `ty_id`. Same reason as above.
- **`Jump` / `BranchIfNot` lazy targets** — `ctx.blocks.entry(target).or_insert_with(|| builder.create_block())` covers the case where a SIR `Label { id }` was reserved but optimized out.
- **Post-translate orphan sweep** — for every block in `ctx.blocks`, check `last_inst(block)`'s opcode for terminator-ness; if not terminated, switch + emit `exit_1`. Cranelift DCEs unreachable blocks. Replaces the earlier single-block check that only covered the current (last-switched) block.
- **`defined_funcs` guard** — `HashSet<FuncId>` in the body-definition pass, skipping second-pass `define_function` calls for FuncIds already defined (handles symbol-collision between user code and stdlib).
- **Bytes dispatch split** — `match arg_ty_id.0 { 4 => str_path, 5 | 6..=14 => emit_int_show, ... }`. TyId 5 (bytes) scalar byte literals (`` `z` ``) route through `%lld`; slice-with-show path removed pending a type-table-aware discriminator.

#### `zo-test-runner/src/main.rs`

- `--target / -t <triple>` arg threaded through `run_dir` / `run_projects` / `run_test` / `run_project`.
- `RUN_TIMEOUT` constant + `run_with_timeout` helper.
- `build_cmd(zo, target) -> Command` factored helper.
- `needless_return` clippy regression fix on the `runtime timeout` arm.

#### `zo-ty-checker/src/tychecker.rs`

- Added `s64_type()` helper (companion to existing `s32_type` / `u32_type`). Currently unreferenced at commit time but kept — it's the natural primitive for a future int-width-widening patch in the executor.

### lessons.

- **The trap hang was a two-layer bug.** The verifier panic from phase 5.10 was already fixed (`trap_and_resume` dead-block switch). The runtime hang was a Rosetta quirk on `ud2`. Both layers had to be fixed to get a deterministic suite; either alone left the runner looking broken.
- **Memoize across cache tiers.** The ARM suite failed after `just zo_test_runner` ran on a committed-but-reverted executor patch — `target/debug/zo` was stale. Memory note
  `project_zo_test_runner_binary.md`
  already covers this ("test-runner
  prefers `target/debug/zo` — rebuild
  debug too after executor changes")
  — worth re-reading whenever a
  test-runner result surprises.
- **Narrow TyId assumptions get punished.** Two independent fixes — `VarDef` init-sized, `Store` value-sized — both came down to "don't trust the SIR-declared `ty_id` when an actual CLIF value is already in hand; defer to the value's type." Generic instantiations expose this because zo's executor reuses generic-parameter `TyId`s that don't always survive width legalization.
- **Width-coercion belongs at every SSA boundary.** Call args, BinOp operands, possibly future UnOp / Cast too. Cranelift's verifier is unforgiving about mixed-width inputs to homogeneous-typed ops. Each SSA consumer should widen or narrow as needed rather than assuming the producer picked the right type.

## sir type invariants — phase 0.

Per `PLAN_SIR_TYPE_INVARIANTS.md`: the CLIF phase 5 wrap
surfaced `coerce_int_width` as a shim — the real bug was
always in tychecker + executor producing mixed-width SIR.
Phase 0 lands the tooling that pins this down before
phases 1–9 start rewriting the executor.

### three-tier test chain.

No `cfg(debug_assertions)` hack, no env-var-gated printers.
Proper chain:

1. **`zo-sir/src/validator.rs`** — `validate(&[Insn]) ->
   ValidationReport` enforcing five invariants from the
   plan: `BinOp` operand parity, `Call` arg-vs-param
   parity, `Store` value-vs-decl parity, `Return`
   value-vs-fn-return parity, placeholder `TyId`s (Error,
   Type, Unknown, misplaced Template) forbidden. Zero
   dependency on `TyChecker`; caller-friendly `Violation`
   types.
2. **Unit tests** (`zo-sir/src/validator.rs:tests`) — 7
   tests, one per `ViolationKind` + clean-stream sanity
   + missing-callee skipped counter.
3. **Integration tests**
   (`zo-analyzer/src/tests/type_invariants.rs`) — 4 tests
   driving the real `parse → tree → SIR` pipeline against
   hand-crafted zo source. Each test pins one *current*
   broken behavior with a flippable assertion tagged for
   the phase that fixes it.
4. **zo-tests program**
   (`zo-tests/programming/literal-decl-widths.zo`) —
   covers `s8`..`s64` / `u8`..`u64` decl-site narrowing
   (which works today via the tychecker) as a regression
   guard for phases 1–4.

### the big finding.

The executor's failure mode on width-mismatched operands
isn't "emit mixed-width SIR" as the pre-plan audit
suggested. It's **silent drop** — the entire enclosing
expression fails to emit.

`imu _y: u16 = x + 5;` where `x: u16` produces:

```
ConstInt(10, u16)
VarDef { x, ty: u16, init: Some(_) }
Load(x, u16)
ConstInt(5, s32)                    ← literal stays s32
VarDef { _y, ty: u16, init: None }  ← BinOp aborted
```

No `BinOp` insn reaches SIR. No diagnostic. No error. The
same pattern applies to `f(42)` when `f(x: s64)` — no
`Call` in SIR at all.

Implications for the plan (recorded in
`PLAN_SIR_TYPE_INVARIANTS.md:Phase 0 findings`):

- The validator only catches **surviving** mismatches. The
  `Return`-with-wrong-ty case fires cleanly; the BinOp /
  Call drops do not.
- **Phases 2 + 3 fix the silent drop as a side effect.**
  When literals adopt the expected width up-front,
  unification never fails, emission proceeds.
- Phase 5 (diagnostics) is still needed for non-literal
  mismatches but its ordering relative to 1–4 is loose.

### first iteration mistake.

Originally wired the validator into `zo-analyzer` under
`cfg(debug_assertions)` with a `ZO_VALIDATE_SIR=1` env gate
that printed violations to stderr. User called it out: not
a test, not reproducible, not CI-covered. Reverted; three-
tier chain above replaces it.

## sir type invariants — phase 1.

Per `PLAN_SIR_TYPE_INVARIANTS.md` Phase 1: thread an
expected-type stack through the executor so `ConstInt` /
`ConstFloat` emission adopts the enclosing context's width
instead of hardcoding `s32` / `f64`.

### what landed.

- `Executor.expected_ty_stack: Vec<Option<TyId>>` — a plain
  stack of optional expected types. Push at context-entry,
  pop at exit.
- `peek_expected_int_ty()` / `peek_expected_float_ty()` —
  helpers that filter the top of the stack to a matching
  kind. Non-int / non-float / `None` → helper returns
  `None`, literal falls back to `int_type()` / `f64_type()`.
  Unit-safe.
- `Token::Int` and `Token::Float` literal emission arms use
  the helpers before the fallback.
- `begin_decl` pushes the annotated type (or `None`) right
  after creating `PendingDecl`.
- `finalize_pending_decl` pops immediately after the
  matching `take()` on `pending_decl`. The push / pop pair
  is balanced across every code path (early-return, no-init
  decl, error paths) because the push is coupled 1:1 with
  `PendingDecl` creation and the pop runs only when that
  `PendingDecl` is consumed.

### over-delivery.

Phase 1 was scoped to "only makes literals cooperate at the
decl site." In practice the decl-site push **flows through
the entire init expression**, so:

- `imu _y: s64 = f(42);` — the `s64` context reaches `42`
  through the Call; the literal lands as `s64`, unification
  at the call site succeeds, and SIR has a clean `Call`.
- `imu _y: u16 = x + 5;` — the `u16` context reaches `5`
  through the BinOp; operand widths match, BinOp emits
  cleanly.

Both scenarios previously **silently dropped** the
enclosing expression at unification failure. Phase 1 makes
them emit correctly. Phases 2–3 remain scoped to the
scenarios Phase 1 *doesn't* cover — bare calls without a
surrounding decl (`f(42)` as a statement), call-in-call
(`f(g(42))`), binops whose enclosing decl has no type
annotation, etc.

## sir type invariants — phase 2.

Per `PLAN_SIR_TYPE_INVARIANTS.md` Phase 2: bare calls
(`f(42);` as a statement), nested calls (`f(g(42))`), and
any other site where Phase 1's decl-scope push doesn't
reach, now also propagate the callee's param types to their
argument subexpressions.

### why at LParen.

The executor is a natural tree walker. Args execute
*eagerly* as the main loop sweeps nodes — by the time the
Call's own finalize runs (at RParen), the arg values are
already on the value stack and the literals inside them
were emitted long ago. The only place we can still steer
literal widths is *before* each arg's subexpression begins.

So Phase 2 doesn't live at the unification site; it lives
at LParen-of-call, where the callee is already resolvable
via the existing `resolve_call_target`.

### what landed.

- `CallCtx` struct (param_tys, arg_starts, rparen_idx,
  arg_idx) + `Executor.call_ctx_stack: Vec<CallCtx>`.
- `begin_call_ctx(lparen_idx)` — at LParen of a user call,
  walks the tree between the parens tracking bracket depth,
  records arg-start indices at depth 1, pushes the ctx, and
  primes `expected_ty_stack` with arg 0's param type.
- `maybe_advance_call_arg(idx)` — called from the top of
  every `execute_node` dispatch. When `idx` matches the
  next arg's start, rotates the top of
  `expected_ty_stack` from arg i's expected to arg i+1's.
- `end_call_ctx(idx)` — at RParen, pops the ctx + its
  final expected ty frame iff this RParen matches the
  recorded `rparen_idx`. Tuple / enum-ctor / grouping
  RParens are a no-op.
- Scoped no-ops: closures, externals, zero-param callees,
  unresolved names. They fall back to the old unconstrained
  path so existing behavior is preserved.

### why it composes with phase 1.

Phase 1's decl-site push and Phase 2's call-site push both
use the same `expected_ty_stack`. Nested calls stack their
own `CallCtx` entries, and each pushes onto the shared
expected-ty stack. Literal emission reads the top, which
is always the **innermost enclosing expected type** — call
arg > decl > outer call > outer decl. Exactly what
type-directed literal elaboration should do.

## sir type invariants — phase 3.

Per `PLAN_SIR_TYPE_INVARIANTS.md` Phase 3: make both
operands of a `BinOp` share one type before emission, in
the scenarios Phases 1 + 2 don't cover (bare assignment
RHS, intermediate expressions, sibling-operand inference).

### two mechanisms, one per operand-order case.

**RHS-literal (`x + 42`):** The tree walker processes `x`
then `+` then `42`. At `+`, the RHS isn't on the stack
yet, so `execute_binop` takes the *defer branch*: pops the
LHS and stashes it in `deferred_binops`. This is the
moment `lhs_ty` is in hand but the RHS subexpression is
about to be walked. Phase 3 pushes `Some(lhs_ty)` onto
`expected_ty_stack` right there; the bare literal RHS
reads it at emission time and adopts the LHS's width.
Matching pop at the top of `apply_deferred_binop`'s drain
loop, right after the deferred entry is consumed.

**LHS-literal (`42 + x`):** The literal is emitted BEFORE
the `+` fires, so the defer-path push is too late — by the
time `+` runs, the default-typed `ConstInt(42, s32)` is
already in SIR. Phase 3 rescues this case post-hoc in the
"both operands present" branch of `execute_binop`: just
before unification, run `narrow_int_literal` symmetrically
(LHS-toward-RHS and RHS-toward-LHS). The function is a
no-op unless its target is a default-typed int, so either
direction only fires when the right side is the literal.
If it fires, the in-place `ConstInt.ty_id` rewrite lets
unification succeed.

### why both are needed.

- `x + 42` → defer-path push covers.
- `42 + x` → post-hoc narrow covers.
- `(x + 2) * 3` → each inner binop's defer fires in order;
  intermediate results are typed when they land on the
  stack, so outer binops see concrete LHS and defer-path
  push handles RHS literals.

Together, a binop has matching operand types regardless of

## sir type invariants — phase 5.

Per `PLAN_SIR_TYPE_INVARIANTS.md` Phase 5.

### the finding.

The audit's "silent-fail unification" framing was wrong.
`TyChecker::unify` **already** reports `TypeMismatch` on
every failure path (`tychecker.rs:396, 413, 456, …`). The
"silent drop" Phase 0 observed is at the SIR-emission
level — the BinOp / Call insn doesn't reach SIR — but the
user already sees a clean diagnostic emitted by the
tychecker.

### first iteration mistake (again).

Added `report_error(Error::new(ErrorKind::TypeMismatch,
span))` at three executor sites: `execute_binop`'s None
arm, the Call arg unification check, and `check@`'s
operand unification. Integration tests passed — but the
full `just zo_test_runner` suite regressed two `fail/`
programs:

- `044-arg-type-mismatch` — expected 1 error, got 2.
- `035-binop-right-truthy` — same.

Both were emitting the SAME TypeMismatch error twice at
the same span, once from tychecker and once from the
executor. Reverted.

### what landed.

- No executor behavior change. Comments at the three
  `unify().is_none()` sites were refreshed to document
  that `unify` already reports the mismatch and the
  executor just bails from emitting SIR to avoid cascading
  errors.
- Integration tests
  (`zo-analyzer/src/tests/type_invariants.rs`):
  - `binop_concrete_width_mismatch_reports_type_mismatch`
    — `u16 + u32` with two concrete locals.
  - `call_concrete_arg_mismatch_reports_type_mismatch` —
    `f(u32)` passed to `fun f(x: u16)`.
  Both use `zo_reporter::collect_errors()` to assert a
  `TypeMismatch` reaches the reporter. End-to-end
  coverage for the error path, guarding against a future
  refactor accidentally dropping the diagnostic.
