# ZO-CL15 — 19-04-2026.

one self-contained fix today: operator
precedence around `(...)` in expression
context. the bug was latent for a long time
— only showed up when a higher-precedence
operator followed the closing `)`. writing
a canonical `binary-search.zo` as a user
would surfaced it via a silent infinite loop
printing `2` forever on every iteration.

---

## issue H — precedence of `EXPR OP (PAREN) HIGH_PREC_OP RHS`.

minimal repro.

```zo
fun main() {
  mut low: int = 3;
  mut high: int = 4;
  imu a: int = low + (high - low) / 2;
  showln(a);
}
```

expected `3 + (1 / 2) = 3`. got `2 = (3 + 1) / 2`.

the real `binary-search.zo` used the same
expression inside a `while low <= high` body
— `imu mid = low + (high - low) / 2` — so
`mid` computed the wrong midpoint every
iteration, `low = mid + 1` never advanced
past 2, and the loop ran forever with
`showln(mid)` printing `2` every time.

### root cause.

`handle_lparen_introducer` always called
`flush_expr()` on `(`. That drains the entire
`operator_stack` and emits pending operators
into the tree **before** the paren contents.
For `3 + (4 - 3) / 2` the tree came out as:

```
3, +, (, 4, 3, -, ), 2, /
```

the `+` was committed before `/` arrived.
the executor's `apply_deferred_binop` fired
at `)` (tuple_ctx empty), consuming the `+`
against the group result `1` — giving
`3 + 1 = 4`. then `/` popped `4` and `2`
and computed `4 / 2 = 2`.

every test case reproduced this shape:

| source | expected | old result |
|---|---|---|
| `3 + (4 - 3) / 2` | 3 | 2 |
| `3 + (4) / 2` | 5 | 3 |
| `10 + (2) * 3` | 16 | 36 |
| `10 - (2) * 3` | 4 | 24 |
| `10 + (2 + 3) * 4` | 30 | 60 |
| `2 + five() * 3` | 17 | 21 |

### fix — parser.

at a non-FunSig / non-TypeAnnotation `(`:

1. drain operators with precedence ≥ 12 (the
   `Dot` — for `a.b(` callee chains) from
   the op stack, moving them to their
   postorder position at the end of the
   buffer.
2. walk the buffer and emit every "settled"
   entry: values, plus operators that have
   already been moved to their final
   position. an operator is **active** (still
   parked in `operator_stack`) iff its
   *rightmost* occurrence in the buffer is
   the parked one — earlier copies of the
   same token were moved by
   `handle_operator`'s precedence pop.
3. save whatever's left of `operator_stack`,
   `expr_buffer`, and `unary_spans` onto the
   per-context save stacks (same mechanism
   array indexing `[` already uses).
4. emit `(`, push an LParen introducer so
   the matching `)` can restore.
5. the matching `)` restores all three
   saved stacks if the introducer's saved
   state wasn't `FunctionSignature` (param
   lists don't save — they never pushed).

with the fix, the same source parses to:

```
3, (, 4, 3, -, ), 2, /, +
```

— correct postorder. executing left-to-right:
`3` pushed; `1` pushed via group; `2` pushed;
`/` → `0`; `+` → `3`. done.

### short-circuit exception.

`&&` / `||` rely on the executor seeing the
logical op **before** the RHS subexpression,
so it can emit a `BranchIfNot` skeleton
ahead of the RHS's SIR. true postorder emits
the op after RHS — too late for lazy eval.

so: if the op stack contains `&&` or `||`
when `(` is hit, the parser falls back to
the old full-flush behavior. preserves
short-circuit for `x || incr(y)` at the
cost of (correct but non-short-circuiting)
postorder for the narrow
`x || a + f() / b` case, which doesn't
appear in any current program.

### secondary fix — executor call detection.

the parser fix made `low + (expr)` emit
`low, LParen` adjacent in the tree. the old
`resolve_call_target` / `ident_is_call_target`
returned "this is a call" for any
`Ident LParen` pair, so the executor tried
to call `low` as a function and crashed
with `Undefined variable a` (the Imu
declaration's body failed, so `a` never
got bound into scope).

fix. both functions now reject the call
interpretation only when the ident resolves
to a known **non-closure local**. a `::` /
`.` / `@` prefix (`Point::new(`, `pack.fn(`,
`check@eq(`) still forces the call
interpretation — `new` / `eq` aren't in
`self.funs` by their bare name. unknown
idents (external or builtin callees, e.g.
`show` in an isolated unit test that
doesn't load preload) stay calls — matches
pre-change behavior for those.

### lessons.

1. **a "test" at the user level beats a unit
   test.** this bug sat in the parser the
   whole CL14 cycle. writing binary-search
   the way a real user would — not the way
   a test author would — exposed it on the
   first run.
2. **silent infinite loops are worse than
   crashes.** the program compiled clean and
   ran to zero output with exit 0. only
   backgrounding it and reading the output
   after a timeout revealed the `2` flood.
3. **array indexing already had the answer.**
   `handle_lbracket` already parked outer
   state on the save-stacks and restored on
   `]`. the paren path should have done the
   same from day one; the call-vs-group
   distinction misdirected the design.
4. **short-circuit is load-bearing on emission
   order.** execution-based compilation
   means the tree's linear sequence *is* the
   SIR emission schedule. "correct postorder"
   and "short-circuit-capable" are in
   tension — one of them has to bend, and
   for `&&`/`||` it's postorder.
