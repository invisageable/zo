# ZO-CL23 — 2026-04-29

## `[v...]` / `[v...n]` — repeat-array literal.

Hare-style repeat. The `...` token is reserved for this
purpose only — todo / spread will pick different syntax
when they land.

```zo
imu zeros: [5]int = [0...];        -- type drives the count
imu sevens: []int = [7...4];       -- explicit count
imu pad:    [5]int = [1, 2, 3...]; -- prefix + spread last
```

### shape.

Three forms, one operator:

| form              | length source        |
| ----------------- | -------------------- |
| `[v...]`          | `[N]T` annotation    |
| `[v...n]`         | literal int after `...` |
| `[v1, v2, v3...]` | `[N]T` annotation, last value spreads |

### approach.

`Token::Ellipsis` was already lexed but had no parser
binding. Wired three sites:

- **parser** — `handle_ellipsis()` flushes the pending
  value, emits an `Ellipsis` node, requires an open `[ `
  introducer in expression state. Outside an array
  literal it's `UnexpectedToken`.
- **executor** — `array_ctx` becomes a struct carrying
  `ellipsis_at: Option<usize>` (the SIR-stack position at
  `...`). The `Token::RBracket` array-literal branch
  reads the field; a count of 0 elements after `...`
  pulls N from `pending_decl.annotated_ty`, a count of 1
  reads it as a literal int. Builds an
  `Insn::ArrayLiteral` with the expanded elements;
  codegen sees the same shape as a hand-written literal
  and needs no changes.
- **errors** — `RepeatRequiresKnownLength` when `[v...]`
  appears without a `[N]T` annotation,
  `RepeatLengthMismatch` when n disagrees with N,
  `RepeatCountNotConst` when the count isn't an integer
  literal.

## `[N]T` reassignment — value semantics.

```zo
mut a: [3]int = [1, 2, 3];
mut b: [3]int = [4, 5, 6];

a = b;
b[0] = 99;

showln(a[0]);  -- now 4 (was 99 — aliased the same block)
```

`a = b` was emitting an 8-byte `STR` of the source's
block address into `a`'s slot, so both names walked the
same memory. Writes through `b` were observable through
`a`, breaking any program that swaps fixed arrays —
including the obvious `row = next` cellular-automaton
update step.

### approach.

`[N]T` variables now own their inline `[len:8][cap:8]
[e0:8]...[eN:8]` block in the function frame. Two new
sites:

- **register allocator** — `mutable_size` no longer
  assumes 8 bytes per Store target. It scans
  `ArrayTyDef`s up front to learn each `[N]T` size,
  then reserves `(2 + N) * 8` per `[N]T` Store name.
- **arm codegen** — new `array_var_blocks: HashMap<u32,
  u32>` maps name → block offset. `Insn::Store` for an
  `[N]T` `ty_id` memcopies word-by-word from the
  source's pointee into the block. `Insn::Load` for an
  `[N]T` local computes `SP + block_off` instead of
  loading 8 bytes — `ArrayIndex` / `ArrayStore` consume
  this base address unchanged.

The bug pre-dated this changelog; repeat-array literals
just exercised it harder by making `mut row: [N]T = [0...];
mut next: [N]T = [0...]; ... row = next;` natural to
write.

## interner — `String` realloc UB.

`zo-interner` held `&'static str` keys in `map` that
pointed into a single growing `storage: String`. The
"SAFETY" note claimed storage was append-only so pointers
stayed valid. `String::push_str` reallocates once
capacity is exceeded; every prior key in `map` becomes
dangling. Under the synth-10K bench (which crosses the
4 KiB initial capacity), about 1 in 30 runs reused the
freed pages mid-compile, lookups missed, names re-
interned to fresh symbols, and the analyzer reported
spurious `[E0572] Undefined variable` mid-function.

### approach.

Replaced `storage: String` with `strings: Vec<Box<str>>`.
Each interned string is its own boxed slice — heap
address is stable. Pushing more boxes into the `Vec`
relocates `(ptr, len)` tuples but leaves every backing
buffer where it was, so `&'static str` keys derived from
those buffers remain valid for the interner's lifetime.
`get(symbol)` now indexes `strings[symbol.0]` directly;
the `spans` field is gone.

100/100 stress runs of the synth-10K bench now pass
(was 29/30).

## soundness — five type-error gaps closed.

| program                              | now rejects                          |
| ------------------------------------ | ------------------------------------ |
| `branch-if-else-wrong-condition.zo`  | non-bool `if`/`else if` condition    |
| `struct-missing-field.zo`            | `ivs.age` when `Person` has no `age` |
| `type-alias-recursion.zo`            | `enum Foo { Bar(Foo) }`              |
| `pack-duplicates.zo`                 | duplicate `pack math::(sqrt);`       |
| direct field write                   | `p.x = expr` outside `apply` blocks  |

All four wip-fail cases now error with a real diagnostic;
the silent `p.x = expr` no-op now lowers to
`Insn::FieldStore` via a new `pending_field_assign`
mirror of the existing `pending_array_assign` path.

New `programming/struct-field-assign.zo` covers the
direct-write case; the four error cases were already in
`programming/wip/wip-fail/`.

## synth-10K cross-compiler bench.

`zo-benches/benches/synth-10k/` — 1000 functions, each
calling its predecessor; main computes
`func999(1)`. Same shape in `synth-10k.zo`, `.c`,
`.rs`. Names are zero-padded `funcNNN` so they never
collide with type keywords (`f32` / `f64`).
`zo-benches/src/bin/generate_synth.rs` is the single
source of truth that emits all three.

End-to-end (source → ARM64 binary, link included):

| compiler | hot avg | LoC/s    | vs zo       |
| -------- | ------- | -------- | ----------- |
| **zo**   | 60 ms   | ~167K/s  | 1×          |
| clang    | 148 ms  | ~67K/s   | 2.4× slower |
| rustc    | 321 ms  | ~31K/s   | 5.3× slower |

## pipeline bench — `zo-codegen/benches/generate.rs`.

Criterion bench at four sizes (100, 500, 1K, 5K
functions) running the full in-process pipeline:
tokenize → parse → analyze → codegen. Throughput
breakdown on the 50K-line case:

| phase     | ~LoC/s | target | vs target  |
| --------- | -----: | -----: | ---------- |
| tokenizer |   13 M |   10 M | +30% over  |
| parser    |   13 M |   10 M | +30% over  |
| analyzer  |  3.8 M |    5 M | -24% under |
| codegen   |  1.8 M |    5 M | -64% under |

Tokenizer and parser exceed the manifesto targets;
analyzer is close; codegen is the gap to close next.
