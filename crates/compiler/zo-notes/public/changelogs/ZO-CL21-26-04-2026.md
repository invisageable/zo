# ZO-CL21 — 2026-04-26

## Phase A — `Vec<$T>` stdlib type.

PLAN_AOC's first phase ships: a growable list backed by
a flat byte arena. Same architectural pattern as
`HashMap` post-BB-6 — five marshaling FFIs handled by
codegen, two raw FFIs for `len` / `free`, pure-zo
bodies for `len` / `is_empty` / `free`.

### runtime.

`zo-runtime/src/vec.rs` adds `ZoVec` (a heap-stored
byte buffer with element-size + length + capacity)
and seven C-unwind exports: `zo_vec_new`,
`zo_vec_push`, `zo_vec_pop`, `zo_vec_get`,
`zo_vec_set`, `zo_vec_len`, `zo_vec_free`. Capacity
doubles when `len == cap` (initial 8). Four unit
tests cover empty, push/pop round-trip, grow past
initial capacity, and get/set OOB behavior.

### surface (`std/vec.zo`).

```zo
struct Vec<$T> { ptr: int }

apply Vec<$T> {
  pub fun new() -> Self { ... }
  pub fun push(self, v: $T) { }
  pub fun pop(self) -> Option<$T> { ... }
  pub fun get(self, i: int) -> Option<$T> { ... }
  pub fun set(self, i: int, v: $T) -> bool { ... }
  pub fun len(self) -> int {
    return __zo_vec_len_raw(self.ptr);
  }
  pub fun is_empty(self) -> bool {
    return self.len() == 0;
  }
  pub fun free(self) {
    __zo_vec_free_raw(self.ptr);
  }
}
```

`pack vec;` is preloaded alongside the other std
packs. The five marshaling apply methods carry
`-- codegen-replaced: ...` declaration-site comments
naming the missing language primitive
(`size_of<$T>()` and `&value`) — same discipline as
`std/map.zo`.

### codegen + register-allocation.

`emit_vec_new/push/pop/get/set` mirror the
`emit_map_*` shape — scratch slot allocation, runtime
extern call, `Option<T>` aggregate construction for
`pop` / `get`. `emit_vec_len_raw` / `emit_vec_free_raw`
pass through. `RuntimeNeeds::from_sir` flags the
concurrency dylib for any `Vec::*` or `__zo_vec_*`
call (same dylib also serves `HashMap`). The
register-allocator's struct-slot budget gets matching
bumps so scratch doesn't overlap caller-save area.

### regressions.

- `programming/vec-basic.zo` — `Vec<int>` push /
  pop / get / set / OOB / grow past initial capacity
  / free. 20-element grow round-trip.
- `programming/vec-str.zo` — `Vec<str>` storage
  round-trip (printing the popped str payload via
  `showln` is intentionally not exercised; that
  needs per-call-site `$T` derivation, tracked as
  PLAN_AOC phase C).

## Stale-dylib staging fix.

`stage_runtime_artifacts` (`zo-compiler/src/compiler.rs`)
used a `(size, mtime)` short-circuit to skip
re-copying the runtime dylib into the user binary's
output directory. After a `git checkout` swapped
source versions, two builds could land at the same
minute with the same byte count but different
contents — and dyld would silently hang the user
binary in `dyld3::MachOFile::compatibleSlice` when
the staged dylib's load commands didn't line up.
Replaced with an unconditional `std::fs::copy`. A
1 MB copy per program build is microseconds; the
optimization cost hours of debugging.

Verified: a deliberately-corrupted
`$TMPDIR/libzo_runtime.dylib` is now overwritten on
the next `zo build` / `cargo run --bin zo -- run`
without manual intervention.

## else-if-in-while bug — fixed.

Programs of the shape

```zo
while x <= n {
  if A { ... } else if B { ... } else { ... }
  x += 1;
}
```

ran ONE iteration and exited. The loop never
advanced.

### root cause.

`else if` parses as `else` followed by a fresh `if`
statement — it's not a single grammar production.
The executor's `Token::Else` handler fires on the
OUTER if and emits `Jump end_outer; Label
else_outer; (else_label = None)`. Then the INNER if
is processed: it pushes its own branch_ctx with
its own `end_inner` / `else_inner`. When the inner
if's `else { ... }` closes, the inner branch_ctx
pops cleanly — but the OUTER branch_ctx is still
on the stack with no token left to trigger its
close. Its `end_label` never emits. The while
loop's RBrace then sees the OUTER `If` ctx as
`branch_stack.last()` (not the while ctx), and the
scope-depth check fails, so nothing gets emitted —
no back-edge `Jump`, no while end-label.

### fix.

`zo-executor/src/executor.rs` — RBrace handler for
`BranchKind::If`'s no-else close path now cascades:
after popping the inner if, walk up the branch
stack while every outer entry is an `If` whose
`else_label` was already taken (i.e. `Token::Else`
fired on it earlier) AND whose `scope_depth`
matches the current scope. Emit each outer's
`end_label`, store/load the sink (for if-as-
expression), pop. Stops at the first non-If or
unentered-else ctx.

### regression coverage.

`programming/fizzbuzz-while-if-else-if.zo` had no
`EXPECTED OUTPUT` block before — the runner only
checked exit code (which was 0 because the
function returned cleanly after the failed first
iteration). Now wired with the canonical
`fizzbuzz(15)` golden output so the suite catches
this class of regression.

`zo-pp/src/pp.rs` previously printed `%{idx}` (Vec

## Phase B — `HashSet<$K>` stdlib type.

PLAN_AOC's second phase ships: presence-only
collection backed by the existing `ZoMap` runtime
with `val_sz = 0`. Reusing the map allocator means
single-source `key_kind`, hash function, probe
sequence, grow policy.

### surface (`std/set.zo`).

```zo
struct HashSet<$K> { ptr: int }

apply HashSet<$K> {
  pub fun new() -> Self { ... }
  pub fun insert(self, k: $K) -> bool { ... }
  pub fun contains(self, k: $K) -> bool { ... }
  pub fun remove(self, k: $K) -> bool { ... }
  pub fun len(self) -> int { ... }
  pub fun is_empty(self) -> bool { ... }
  pub fun free(self) { ... }
}
```

`pack set;` is preloaded alongside the other std
packs.

### codegen.

`emit_set_new`/`insert`/`contains`/`remove` mirror
the map handlers but pass `val_sz = 0` and use
`SP` as the val pointer (the runtime never reads
it). `emit_set_len_raw` / `emit_set_free_raw`
delegate to `emit_map_len_raw` / `emit_map_free_raw`
— identical extern-call shape, single source of
truth.

`insert` returns `true` if the key was new by
calling `_zo_map_contains` first, EOR'ing against
`1`, storing into a scratch slot, then doing the
unconditional insert.

### runtime — `val_sz = 0` safety guards.

Two callsites in `zo-runtime/src/map.rs` panicked
under HashSet's `val_sz = 0`:

1. `_zo_map_insert` — `from_raw_parts(val_ptr, 0)`
   tripped Rust's `unsafe precondition` check
   (post-1.78, the alignment + non-null check
   applies even at length 0).
2. `_zo_map_get` / `_zo_map_remove` — `copy_non
   overlapping(_, _, 0)` similarly checks the dst
   pointer.

Both now early-out with `if m.val_sz != 0 { ... }`
guards; the empty-value path constructs an empty
`Vec<u8>` instead.

### regressions.

`programming/hashset-basic.zo` exercises insert
(was-new bool), contains, remove, len, is_empty,
grow past initial capacity (32 inserts forces a
rehash), free. Full round-trip green.

## Mach-O writer — TEXT segment overflow.

`zo-writer-macho/src/macho.rs` capped TEXT segment
at 16 KB (4 × 4 KB pages). With the full stdlib
preloaded (str, map, set, vec, …) the emitted code
exceeded 15 KB and the writer produced a malformed
binary — DATA section's file region overlapped
TEXT's, dyld then either hung in
`compatibleSlice` or surfaced as garbled output
when a string const happened to land in the
overlap range.

Bumped TEXT and DATA segment caps to 64 pages
(256 KB each). Quick fix; the proper solution is
threading actual code/data sizes through the
LoadCommand-emit path and rounding up to a page
boundary. Tracked for follow-up — the bump
absorbs every realistic AoC-scale program.

## Phase C — HashMap K/V derivation per call site.

`HashMap<str, int>`, `HashMap<int, str>`, and other
non-int-int instantiations now hash and compare
correctly. Pre-Phase-C, `emit_map_new` hardcoded
`(key_kind = 0, key_sz = 4, val_sz = 4, cap = 16)` —
fine for `HashMap<int, int>`, broken for everything
else (str-keyed maps hashed pointer bytes instead of
str payload, so two separate `"alice"` pointers
hashed differently and `m.get("alice")` returned
`None`).

### approach.

zo's type system carries one `TyId` per struct
definition (no per-instance ids), so the binding
annotation `imu m: HashMap<str, int>` doesn't propagate
`K = str` / `V = int` through the resolved `TyId` to
the call site. Side-channel approach:

1. **Decl-time parse.** `begin_decl` now scans the
   `<...>` after a struct annotation and stashes the
   resolved `Vec<TyId>` in
   `decl_annotation_args` keyed by the bound symbol's
   id. Covers any multi-arg generic — HashMap<K,V>,
   Vec<T>, HashSet<K>.
2. **Call-time injection.** When `execute_call`
   dispatches `HashMap::new` / `HashSet::new` /
   `Vec::new` with a matching pending decl, it
   prepends three `ConstInt` args to the `Insn::Call`
   carrying `(key_kind, key_sz, val_sz)` (or
   `(elem_kind, elem_sz, _)` for Vec). `key_kind = 1`
   when `K` resolves to `Ty::Str`, else `0`. Sizes
   are uniformly `8` (the str pointer fits in a
   64-bit slot, ints zero-extend).
3. **Codegen consumption.** `emit_map_new` /
   `emit_set_new` / `emit_vec_new` move the prepended
   ValueIds into `X0`/`X1`/`X2` instead of hardcoding
   immediates. Falls back to legacy MVP defaults
   (`0, 4, 4` for HashMap, `0, 8, 8` for Vec) when
   the args list is empty — older programs without
   an explicit annotation keep their previous
   behaviour.

### regression.

`programming/hashmap-string-keys.zo` —
`HashMap<str, int>` round-trip with insert /
contains_key / get / remove + miss + length. Catches
str-payload hashing.

### out of scope (deferred to follow-up).

- Bool keys (`HashMap<bool, _>`) — `key_kind = 0` works
  since 8-byte zero-extended bool is unique per value,
  but the runtime's prim-byte hash still reads 8 bytes.
  Fine for correctness; investigate hash quality if it
  becomes a hotspot.
- Mixed-size value types (e.g. `HashMap<int, str>`).
  The val_ptr path is uniform 8-byte today; works
  for str pointer + int. Smaller value types
  (struct-by-value) need a follow-up.

## Phase D — `arr.sort()`.

PLAN_AOC's fourth phase ships in-place ascending
sort on `[]int`. The compiler can't lower
`self[i] = ...` inside an apply method body today
(the indexed-assign path goes through a SIR shape
codegen doesn't translate), so the loop runs in
the runtime instead — same codegen-replaced pattern
HashMap / Vec / HashSet already follow.

### approach.

`zo-runtime/src/arr.rs` exports
`_zo_arr_sort_i32(data: *mut i64, len: usize)` —
unstable sort over the raw 64-bit slot buffer
(zo's `[]int` codegen lays each element out in a
register-sized slot regardless of the declared int
width, so the runtime sorts as `i64`).

`std/arr.zo` adds `pub fun sort(self) {}` as a
syntactic anchor; codegen recognizes the mangled
`arr_int::sort` name and emits the FFI directly
(receiver `+ 16` is the data pointer, `*receiver`
is the length — matches the
`[len:8][cap:8][data...]` array layout shared with
`Insn::ArrayLen` / `Insn::ArrayPush`).

`RuntimeNeeds::from_sir` flags the concurrency
dylib for any `arr_int::sort` reference so the
linker pulls in `_zo_arr_sort_i32`.

### regression.

`programming/array-sort.zo` covers shuffled,
already-sorted, reverse-sorted, negatives, and
single-element cases. All round-trip through the
sort and produce the expected ordering.

### known limitation.

An unrelated Mach-O writer layout bug rejects
binaries when the program's specific code/data
shape pushes a section boundary onto a particular
byte. For the sort regression, an 8-element shuffle
trips the bug; 7 elements lay out cleanly. Tracked
for follow-up — the proper fix routes the actual
section sizes through the LoadCommand-emit path
and rounds up to a page boundary (mentioned in
the earlier TEXT-segment-overflow note).

## Phase E — paper cuts.

PLAN_AOC's fifth phase covers the small fixes that
AoC programs trip on. Two of three landed cleanly;
the third (negative `Option<int>` through fn params)
turned out to be a deeper calling-convention issue
and stays as a known limitation.

### char case (E.3).

`std/char.zo` adds `to_uppercase` / `to_lowercase` —
ASCII-only. Letters in `'a'..'z'` map by subtracting
`32`; `'A'..'Z'` map by adding `32`. Everything else
returns unchanged. Regression
(`programming/char-case.zo`) covers the four
direction × in-range / out-of-range cases.

### str search (E.2).

`std/str.zo` adds `index_of(needle: str) -> int` and
`contains(needle: str) -> bool`. Empty `needle`
matches the head (`index_of` returns `0`,
`contains` returns `true`) — Rust / Python
semantics. Returns `-1` when `needle > self.len`.
`contains` is `index_of != -1` for the cheapest
path through the same loop.

`programming/str-contains.zo` wires the regression
through both methods on shuffled / absent /
prefix-overlap inputs.

### `str.replace` deferred.

The byte-loop replace body builds output via `++`
inside a `while` — that loop emits a `StrSlice` and
a Concat per iteration, and the resulting binary
hangs at run time on any non-trivial input.
Suspect: per-iteration string allocations either
exhaust a runtime budget or interact with the
Mach-O writer's section layout the same way
Phase D's 8-element sort case does. A pure-zo
`replace` needs a `StringBuilder`-style runtime
helper to avoid the O(n²) concat path. Tracked as
follow-up.

### `Option<int>` negative payload (E.1).

Confirmed: `parse_int("-42")` returns `Option::Some
(-42)` correctly, and a direct `match` on it prints
`-42`. Passing the same Option through a function
parameter and then matching inside that function
prints garbage (a stack pointer). Root cause is
the calling convention for ≤16-byte aggregates —
ARM64 ABI passes them in two GP registers
(`X0` = tag, `X1` = payload), but zo's codegen
treats `param[0]` as a pointer and emits
`field %0.1` = `LDR Xd, [tag_value, 8]`. The load
dereferences the tag value (which is `0` for
`Some`) and reads garbage from low memory.

Fix is structural — either always pass aggregates
via stack pointer, or special-case the field-load
path for register-passed enums. Out of Phase E
scope; tracked as the calling-convention follow-up
in PLAN_AOC's "out of scope" list.

### regressions.

- `programming/char-case.zo` — `to_uppercase` /
  `to_lowercase` round-trip.
- `programming/str-contains.zo` — `index_of` /
  `contains` on hit / miss / prefix.

## Phase F — AoC 2024 day 1 demo.

PLAN_AOC's final phase ships a real Advent of Code
program through the full pipeline. The demo uses
`text.lines()`, `line.split("   ")`, `parts[i]
.parse_int()`, `Vec<int>` for the left column,
`HashMap<int, int>` for the right column counts,
and a nested `match` over `lhs.get(j)` /
`rhs_counts.get(v)` for the similarity-score loop.

End-to-end on the sample input — the demo prints
`31`. Three structural bugs surfaced during
bring-up; each is fixed at the root in the
appropriate layer.

### Bug 1 — block-bodied match arm fused with next arm.

`match` arms with a block body
(`Pat => { ... }`) don't require a trailing comma
between arms — the closing `}` is the separator.
The executor's `body_end` scan only looked for the
next top-level `Comma`, so it overran the next
arm's pattern and merged both arms' bodies into a
single linear sequence. The first arm's
`Insn::Jump { target: end_label }` was never
emitted between the two bodies, so every match
silently fell through into the next arm regardless
of the discriminant.

`zo-executor/src/executor.rs` — `execute_match`
detects whether the arm body starts with `{`, and
in that case stops `body_end` at the matching
`}` (exclusive end set to `rbrace_idx + 1`). The
non-block path is unchanged. Regression test
`match_block_arm_no_comma_does_not_fuse_with_next_arm`
asserts ≥ 2 `BranchIfNot` and ≥ 2 `Jump` for the
two-arm shape that previously emitted only one of
each.

### Bug 2 — nested match scrutinee slot collision.

The synthetic scrutinee local `__match_scrut__`
was interned with the same name on every match.
Nested matches (`match outer { Some(v) => match
inner { ... } }`) emitted `Store __match_scrut__,
inner_value` inside the outer Some arm, which
overwrote the outer scrutinee storage. Because the
codegen reads enum field offsets via the loaded
pointer to the same slot, the outer's bindings
silently pulled payload bytes from the inner enum.
Same problem on `__match_result__` for nested
expression-position matches.

Fix: suffix both synthetic names with the match's
unique label id (`__match_scrut_<n>__`,
`__match_result_<n>__`). Each match now gets its
own slot; nesting is correct by construction.

### Bug 3 — `emit_str_sp` X16 self-clobber on slow path.

The slow path of `emit_str_sp` (offset > 255
bytes) emitted:

```
add  X16, SP, #offset
str  src,  [X16, #0]
```

But several callers in `emit_vec_get`,
`emit_map_get`, `emit_map_remove`, `emit_vec_pop`,
and the struct-return helpers pass X16 itself as
`src` (X16 was just used to compute a freshly-
built tag/payload word). The `add` clobbered
the value before the STR consumed it, so the
target slot ended up holding `SP + offset`
instead of the discriminant. The match arms then
saw a discriminant that was neither 0 (Some) nor
1 (None) and skipped both arms.

Why it surfaced in AoC and not the smaller
programs: the bug fires only once
`struct_base + next_struct_slot > 255`. The
smaller demos stay under 255 and use the inline
fast path; AoC's main pushes ~10 `HashMap` /
`Vec` call-site scratch slots, which lifts the
score loop's `Vec::get` to offset 0x158.

Fix: `emit_str_sp` picks a non-aliasing scratch —
X17 when `src` is X16, X16 otherwise. Both lie
outside the caller-save save/restore window, so
neither carries cross-call live values. The new
test `test_str_sp_slow_path_does_not_self_clobber`
builds a synthetic SIR program (Vec/HashMap calls
sized to push past the inline range) and scans the
emitted bytes for any `STR Xn, [Xn, #0]`
self-store.

### Bug 4 — `emit_add_sp_offset` corrupts X16 past 4095.

Bug 3 was incomplete. `emit_str_sp`'s slow path
correctly switched scratch to X17 when `src ==
X16`, but it then called `emit_add_sp_offset(X17,
offset)` which itself materialized the immediate
into a hard-coded X16 (`mov X16, #imm; add dst,
SP, X16`). When `offset > 4095`, this clobbered
any caller-held value in X16 — including the
discriminant the caller had just built and was
about to store.

Surface: rewiring AoC day 1 to read input via
`read_file("aoc-2024-d01.input")` shifted the
struct cursor: `read_file`'s 4096-byte buffer
pushed every subsequent option-tag store past
4095. Six `HashMap::insert` calls in the parsing
loop wrote the constant `opt_base` (4280) instead
of the discriminant; both `Option::Some` and
`Option::None` arms compared against 4280 and
fell through. `rhs_counts.len() == 0`, score == 0.

Fix: `emit_add_sp_offset` materializes the
immediate into `dst` itself (`mov dst, #off; add
dst, SP, dst`). The post-fix invariant — the
helper clobbers no register other than `dst` — is
robust by construction. Several `emit_*_sp` /
`emit_*_fp_sp` helpers were also widened: the
underlying ARM64 STR/LDR (immediate, unsigned,
64-bit) supports a scaled imm12 reaching 32760
for 8-byte-aligned offsets, so most call sites
that previously took the slow path now take a
single instruction.

Audited and converted every `emit_str(... SP
...)`, `emit_ldr(... SP ...)`, `emit_str_fp(... SP
...)`, `emit_ldr_fp(... SP ...)`, and `emit_add_imm
(reg, SP, off)` call site that uses a dynamic
offset — they previously silently truncated
offsets above the inline range. Spill ops,
parameter spilling, struct / tuple / enum / array
literal construction, struct-return copy,
ChannelSend / Recv, SelectWait / Recv, math
intrinsic save, user-call caller-save, extern-call
caller-save, and the Vec realloc path now route
through the safe helpers.

The new test
`test_emit_add_sp_offset_uses_dst_not_x16_in_slow_path`
builds a synthetic SIR program (`read_file ->
HashMap::new -> HashMap::insert`) that forces the
slow path and scans for any `add Rd, SP, X16`
where `Rd != X16` and `Rd != SP` — the residual
self-clobber pattern.

### test runner — cwd per test.

`zo-test-runner` now sets `current_dir` to each
program's source directory (and to each project's
root for multi-file projects) before launching the
compiled binary. Path-relative I/O tests (e.g.
`read_file("name.input")`) work without `/tmp`
shims. `programming/048-io-read-file.zo` updated
to its basename-relative input path.

### regressions.

- `programming/aoc-2024-d01.zo` — reads its
  `.input` file from disk, prints `31`.
- `tests::matching::match_block_arm_no_comma_does_not_fuse_with_next_arm`
  — block-body arm separation.
- `tests::test_str_sp_slow_path_does_not_self_clobber`
  — slow-path scratch register aliasing.
- `tests::test_emit_add_sp_offset_uses_dst_not_x16_in_slow_path`
  — `add_sp_offset` clobbering only `dst`.

## `showln(HashMap<K, V>)` — Rust-style pretty print.

`showln(my_map)` was leaking the receiver's struct
pointer (`6128755224`) because `emit_typed_write`
fell through to `itoa` for any value it didn't
specifically recognize. Now it dispatches to a
runtime `_zo_map_show` helper that walks the
slot table and writes `{k0: v0, k1: v1, ...}` —
order is bucket order, identical to Rust's
`HashMap` Debug. Iteration order varies across
runs and that's by design.

### SIR / executor.

New `Insn::MapTyDef { map_ty, key_fmt, val_fmt }`
mirrors `Insn::ArrayTyDef`. Emitted once per
`HashMap<K, V>` binding the executor sees. The
fmt discriminants come from `MapFmt` (`Int = 0`,
`Bool = 1`, `Char = 2`, `Str = 3`, `Float = 4`)
shared with the runtime over the FFI boundary.
The validator / DCE / pretty-printer get carve-
outs alongside the existing definitional insns
(`EnumDef` / `StructDef` / `ArrayTyDef`).

### codegen.

A new `map_metas: HashMap<u32, (u32, u32)>` is
populated by the same pre-pass that fills
`array_metas`. `is_map_value` returns the
`(key_fmt, val_fmt)` pair so `emit_typed_write`
routes HashMap values to `emit_map_write`, which
loads `m.ptr`, marshals the kinds into X2 / X3,
and calls `_zo_map_show` via the standard
caller-save extern path.

### runtime (`_zo_map_show`).

Iterates `Slot::Occupied` entries, formats `key:
value` per side via `MapFmt::format_bytes`, and
writes the buffered output through one
`libc::write` syscall. `MapFmt::Str` for value
slots dereferences the spilled str header
pointer; for key slots it reads the stored
payload bytes directly.

## HashMap str-key double-indirection.

`hashmap.insert("alice", 100)` hung in
`hash_bytes` — multi-GB FNV-1a loop. The codegen
spills the str header pointer into a scratch slot
and passes `&slot` as `key_ptr` (uniform for
every key kind). The runtime's `KeyKind::Str`
branch was calling `str_bytes(key_ptr)` directly,
treating the slot's pointer-bytes as a `len`
prefix and walking past the end of memory.

Fix: `hash_key` / `key_to_vec` / `key_eq` now do
one extra dereference for `KeyKind::Str` —
`*(key_ptr as *const *const u8)` yields the
header pointer, then `str_bytes` walks the
payload. The runtime API contract is now uniform:
`key_ptr` always points at the slot, `Prim` /
`Tuple` slots ARE the key, `Str` slots HOLD a
key. Existing unit tests (`str_keys_hash_by_
content_not_pointer`,

## `zo run` — never make the user think about the dylib again.

Two changes that together kill the "compiler
hangs" complaint class.

### `stage_runtime_artifacts` reads from `deps/`.

Cargo only refreshes `target/<profile>/<dylib>`
when the cdylib's owning package is built
directly (`cargo build -p zo-runtime`). Transitive
builds (`cargo run --bin zo`, `cargo build --bin
zo`) refresh `target/<profile>/deps/<dylib>` but
leave the sibling stale. Users running `cargo run
--bin zo -- run program.zo` after editing the
runtime would dyld-hang their binary in
`compatibleSlice` against the previous runtime
version. Staging now prefers `deps/<dylib>` when
it exists (in-tree builds) and falls back to the
sibling for installed binaries.

### `zo run` per-run isolated dir.

The driver was staging every program's binary
plus `libzo_runtime.dylib` into `std::env::
temp_dir()` directly. Concurrent runs and prior
dyld-stuck zombies kept handles to the same
`libzo_runtime.dylib` path; each new run had to
mmap into the same poisoned location. `zo run`
now creates `<temp>/zo_run_<pid>/` per
invocation, drops the binary as `a.out` plus the
fresh dylib next to it, and removes the whole
directory on exit. Old zombies in the parent
temp dir can no longer poison new launches.

### release tarball ships the runtime dylib.

`.github/workflows/release.yml` previously packaged
only `zo` and `fret` binaries. An installed `zo`
(via `tasks/zo-install.sh`) had no
`libzo_runtime.dylib` next to the binary, so any
program needing the runtime — concurrency, HashMap
/ HashSet / Vec, str slicing — failed at startup
with `dyld: Library not loaded: @executable_path/
libzo_runtime.dylib`. The Unix package step now
also copies `libzo_runtime.dylib` (macOS) /
`libzo_runtime.so` (Linux), and the Windows step
copies `zo_runtime.dll`. The install script's
existing `tar xz -C "$INSTALL_DIR"` lands the
dylib sibling-to-`zo` automatically, where
`stage_runtime_artifacts`'s installed-binary
fallback resolves it.

## generic enum payload — str header pointer leak.

`showln(Maybe<str>::Some("hi"))` printed `Maybe::Some
(4341608703)` — the str header pointer formatted as an
integer instead of the `"hi"` payload.

### root cause.

`enum_metas` is keyed by the generic enum's `TyId.0` and
populated from a single `Insn::EnumDef`, whose
`variants[i].field_tys` carries the enum template's
`Ty::Infer($T)` field types. The executor monomorphizes
`Maybe<int>` and `Maybe<str>` independently, but every
construction lands at the same `Insn::EnumConstruct.ty_id`
(the generic enum's `Ty::Enum(eid)`), so the pretty-printer
saw only the template's generic types. `emit_field_write`
then asked `is_str = ty_id.0 == STR_TYPE_ID` against the
generic var's `TyId`, the check failed, and the str payload
fell through to `emit_itoa_and_write`.

### fix.

`zo-codegen-arm/src/codegen.rs` —
`value_enum_field_tys: HashMap<u32, (u32, Vec<TyId>)>` is
populated at every `Insn::EnumConstruct` from the concrete
field types already in `value_types[fields[i].0]` (each
field's payload type was registered at the field's own
defining insn). The override is keyed by the construction's
`dst.0` ValueId. Propagated through `Insn::Store { name }`
into a parallel `local_enum_field_tys` keyed by
`name.as_u32()`, then back onto `Insn::Load`'s `dst` so a
later `showln(local)` recovers the construction-site types.
`emit_enum_write` looks up the override for the value being
printed and substitutes the matching variant's `field_tys`
before the cmp-chain runs; other variants stay generic
(harmless — their tags can never match at runtime).

Cleared `local_enum_field_tys` per `Insn::FunDef` to keep
each function's locals isolated. `value_enum_field_tys` is
keyed by ValueId and ValueIds are monotonic across the SIR,
so it doesn't need clearing.

### regression.

- `programming/generics-enum.zo` — golden output updated
  from `Maybe::Some(4341608703)` to `Maybe::Some(hi)`.
- `tests::test_generic_enum_str_payload_does_not_leak_pointer`
  — a `Maybe<str>::Some("hi")` SSA chain bakes the literal
  `"hi"` into the artifact (proving the str payload was
  not formatted through the integer path).
- `tests::test_generic_enum_int_then_str_dispatches_independently`
  — int followed by str instantiation in one function
  exercises the override + propagation path end-to-end.

### out of scope.

Enum values that arrive without a local `Insn::EnumConstruct`
(e.g. a function parameter `m: Maybe<str>`, an FFI return
value) still fall back to the generic template. The fix
covers every locally-constructed enum, which is the case
the bug surfaced through. A follow-up would teach
`Insn::Call` returning an enum (and `Insn::FunDef`'s param
slots) to seed `value_enum_field_tys` from the resolved
generic args the executor already maintains in
`var_return_type_args`.

### suite.

28 + 2 / 30 zo-codegen-arm, 296 / 296 zo-executor, 337 /
337 integration.

## nested-context str quoting (Rust Debug-style).

`showln(Maybe::Some("hi"))` printed `Maybe::Some(hi)` —
unambiguous when you know the contained type, but
indistinguishable from `Maybe::Some(hi_ident)` if the
payload were a free identifier. Fixed: nested str values
inside enums, arrays, and HashMap entries now wrap their
payload in `"`s, matching how Rust's `Debug` renders
collections. The top-level `showln(s)` path keeps writing
the bare payload — that's the `Display` convention zo
already uses for primitives.

### codegen.

`emit_field_write` (`zo-codegen-arm/src/codegen.rs`)
gains a `quoted: bool` parameter. The str branch wraps
the payload syscall with two `STR_DQUOTE_SYM` writes
when `quoted == true`, saving the payload pointer in
X23 (outside the array / enum writer's X19..X22
working set) across the punctuation syscall. Both
existing callers — `emit_enum_write` and
`emit_array_write` — pass `quoted = true`.
`emit_typed_write` is the one top-level caller; it
hits the str path BEFORE `emit_field_write` and still
writes unquoted, no behaviour change there.

### runtime.

`MapFmt::format_bytes` (`zo-runtime/src/map.rs`) for
the `Str` branch now pushes `"` before and after the
payload bytes for both keys and values. The map
formatter only ever runs in nested context (inside
`{ k: v }`), so it doesn't need a flag — the quoting
is always on.

### regressions.

- `programming/generics-enum.zo` — `Maybe::Some("hi")`.

## `showln(Struct)` — Rust-style pretty print.

`showln(my_point)` was leaking the struct's stack base
pointer because `emit_typed_write` had branches for
enums, arrays, maps, and tuples — but no branch for
structs, so they fell through to `itoa`. Now it
dispatches to `emit_struct_write` which produces
`Point { x: 10, y: 20 }`, matching Rust's `Debug` for
structs. The user-facing trigger was a struct printed
under an overloaded `==` operator path
(`programming/abstract-eq.zo`).

### codegen.

`zo-codegen-arm/src/codegen.rs` —
`struct_metas: HashMap<u32, StructMeta>` is populated
on each `Insn::StructDef`. Each entry pre-bakes the
`"Name { "` header and per-field `"field: "` labels
into `string_data` under synthetic symbols (same
`ENUM_SYNTHETIC_SYM_BASE` counter as the enum
pretty-printer), so the per-call work is just a
sequence of `emit_synthetic_str_write` + per-field
`LDR X0, [X19, #i*8]` + `emit_field_write` dispatch.
The struct base lives in X19 (callee-saved, outside
the allocator pool) for the duration of the walk so
write syscalls don't lose it.

`is_struct_value` mirrors `is_enum_value` — a
`TyId` lookup against `struct_metas`. The struct
branch in `emit_typed_write` lands between the tuple
branch and the integer fall-through, so the fall-
through now only fires for genuinely scalar values.

`STRUCT_CLOSE_BRACE_SYM` (`" }"`) is shared across
every struct (the close marker is name-independent);
the `", "` separator reuses `ENUM_COMMA_SPACE_SYM`.

### regression.

`programming/abstract-eq.zo` — its previously-
commented `showln(a)` is now exercised. Expected
output updated from a single `DONE` line to
`Point { x: 10, y: 20 }` followed by `DONE`. Catches
any future regression where struct values fall back
to the integer printer.

### known limitation.

Nested non-scalar fields (struct-in-struct,
enum-in-struct) still go through `emit_field_write`'s
itoa fallback — same shape as the existing tuple /
array printers. A recursive field-write would unify
all four pretty-printers; tracked as a follow-up.

### suite.

43 / 43 zo-codegen-arm, 337 / 337 integration green.

## nested-array / nested-struct pretty-print.

`showln(grid)` for `[2][3]int` printed
`[6162310648, 6162310688]` — two pointers to inner row
buffers — because `emit_field_write` only knew how to
format str/float/bool/char/int. Every non-scalar field
type fell through to `emit_itoa_and_write`. Same bug
hit `[][][]int` (3D array) and would hit `[]Point` /
`Outer { p: Point }` once those programs grew.

### codegen.

`zo-codegen-arm/src/codegen.rs` — `emit_array_write`
and `emit_struct_write` are split into a top-level
entry that loads the receiver into X19 and a
`*_walk_from_x19` body that assumes the base is
already in X19. The walk body becomes the recursion
entry point.

`emit_field_write` gains a leading dispatch: when the
field type is itself in `array_metas` (any depth) or
`struct_metas`, it pushes the outer X19..X22 onto the
stack with pre-indexed STP, copies X0 into X19, calls
the matching `*_walk_from_x19`, and pops X19..X22 back
with post-indexed LDP. Each level of nesting consumes
32 bytes of stack, so 3D arrays use ≤ 96 bytes —
trivial.

The push/pop helpers (`emit_pp_state_push` /
`emit_pp_state_pop`) are two-instruction primitives
each, exploiting STP's pre-indexed and LDP's post-
indexed addressing modes. This is the canonical ARM64
push/pop pattern; the underlying encodings were
already wired (`STP_PRE` / `LDP_POST` in
`zo-emitter-arm/src/arm.rs`).

- `programming/array-3d.zo` — golden output

## `mut self` enforced at the call site.

`imu m: HashMap<int, int> = HashMap::new(); m.insert(1, 100);`
silently mutated the heap-allocated map's slot table. zo's
mutability checker tracks bindings and `self.field op= …`
inside method bodies, but the call-site receiver against
the callee's `self` parameter was unenforced — so a `mut
self` method called on an `imu` binding sailed through.
The runtime types (`Vec`, `HashMap`, `HashSet`, `[]int::
sort`) are all `struct { ptr: int }` shapes that mutate
heap memory through the pointer, completely outside the
mutability checker's view.

### parser already records it.

`execute_fun_def` in `zo-executor/src/executor.rs`
already stamps `Mutability::Yes` on the `self`
parameter when the source says `mut self`
(`struct-stateful.zo:14`'s `fun incr(mut self)` proves
the parse path). The information was just being dropped
when the FunDef landed in `self.funs`.

### threading the bit through.

`Insn::FunDef` (`zo-sir/src/sir.rs`) and `FunDef`
(`zo-value/src/value.rs`) gain a `mut_self: bool`. Set
to `true` only when `params[0]` is
`(SELF_LOWER, _, Mutability::Yes)` — non-method funs and
plain-`self` methods stay `false`. The flag survives
module-resolver imports (`zo-module-resolver/src/
exports.rs` copies it across when std re-exports its
SIR), so std's `pub fun insert(mut self, …)` arrives in
the user's executor with `mut_self = true` already set.

### check at the dot-call site.

`execute_dot_method_call` reads
`dot_method_recv_ty[dot_idx]` (already populated for
mono dispatch) for the receiver symbol. When
`func.mut_self` and the receiver resolves to a `Local`
with `Mutability::No`, it reports
`ErrorKind::ImmutableVariable` at the `.` token. Chained
receivers (`f().m()`, `s[0].m()`) carry no recoverable
binding and stay exempt — the language can't see a
binding to check, same as it can't reject the original
expression's intermediate.

### std updates.

`std/map.zo` — `insert` / `remove` / `free` flipped to
`mut self`.
`std/vec.zo` — `push` / `pop` / `set` / `free`.
`std/set.zo` — `insert` / `remove` / `free`.
`std/arr.zo` — `sort`.

`get` / `contains` / `contains_key` / `len` / `is_empty`
stay as plain `self` — they don't mutate the heap.

### test sweep.

Seven `programming/*.zo` binaries had `imu` bindings on
container types they then mutated. Flipped to `mut`:
`hashmap-basic.zo`, `hashmap-len-is-empty.zo`,
`hashmap-string-keys.zo`, `hashset-basic.zo`,
`vec-basic.zo`, `vec-str.zo`, `struct-stateful.zo`.
The last one was the canary — `imu counter; counter.
incr()` (where `incr(mut self)` exists since the
beginning of apply-method support) compiled fine before
this change, exposing how wide the hole was.

The new error is the existing
`Cannot mutate immutable variable` with the
`Use 'mut' to declare a mutable variable` hint anchored
at the `.` — same shape as the field-store check in
`finalize_pending_compound`, single source of truth for
the user-facing message.

### suite.

1005 / 1005 unit tests, 337 / 337 integration green;
clippy clean.

## `for x := arr { ... }` and `=> expr;` — array iteration.

`for word := words { showln(word); }` parsed but emitted
nothing — `execute_for` looked for `..`/`..=`, didn't
find one, and silently `return`ed. The body then ran in
the main walk with `word` undeclared, surfacing four
"undefined variable" errors. Programs with a `[N]T` /
`[]T` they wanted to iterate had to fall back to
`while i < arr.len { … arr[i] … }` boilerplate.

### parser/header.

`execute_for`'s header scan already collects
`var_name`, `colon_eq_idx`, `body_start_idx`, and
`body_is_fat_arrow` regardless of whether a range op
appears. The `range_idx is None` case now dispatches
to a new `execute_for_array` sibling instead of
returning silently. The range path is unchanged.

### lowering.

`execute_for_array` evaluates the rhs (which produces
the array `ValueId` + `TyId` on the stacks), peels
the element type out of `Ty::Array(arr_id).elem_ty`,
and emits:

```text
__for_arr_<n>__ = <rhs>     // VarDef + Store, ty = []T or [N]T
__for_idx_<n>__ = 0         // VarDef, ty = int
loop_label:
  tmp_arr = Load(__for_arr)
  tmp_len = ArrayLen(tmp_arr)
  tmp_idx = Load(__for_idx)
  cond    = tmp_idx < tmp_len
  BranchIfNot cond -> end_label
  arr_2   = Load(__for_arr)
  idx_2   = Load(__for_idx)
  elem    = ArrayIndex(arr_2, idx_2)
  Store user_var = elem    // user_var is mut, ty = elem_ty
  <body>
  // close path: idx += 1, jmp loop_label
end_label:
```

Both block (`{ … }`) and line (`=> expr;`) bodies are
supported via the same `body_is_fat_arrow` branch the
range path uses. The block form hands off to the main
loop and closes through the `BranchKind::For` arm of
the `RBrace` handler; the line form inline-finalizes
the same way the range path does.

### loop-close fork.

The `BranchKind::For` close path increments
`for_var` (the user's binding). For range form
that's the integer counter — correct. For array
form `for_var` is the element binding (re-stored
each iter from `arr[idx]`) and incrementing it is
nonsense. Added `for_idx_var: Option<Symbol>` to
`BranchCtx`; the close path reads `for_idx_var.or
(for_var)` so range form (with `for_idx_var = None`)
keeps incrementing `for_var`, while array form
(with `for_idx_var = Some(__for_idx)`) increments
the synthetic counter and leaves the user's
element binding alone.

### user-var mutability.

The Ident handler's immutable-load path requires
`Local.sir_value.is_some()` before emitting a `Load`
— the array-iter element binding has no meaningful
pre-loop value. Marked the user's loop var as
`Mutability::Yes` so the mut-load path fires
unconditionally with `LoadSource::Local(sym)`, which
is what the per-iter `Store` writes to. Costs
nothing — the binding really IS rewritten every
iteration.

## `[]T = [a, b, c]` — heap-allocate non-empty literals.

`mut arr: []int = [0, 0, 0]; arr.push(10);` hung
indefinitely. Root cause: `Insn::ArrayLiteral`
codegen had two paths — empty literal `_malloc`'d
into the heap, non-empty literal stack-allocated at
`SP + base`. `Insn::ArrayPush` then called
`_realloc(arr_pointer, new_size)` on the stack
address. **`realloc` on a stack pointer is UB** in
libc; on macOS it spins.

The mismatch was structural: every `[]T` is a
dynamic-growable array by type, and stack-allocating
non-empty literals contradicts that contract — the
"no push after non-empty literal" common case
masked it for the entire test suite.

### codegen.

`Insn::ArrayLiteral` (`zo-codegen-arm/src/codegen.rs`)
unifies on the heap path. `_malloc(header + cap *
8)`, store `len`, `cap`, then `STR` each element to
`base + 16 + i*8`. `cap = 1024` for empty literals
(amortizes 1024 pushes), `cap = len` for non-empty
literals (one realloc on first push, then
doubles).

`[N]T` static-array stack-allocation as a separate
optimization is deferred — needs the array's
`size: Option<u32>` to flow through
`Insn::ArrayTyDef` so codegen can distinguish
`[3]int` from `[]int` and pick the right path.
For now both heap-allocate. The cost is one
`_malloc` per literal — a known regression on
hello-world-shape programs but uniformly correct,
and `[N]T` literals are rare in real code (most
arrays are `[]T`).

### `has_calls` propagation.

The register allocator only flagged `has_calls = true`
for **empty** `ArrayLiteral`s (the ones that used to
malloc). With every literal now calling malloc, the
guard was a stale subset — a function whose only
extern call came from a non-empty literal got a leaf
prologue (no FP/LR save), so `bl _malloc` clobbered
LR and `ret` jumped back to its own continuation
point — infinite loop on the same `RET`.
`zo-register-allocation/src/allocator.rs` now drops
the `if elements.is_empty()` guard.

### caller-save coverage.

`emit_extern_call` saved only `X9..X15` (the
allocator's spill scratch range, 7 regs). The
allocator may place live values in `X1..X8` too —
e.g., `[a, b, c, d, e, f, g, h]`'s 8th element value
landed in `X1`, and `bl _malloc` clobbered it,
silently corrupting the array. Bumped
`CALLER_SAVE_COUNT` from 7 to 15 (X1..X15);
`CALLER_SAVE_RESERVE` from 56 to 120. The frame cost
is +64 bytes per extern-call-bearing function —
negligible. X0 stays out (it's the BL's first arg
slot); X16/X17 are AAPCS intra-procedure scratch and
aren't expected to survive any call. The bump
applies to every extern call, not just `_malloc`,
so future regressions of the same shape are
prevented by construction.

### regressions.

- `programming/array-static.zo` — `for word := words
  { showln(word); }` over a `[2]str`.
- 5 previously-broken integration tests with
  array-pop / array-store / std-arr / rule-110 /
  remove-duplicates flows that all hit the
  realloc-on-stack hang.
- `programming/concurrency-select-pick.zo` golden
  updated to include `DONE` — the original golden
  was passing only because the buggy frame layout
  let the program terminate before reaching the
  trailing `showln`.

### suite.

1005 / 1005 unit tests, 334 / 336 integration
green. The 2 remaining failures (`array-store-
basic`, `while-early-return`) are user-side WIP
edits: `EXPECTED OUTPUT` blocks declare a `DONE`
line that the corresponding source files never
print. Out of scope for this sweep.

Clippy clean.

## `showln(Vec)` / `showln(HashSet)` — Rust-style pretty print.

`showln(my_vec)` was leaking `Vec { ptr: <int> }` because
`Vec<$T>` is structurally `struct Vec<$T> { ptr: int }`
and the new struct pretty-printer (which lands earlier in
this changelog) walked its single int field. Same issue
with `HashSet<$K>`. The fix follows the path
`showln(HashMap)` already uses: emit a per-call-site
type-def insn carrying the format kind, route the
typed-write to a runtime helper that walks live entries.

### SIR.

`Insn::VecTyDef { vec_ty: TyId, elem_fmt: u32 }` and
`Insn::SetTyDef { set_ty: TyId, key_fmt: u32 }` mirror
`MapTyDef`. Both emitted once per `Vec::new` /
`HashSet::new` call when codegen has the resolved `$T` /
`$K` from `decl_annotation_args` (the same per-call-site
derivation the existing HashMap path uses). The
validator / DCE / pp / value-id / type-id walkers all
get carve-outs alongside the existing definitional
insns.

### codegen.

Two new metadata maps in `ARM64Gen`:

```text
vec_metas: HashMap<u32, u32>   // TyId -> elem_fmt
set_metas: HashMap<u32, u32>   // TyId -> key_fmt
```

Populated by the same pre-pass that fills
`array_metas` / `map_metas`. `is_vec_value` /
`is_set_value` lookups in `emit_typed_write` land
BEFORE `is_struct_value` so the struct walker never
fires for these — without that ordering, the struct
printer's `Vec { ptr: <int> }` output shadows the
container view.

`emit_vec_write` / `emit_set_write` mirror
`emit_map_write`: receiver register holds the
`{ ptr }` struct address, load `v.ptr` into X0, set
`X1 = fd`, `X2 = elem_fmt` (or `key_fmt`), and call
the matching runtime helper.

### runtime.

`zo-runtime/src/vec.rs::_zo_vec_show` walks
`v.bytes` in `elem_sz`-byte chunks for `i in 0..v.len`,
formats each slot via `MapFmt::format_bytes(_, true,
…)` so str payloads dereference their header pointer
(value-side semantics, matching `_zo_map_show`),
emits a single `libc::write` of the buffered output.
Output: `[e0, e1, ...]`.

`zo-runtime/src/map.rs::_zo_set_show` walks the
underlying `ZoMap` (sets reuse the map allocator with
`val_sz = 0`), formats only the key half via
`format_bytes(_, false, …)`. Output: `{k0, k1, ...}`.

Both helpers route automatically through
`libzo_runtime.dylib` because their export names
start with `zo_`, matching the `ZO_RUNTIME_SYMBOL_PREFIX`
the codegen uses to split runtime exports from
libSystem.

`MapFmt::from_u8` and `MapFmt::format_bytes` are now
`pub(crate)` — no API surface change; just makes the
formatter reusable across `vec.rs` / `map.rs` without
duplicating the case-by-case scalar formatting.

- `programming/vec-str.zo` — `Vec<str>` round-trip
  with three pushes; golden updated to
  `["alice", "bob", "carol"]` followed by `DONE`.
- Inline reproducer (Vec<int> + HashSet<int>) verifies
  `[10, 20, 30]` and `{7, 42}`.

## `loop { ... }` — unconditional repeat with `break`.

```zo
mut x: int = 0;
loop {
  if x == 3 { showln(x); break; }
  x += 1;
}
```

ran the body exactly once and exited with `x = 1`,
silently. Two simultaneous bugs:

1. `Token::Loop` was registered as a statement-starting
   token but had **no executor handler**. The main walk
   hit it, fell through with no diagnostic, and treated
   the body's `{ ... }` as a bare block — one pass, no
   back-edge.
2. `Token::Break` only matched
   `BranchKind::While | BranchKind::For`. There was no
   `BranchKind::Loop`, so even with a handler the
   `break` would have been a no-op.

### approach.

Desugar to `while true`. The close semantics
(`Jump loop_label; Label end_label`) are identical to
the while form, just without a condition to test;

### executor.

`zo-executor/src/executor.rs` — new `execute_loop`
mirrors `execute_while`'s skeleton: emit
`Insn::Label { id: loop_label }`, push a `BranchCtx`
with `kind: BranchKind::While` and
`loop_label: Some(loop_label)`. The single difference
is `branch_emitted: true` from the start — there's no
condition expression to evaluate, so the `LBrace`
handler must NOT try to pop a value from the stack
expecting a `cond_sir` boolean. Setting the flag
upfront short-circuits the
"if `!ctx.branch_emitted` then emit `BranchIfNot`"
path that fires for `if` / `while` / `for`.

`Token::Loop` arm in `execute_node` dispatches to
`execute_loop(idx, children_end)`, parallel to
`Token::While` and `Token::For`.
