# ZO-CL10 ‚Äî 14-04-2026.

## feature ‚Äî module system with `pack`/`load` gate.

Modules now require declaration in `lib.zo` via `pub pack`.
`load foo;` looks up the module table (populated from lib.zo
packs) before falling back to filesystem resolution. This
matches Rust's `mod`/`use` two-phase model.

### how it works.

```
myproject/
  lib.zo         # pub pack mylib;
  main.zo        # load mylib;
  mylib.zo       # pub fun add(a: int, b: int) -> int { ... }
```

1. Compiler discovers `lib.zo` next to the input file
2. Scans `lib.zo` for `pub pack` declarations
3. For each pack: compiles the `.zo` file, extracts exports,
   stores in `module_table: HashMap<Symbol, ModuleExports>`
4. User `load` statements check `module_table` first
5. If lib.zo exists but module not declared ‚Üí error:
   "module `foo` not declared ‚Äî add `pub pack foo;` to lib.zo"
6. If no lib.zo ‚Üí filesystem fallback (single-file projects)

### bugfix ‚Äî function return type detection (zo-codegen-arm).

`showln(fn_returning_str())` printed a raw pointer instead of
the string. Root cause: `is_string_value`, `is_bool_value`, and
`is_char_value` didn't match `Insn::Call { ty_id, .. }`. Function
return values were unrecognized and fell through to `itoa`.

- add `Insn::Call { ty_id, .. }` to `is_string_value` (str)
- add `Insn::Call { ty_id, .. }` to `is_bool_value` (bool)
- add `Insn::Call { ty_id, .. }` to `is_char_value` (char)
- `is_float_value` already had it ‚Äî no change needed
- moved `01-simple` and `02-test-project` from wip/ to top level
