# ZO-CL24 — 2026-04-30

## `[]T` annotations accept user-defined element types.

```zo
struct Todo { text: str, done: bool }

mut todos: []Todo = [];   -- previously rejected as type mismatch
```

`resolve_array_type` only accepted the element-type slot
when `token.is_ty()` was true — `int`, `bool`, etc. User
types are tokenized as `Token::Ident`, so `[]Todo`,
`[3]Point`, etc. fell through with no element type, the
annotation collapsed to unit, and the unify at the decl
site reported a confusing `Type mismatch`.

### approach.

Extend the leading-token check in `resolve_array_type` to
also accept `Token::Ident` and `Token::SelfUpper` — the
same set already accepted by `resolve_type_token`. The
existing user-type lookup path inside `resolve_type_token`
(line 5436) handles the resolution via
`resolve_ty_symbol`.

## `!self.field` / `!recv.field` no longer crash.

```zo
fun toggle(mut self) {
  self.done = !self.done;   -- "Undefined variable" before
}
```

Postorder for `!f.on` was emitting `f, on, Bang, Dot` —
the executor's `is_dot_member` check at the field ident
looks at `idx + 1` for `Dot`; finding `Bang` there meant
the lookup fell to the regular-variable path and the
field name (`on`) was reported as an undefined variable.

### approach.

The parser's operand drain decided whether to emit pending
unary prefixes based only on the **next** token. For
`!f.on`, by the time `on` is processed the `Dot` is
already on the operator stack — but `next` is `;`, so the
drain fired prematurely and `Bang` landed before `Dot`.

Two changes in `zo-parser`:

- Operand drain now also checks `operator_stack.last()`.
  If a `Dot` is pending, the unary stays held back.
- `flush_expr` drains any leftover unary prefixes after
  the binary-operator commit pass, so deferred `Bang`s
  reliably land at the very end of the postfix sequence.

Postorder is now `f, on, Dot, Bang` — the shape the
executor expects.

## `self.field = expr` lowers to `Insn::FieldStore`.

The previous `pending_field_assign` path looked up the
receiver by walking SIR backwards for a
`LoadSource::Local`. `self` lowers to `LoadSource::Param(0)`,
not `Local(SELF)`, so the search came up empty and the
assignment fell through as a silent no-op inside `apply`
methods. The compound form `self.x += 1` already worked
because its finalize pulls the receiver from the parse
tree; the simple-assign path didn't.

### approach.

Pull the receiver symbol straight from the parse tree
(`recv_idx = target_idx - 2`) — `Token::SelfLower` resolves
to `Symbol::SELF_LOWER`, `Token::Ident` to its symbol
node-value. Both reach `finalize_pending_field_assign`,
which emits `Insn::FieldStore` regardless of param-vs-local.

## array destructuring — `imu [a, b, c] = arr`.

```zo
imu arr: [3]int = [10, 20, 30];
imu [a, b, c] = arr;
-- a = 10, b = 20, c = 30
```

Mirrors the existing `imu (a, b, c) = tup` and
`imu { x, y } = point` pattern forms. Works on both
`[N]T` (compile-time arity check) and `[]T` (no length
guard — first N elements bound).

### approach.

- **parser** — `handle_binding_keyword` accepts
  `Token::LBracket` as a fourth pattern opener alongside
  `Ident`, `LParen`, `LBrace`.
- **executor.begin_decl** — pattern detection extended to
  `LBracket`, closer mapped to `RBracket`. Names and
  optional `: T` annotation collected the same way as
  tuple/struct.
- **executor.finalize_tuple_pattern_decl** — new
  `Ty::Array(aid)` branch validates static-size arity
  against `names.len()` (errors `ArraySizeMismatch` on
  mismatch); per-name emission synthesizes a fresh
  `Insn::ConstInt` for the position then
  `Insn::ArrayIndex` (instead of `TupleIndex`, which
  expects no `[len:8][cap:8]` header offset).

## `Vec<T>::remove(idx) -> Option<T>`.

```zo
mut v: Vec<int> = Vec::new();
v.push(10); v.push(20); v.push(30); v.push(40);

match v.remove(1) {
  Option::Some(x) => showln(x),       -- 20
  Option::None    => showln("oob"),
}
```

Mirrors `Vec::get` but mutates: shifts the tail down by
one slot, decrements `len`. O(n) in the tail length;
returns `Option::None` on out-of-bounds.

### approach.

- **runtime** — `_zo_vec_remove(vec, idx, val_out) -> bool`
  in `zo-runtime/src/vec.rs`. Single `as_mut_ptr` base,
  `copy_nonoverlapping` to `val_out`, `copy` (memmove) for
  the tail shift.
- **std/vec.zo** — `__zo_vec_remove` ffi + `pub fun
  remove(mut self, idx: int) -> Option<$T>` codegen-
  replaced.
- **arm codegen** — `Vec::remove` dispatch routes to a
  new shared helper `emit_vec_option_idx_call`. `Vec::get`
  was refactored onto the same helper — both follow the
  runtime's `(ptr, idx, val_out_ptr) -> bool` ABI and
  build the same `Option<T>` aggregate.

### the load-bearing bug.

`zo-register-allocation/src/allocator.rs` carries a
per-method scratch-slot budget for the Vec/HashMap/
HashSet intrinsics. The function frame is sized from
this table; if a method is missing, the codegen's
`next_struct_slot += N * 8` writes land *past* the
function's reserved frame, clobbering the caller-save
area / saved LR.

`Vec::remove` was missing. Codegen reserved 3 slots per
call, allocator reserved 0. Symptom: program ran main's
body cleanly (output appeared on TTY), then `ret` jumped
to a garbage address — kernel fault during exit, process
stuck in `UE` (uninterruptible-exiting) state. The test
runner saw stdout never EOF and reported a 10 s timeout.

The fix is one line:

```rust
"Vec::remove" => struct_slots += 3,
```

`Vec::get` worked because it was already in the table —
identical scratch shape, properly reserved.

The lesson: when adding a new intrinsic to the codegen,
the register-allocation pre-pass needs the matching slot
budget. The codegen comment at the existing entries
already calls this out:
> Mirror the bumps in `emit_vec_*` exactly — mismatched
> slots overlap with caller-save area and corrupt
> restored registers.

## generic type aliases — `type Pair<$T> = ($T, $T);`.

```zo
type Id<$T>   = $T;
type Pair<$T> = ($T, $T);
type Grid<$T> = []$T;

imu x:   Id<int>   = 42;
imu pos: Pair<int> = (3, 7);
imu row: Grid<int> = [10, 20, 30];
```

`execute_type_alias` never set up `self.type_params` and
walked the body with an ad-hoc loop. `$T` resolved to no
known name, fell back to unit, and `Pair<int> = (3, 7)`
reported a spurious `Type mismatch`.

### approach.

- **executor** — `execute_type_alias` now parses
  `<$T, ...>` into `self.type_params` (saving/restoring
  the outer scope), then dispatches body resolution
  through `resolve_param_or_return_ty` so `$T`, tuples,
  arrays, and `Fn` types all share one path.
- **executor** — new `generic_aliases:
  HashMap<Symbol, (Vec<TyId>, TyId)>` field stores the
  alias's param vars + body. At each use site, the
  let-annotation arm in `begin_decl` builds a substitution
  from params to the trailing `<...>` args (or fresh vars
  if omitted) and runs `substitute_ty` on the body.
  Without per-use substitution the alias's shared
  inference vars would leak — a second `Pair<str>` after
  `Pair<int>` would unify `int` against `str`.
- **executor** — `resolve_array_type` and
  `resolve_tuple_type` accept `Dollar + Ident` so `[]$T`
  and `($T, $T)` resolve through the active type-param
  map.

## `when` ternary inside template `{…}` interpolation.

```zo
imu view ::= <>{when count == 1 ? "time" : "times"}</>;
#dom view;
```

The `When` introducer was closed only on `;`. Inside a
template `{…}` interp there is no `;`, so When stayed
open and adopted every following sibling (TemplateText,
TemplateFragmentEnd, even the `#dom` directive) as a
ternary child. That malformed parse tree propagated into
the fragment's `children_end`; the executor's
`skip_until = children_end` then skipped past `#dom view;`
entirely — no `Insn::Directive`, no window. Programs
using ternaries inside `{…}` exited cleanly with no
output.

### approach.

`zo-parser` — `handle_rbrace_closer` now closes a `When`
introducer on top of the stack before falling through to
the regular `}` path, mirroring the existing `Hash`
close. Pairs with `Fn` (inline closures) and `Hash`
(`{#html}`) — same shape.

## reactive computed bindings for compound `{expr}`.

```zo
fun main() {
  mut count: int = 0;
  imu counter ::= <>
    <button @click={fn() => count += 1}>
      clicked {count}
      {when count == 1 ? "time" : "times"}
    </button>
  </>;
  #dom counter;
}
```

Initial render shows `clicked 0 times`. Click increments
→ `clicked 1 time`, `clicked 2 times`, …. Both `{count}`
and `{when …}` update in lock-step.

The previous synthetic-decl path for compound `{expr}`
emitted SIR but resolved to a runtime-only `Load`
(`Value::Runtime`), so `value_to_string` returned an
empty string and no `UiCommand::Text` was pushed. There
was also no mechanism to recompute the expression on
state change — only the simple-ident `{count}` fast
path had reactive tracking via `template_bindings.text`.

### approach.

- **`zo-sir`** — `TemplateBindings` gains
  `computed: Vec<(usize, ComputedBinding)>` and a new
  `ComputedBinding { closure_name, captures }` struct.
- **`zo-executor`** — the `Token::LBrace` arm of
  `execute_template_fragment` synthesizes a
  `__interp_<n>` closure (FunctionKind::Closure) for each
  compound `{expr}`. Runs `identify_captures` on the
  brace token range, mirrors `execute_closure`'s
  state-save / SIR-swap / FunDef-emit / body-execute /
  Return-emit pattern, drains the closure SIR into
  `deferred_closures`. The single-ident fast path
  (`{count}`) stays on `bindings.text` so existing
  reactivity is preserved.
- **`zo-runtime-render`** — `HandlerEvaluator::execute`
  now PC + label-index based with arms for
  `Insn::ConstString`, `Jump`, `BranchIfNot`, `Label`. A
  separate `locals` register file handles ternary sink
  stores (`store __branch_result_0__, %X` /
  `load Local(__branch_result_0__)`) which aren't state
  cells. Returns `Option<Val>` so computed bindings can
  surface the closure's return value. `Val` is now
  `pub` and has `display()` for symmetric rendering with
  `StateValue::display`.
- **`zo-driver`** — `run.rs` drains
  `bindings.computed`, registers state cells for every
  capture across all bindings, and re-runs each computed
  closure at every event via the new
  `apply_computed_bindings` helper. The same helper runs
  once before `runtime.run()` so the first frame shows
  the correct text instead of the empty placeholder the
  executor pushed.
- **`zo-dce`** — `collect_event_handler_syms` now also
  pins `bindings.computed[*].closure_name`; without
  this, DCE would drop every `__interp_N` closure as
  unreachable (it's referenced via the side-channel,
  not as a `UiCommand::Event` handler).
- **`zo-interner`** — new `Interner::snapshot() ->
  Vec<String>` so the evaluator can resolve
  `Insn::ConstString` from `Send` handler closures —
  `Interner` itself isn't `Clone` due to internal
  self-referential map keys.

## binding-aware text merge.

The template optimizer's `merge_adjacent_text` was
clobbering reactive prefixes. With
`Text("clicked ") + Text("0") + Text(" times")` adjacent,
the optimizer merged them to `Text("clicked 0 times")`.
The reactive `{count}` binding (cmd_idx=1, pre-merge)
became stale post-merge — the runtime's
`if let Some(UiCommand::Text(s)) = …` patch then
overwrote the entire merged string with just the count
value, dropping the `clicked ` prefix and `times` suffix
on every click.

### approach.

`zo-template-optimizer` — `optimize_with_indices` now
takes `bound_indices: &[usize]` and treats each as a
merge barrier: bound-target Texts stay isolated, runs
on either side merge separately. Returns an
`old_idx → new_idx` map; the executor remaps every
`bindings.{text,attrs,computed}` cmd_idx through it
after the merge runs. Stylesheet-prepend index shift
also covers `bindings.computed`.

## `=:>` — closure body that returns a template fragment.

```zo
imu wrap := fn(t: str) =:> <li>{t}</li>;
```

Step 1 of `PLAN_ZSX_TODOLIST.md`. The closure's body is a
template literal. Distinct from `::=` (assignment) — there
is no binding, the closure simply *returns* the fragment.
Sets up gap C of the plan: `arr.map(fn(t) =:> <li>{t}</li>)`
returning `[]</>` for reactive list rendering.

### shape.

`=:>` is a 3-byte token recognized in the tokenizer's `b'='`
arm. Lexing it switches the lexer into TEMPLATE mode the
same way `::=` does, so the bytes that follow tokenize as
template markup, not arithmetic.

### approach.

- **`zo-token`** — `Token::TemplateFatArrow` variant.
- **`zo-tokenizer`** — `b'='` arm peeks for `:` then `>`,
  emits the token with span 3, calls
  `state.set_mode(ModeState::TEMPLATE)`. Mirrors the
  `TemplateAssign` mode shift.
- **`zo-parser`** — `handle_template_fat_arrow` mirrors
  `handle_template_assign`: emit the marker, switch to
  `ParserState::TemplateMode`, auto-wrap a leading named
  tag (`<li>` etc.) in a synthetic `TemplateFragmentStart`
  introducer so the executor's
  `execute_template_fragment` sees the same shape
  regardless of whether the template came from a binding
  or a closure return.
- **`zo-executor`** — `execute_closure` recognizes
  `Token::TemplateFatArrow` as a third body opener
  alongside `=>` and `{`. Forces the closure's return type
  to `Ty::Template` so capture/store unifications agree
  with the fragment value the body produces. The
  return-type-annotation scan also breaks on
  `TemplateFatArrow` (was `LBrace | FatArrow`).

## `@input` / `@change` event payloads.

```zo
mut input_val: str = "";

imu view: </> ::= <>
  <input value={input_val}
         @input={fn(e) => input_val = e.value} />
  <p>{input_val}</p>
</>;
```

Step 2 of `PLAN_ZSX_TODOLIST.md`. Closures bound to
payload-bearing events receive a runtime-built `Event`
struct as their explicit user param. `e.value` reads the
input element's current text and feeds reactive bindings —
the `<p>{input_val}</p>` line above echoes typed input live.

### prelude.

```zo
pub struct Event {
  value: str,
}
```

Lives in `std/preload.zo` next to `Option`/`Result` so any
template-using pack picks it up without an explicit `load`.

### approach.

Six coordinated layers — verified by reading every
`zo-runtime-*` crate end-to-end before editing. None of them
existed before:

- **`zo-runtime-render::EventPayload`** — `{ value: String }`
  carries the runtime-built payload through the registry.
  `Display` impl renders `value` directly so debug prints
  work without `{:?}`.
- **`zo-runtime-render::EventHandler`** — signature changes
  from `Box<dyn Fn() + Send>` to
  `Box<dyn Fn(&EventPayload) + Send>`. `EventRegistry::dispatch`
  takes the payload by reference and threads it into every
  registered handler.
- **`zo-runtime-native`** — `pending_events` becomes
  `ThinVec<(WidgetId, EventKind, EventPayload)>`; the
  `<input>` arm captures `text.clone()` into the payload at
  `response.changed()`. `<button>` builds an empty default.
  Native dispatch loop forwards the payload.
- **`zo-runtime-web`** — new `parse_ipc_event` helper
  splits `assets/bridge.js`'s frames into
  `(widget_id, EventPayload)`. `click:`/`focus:`/`blur:`/
  `hover:` carry no payload; `input:`/`change:` peel
  `:value` after the id (the value field can contain `:`,
  the helper keeps trailing colons intact). Unknown
  prefixes return `None` so the runtime drops them silently
  rather than panicking.
- **`zo-runtime-render::HandlerEvaluator`** — `execute`
  gains `event: Option<&EventPayload>`. New `Val::Event`
  variant materializes when the closure body loads its
  event-payload param. `TupleIndex` projects a field —
  index 0 → `value`. Computed-binding closures and
  `@click` handlers pass `None`; field reads collapse to
  `Val::Unit` instead of panicking.
- **`zo-driver::cmd::run`** — registered handlers move
  from `Box::new(move || …)` to
  `Box::new(move |payload| …)`. The reactive variant
  passes `Some(payload)` to `eval.execute`; the static
  log-only variant ignores it but keeps the new
  signature.

### auto-typing the closure param.

`zo-executor`'s `Token::At` arm — for `@input`/`@change`
specifically — synthesizes a `Fn(Event) -> ?` annotation
through `pending_decl` *before* calling `execute_closure`.
Existing param-propagation (executor.rs:5897) then unifies
the user param `e` with `Ty::Struct(Event)` even when the
closure is written `fn(e) => ...` without an annotation.
Original `pending_decl` is restored after the closure
emits.
