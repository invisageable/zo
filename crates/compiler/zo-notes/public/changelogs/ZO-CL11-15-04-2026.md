# ZO-CL11 — 15-04-2026.

## bugfix — block arms in match consume outer while context.

`_ => {}` inside a `while` loop's `match` broke the loop:
the closing `}` of the block arm consumed the outer `while`'s
`BranchCtx`, emitting the back-edge jump inside the wildcard
arm and leaving no back-edge for the actual while body. The
loop executed exactly once.

### root cause.

`RBrace` handler checked `branch_stack.last()` unconditionally.
Inner blocks (match arm `{}`) at deeper scope depths found and
consumed branch contexts belonging to outer control flow.

### fix — scope depth guard on `BranchCtx` (zo-executor).

- add `scope_depth: usize` to `BranchCtx`, set at creation
- `RBrace` only closes a control flow block when
  `scope_stack.len() == ctx.scope_depth + 1` (same pattern
  already used for function body closing)
- inner blocks at deeper scope depths pass through harmlessly
- also added `apply_deferred_binop()` before match arm
  finalization — match arms use commas not semicolons, so
  deferred binops were never flushed for expressions like
  `tape[0] = tape[0] + 1`

### files changed.

| file | change |
|------|--------|
| `zo-executor/src/executor.rs` | `scope_depth` in `BranchCtx`, guard in RBrace, deferred binop flush in match arms |

### verified with.

- `brainfuck.zo` — full interpreter: `5, 6, 72, done`
- `match-array-test.zo` — isolated reproducer: `tape[0]`
  accumulates to `5` across 5 while iterations

## feature — `as` type cast.

`expr as Type` — type conversion between primitives. all GP
conversions (int/char/bytes/bool) and float<->int.

```zo
imu c: char = 72 as char;   -- 'H'
imu n: int = 'Z' as int;    -- 90
imu f: float = 42 as float; -- 42.0
imu i: int = 3.14 as int;   -- 3 (truncates)
```

### tokenizer (zo-token, zo-tokenizer).

- `Token::As` already existed in enum and keyword map (0x7361)
- no changes needed

### parser (zo-parser).

- handle `Token::As` explicitly: flush expression, emit As,
  consume and emit the adjacent type token directly so it
  appears right after As in the tree
- without this, the type token was buffered by `handle_type`
  and reordered away from As

### executor (zo-executor).

- `Token::As` handler: pop value from stacks, resolve target
  type via `resolve_type_token(idx + 1)`, emit `Insn::Cast`
- skip the type token (skip_until = idx + 2)

### SIR (zo-sir).

- new instruction: `Insn::Cast { dst, src, from_ty, to_ty }`
- added to `emit` (dst extraction), `visit_value_ids_mut`

### codegen (zo-codegen-arm).

- Cast dispatch on (from, to) type pairs:
  - float → int: `emit_fcvtzs(gp_dst, fp_src)`
  - int → float: `emit_scvtf(fp_dst, gp_src)`
  - GP → GP (int/char/bytes/bool): `emit_mov_reg` if different
    registers, no-op if same
- `value_types` map populated with `to_ty` for Cast

### register allocation (zo-register-allocation).

- `insn_is_fp`: Cast checks `to_ty` for float range (15-17)

### liveness (zo-liveness).

- `compute_value_ids`: Cast produces dst ValueId
- `insn_uses`: Cast uses src ValueId

### pretty printer (zo-pp).

- Cast renders as `%dst = cast %src (from_ty -> to_ty)`

## cleanup — remove redundant `ext` declarations in libm-all.zo.

`libm-all.zo` had 8 explicit `ext` declarations for math
functions (`pow`, `sin`, `cos`, `tan`, `log`, `log2`, `log10`,
`exp`). These are already declared as `pub ext` in
`std/math.zo` which is auto-preloaded by the compiler. The
redundant declarations shadowed the preloaded ones harmlessly
but were unnecessary noise.

| file | change |
|------|--------|
| `zo-tests/programming/libm-all.zo` | remove 8 redundant ext declarations |
