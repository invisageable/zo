# ZO-CL11 — 15-04-2026.

## bugfix — block arms in match consume outer while context.

`_ => {}` inside a `while` loop's `match` broke the loop:
the closing `}` of the block arm consumed the outer `while`'s
`BranchCtx`, emitting the back-edge jump inside the wildcard
arm and leaving no back-edge for the actual while body. The
loop executed exactly once.

### root cause.

`RBrace` handler checked `branch_stack.last()` unconditionally.
Inner blocks (match arm `{}`) at deeper scope depths found and
consumed branch contexts belonging to outer control flow.

### fix — scope depth guard on `BranchCtx` (zo-executor).

- add `scope_depth: usize` to `BranchCtx`, set at creation
- `RBrace` only closes a control flow block when
  `scope_stack.len() == ctx.scope_depth + 1` (same pattern
  already used for function body closing)
- inner blocks at deeper scope depths pass through harmlessly
- also added `apply_deferred_binop()` before match arm
  finalization — match arms use commas not semicolons, so
  deferred binops were never flushed for expressions like
  `tape[0] = tape[0] + 1`

### files changed.

| file | change |
|------|--------|
| `zo-executor/src/executor.rs` | `scope_depth` in `BranchCtx`, guard in RBrace, deferred binop flush in match arms |

### verified with.

- `brainfuck.zo` — full interpreter: `5, 6, 72, done`
- `match-array-test.zo` — isolated reproducer: `tape[0]`
  accumulates to `5` across 5 while iterations
- 247/247 zo_test pass, zero regressions

## feature — `as` type cast.

`expr as Type` — type conversion between primitives. all GP
conversions (int/char/bytes/bool) and float<->int.

```zo
imu c: char = 72 as char;   -- 'H'
imu n: int = 'Z' as int;    -- 90
imu f: float = 42 as float; -- 42.0
imu i: int = 3.14 as int;   -- 3 (truncates)
```

### tokenizer (zo-token, zo-tokenizer).

- `Token::As` already existed in enum and keyword map (0x7361)
- no changes needed

### parser (zo-parser).

- handle `Token::As` explicitly: flush expression, emit As,
  consume and emit the adjacent type token directly so it
  appears right after As in the tree
- without this, the type token was buffered by `handle_type`
  and reordered away from As

### executor (zo-executor).

- `Token::As` handler: pop value from stacks, resolve target
  type via `resolve_type_token(idx + 1)`, emit `Insn::Cast`
- skip the type token (skip_until = idx + 2)

### SIR (zo-sir).

- new instruction: `Insn::Cast { dst, src, from_ty, to_ty }`
- added to `emit` (dst extraction), `visit_value_ids_mut`

### codegen (zo-codegen-arm).

- Cast dispatch on (from, to) type pairs:
  - float → int: `emit_fcvtzs(gp_dst, fp_src)`
  - int → float: `emit_scvtf(fp_dst, gp_src)`
  - GP → GP (int/char/bytes/bool): `emit_mov_reg` if different
    registers, no-op if same
- `value_types` map populated with `to_ty` for Cast

### register allocation (zo-register-allocation).

- `insn_is_fp`: Cast checks `to_ty` for float range (15-17)

### liveness (zo-liveness).

- `compute_value_ids`: Cast produces dst ValueId
- `insn_uses`: Cast uses src ValueId

### pretty printer (zo-pp).

- Cast renders as `%dst = cast %src (from_ty -> to_ty)`

## feature — match on string literals.

`match s { "hello" => ... }` now compares by content, not
by pointer address. Uses length check + `_memcmp`.

```zo
imu s: str = "hello";
match s {
  "hello" => showln(1),
  "world" => showln(2),
  _ => showln(0),
}
-- outputs: 1
```

### executor (zo-executor).

- added `Token::String` to match primitive pattern list
- string patterns emit `ConstString` + `BinOp::Eq` with
  str type, same as other literal patterns

### codegen (zo-codegen-arm).

- `BinOp::Eq` / `BinOp::Neq` on `ty_id == STR_TYPE_ID`:
  1. load lengths from both string structs (offset 0)
  2. `CMP` lengths — if different, `B.NE` to "not equal"
  3. `ADD X0, l, #8` / `ADD X1, r, #8` — compute data
     pointers (inline layout: `[len:8][data][null]`)
  4. `MOV X2, len` — pass length to memcmp
  5. `_memcmp(ptr1, ptr2, len)` via extern call
  6. `CMP X0, XZR` + `CSEL` for boolean result
- avoids `_strcmp` because zo strings are not null-terminated
  (they carry explicit length)

## bugfix — realloc clobbers ArrayPush scratch values.

pushing > 1024 elements to a dynamic array corrupted the
newly pushed value. `arr[1024]` returned garbage after the
first realloc.

### root cause.

`ArrayPush` saved `new_cap` (X17) and `val_reg` to
`caller_save_base` before calling `_realloc`. But
`emit_extern_call` saves X9-X17 to the SAME
`caller_save_base` area, overwriting both values. After
the call, the restore loaded clobbered data.

### fix (zo-codegen-arm).

save ArrayPush scratch values PAST the caller-save area:
`extra_base = caller_save_base + CALLER_SAVE_COUNT * 8`.
This places them at slots that `emit_extern_call` never
touches.

### files changed.

| file | change |
|------|--------|
| `zo-codegen-arm/src/codegen.rs` | ArrayPush saves past caller-save area |
| `zo-tests/programming/array-realloc.zo` | push 2000 elements, read after realloc |

### verified with.

- `array-realloc.zo` — 2000, 0, 999, 1999
- 251/251 zo_test pass

## bugfix — recursion (`fact(5) = 120`).

recursive functions returned wrong values. three root causes
fixed in one pass.

### root cause 1 — stale condition leaks onto value stack.

`if x <= 1 { return 1; } return x * fact(x - 1);` — the
condition boolean from `x <= 1` stayed on the value stack
after BranchIfNot. The `*` operator consumed it as an operand
instead of `x`.

**fix (zo-executor):** pop condition from all three stacks
after emitting BranchIfNot in the LBrace handler.

### root cause 2 — stale return value leaks onto value stack.

`return 1;` inside the if body left the return constant on
the stack. After the if block, `x * fact(x-1)` used it as
an operand.

**fix (zo-executor):** pop return value from stacks after
emitting `Insn::Return` in `check_pending_return`.

### root cause 3 — register allocator reload mismatch.

after a `Call` instruction, live values are spilled (saved)
and reloaded. the reload allocated a FRESH register instead
of the ORIGINAL one. since the `assignments` HashMap stores
only one register per ValueId, the reload overwrote the
original assignment. the codegen then used the new register
for ALL instructions (including the initial Load), but the
spill saved the original register. result: Load emits to X9,
spill saves X10 (stale), Mul reads X10 = 0.

**fix (zo-register-allocation):** reload into the SAME
register the value occupied before the call. reserve the
original register from the free pool instead of allocating
a fresh one.

### programs promoted from wip.

- `fact.zo` — `f(5) == 120` via if/else recursion
- `factorial.zo` — ternary recursion (`when x <= 1 ? 1 : x * fact(x-1)`)
- `greatest-common-divisor.zo` — ternary recursion with modulo
- `fibonacci.zo` — double recursion (`fib(n-1) + fib(n-2)`)

### root cause 4 — ternary true arm value leaks.

the Colon handler emits Return for the true arm but doesn't
pop the value from stacks. it leaks into the false arm,
causing `*` to consume the stale true value instead of
deferring. same class of bug as root cause 2.

**fix (zo-executor):** pop value from stacks after emitting
the true arm Return in the Colon/Ternary handler.

### root cause 5 — ternary false arm Return never emitted.

ternary Colon sets `has_explicit_return = true` for the
true arm. the function body RBrace handler then skips the
implicit return because `has_explicit_return` is true. but
the false arm also needs a Return — it never gets one. no
epilogue, code falls through to next function.

**fix (zo-executor):** in the function body handler, detect
ternary on branch_stack with has_explicit_return. emit the
false arm Return and pop the ternary before the function
epilogue. also flush deferred binops before the function
body implicit return.

### root cause 6 — Call reload clobbers Call result.

V11 (first fib call result, in X0) is saved before the
second Call. the reload restores V11 into X0 (same register
fix), but X0 now holds fib(n-2). reload overwrites it:
`fib(n-1) + fib(n-1) = 2*fib(n-1)` instead of
`fib(n-1) + fib(n-2)`.

**fix (zo-register-allocation):** reload into the same
register UNLESS it's reg 0 (X0, the Call result register).
when orig_reg == 0, allocate a fresh register for the reload.

## feature — struct `::default()` auto-constructor.

when a struct has ALL fields with default values, the
compiler auto-generates `Type::default()`. no `apply`
block needed.

```zo
struct Rect {
  x: int = 10,
  y: int = 20,
  w: int = 100,
  h: int = 200,
}

fun main() {
  imu r: Rect = Rect::default();
  showln(r.x);  -- 10
}
```

### executor (zo-executor).

- struct definition handler now captures default value
  expressions (previously skipped at line 5451)
- after emitting StructDef, if all fields have defaults,
  emits a synthetic `FunDef "Type::default"` with
  ConstInt/ConstFloat/ConstBool/ConstString for each
  default, StructConstruct, and Return
- registers the synthetic function in `self.funs`

### bugfix — struct-returning call frame overflow.

caller's frame didn't reserve space for struct copies
from struct-returning function calls. the codegen copies
callee's struct into the caller's struct area, but the
register allocator only counted StructConstruct within
the function body — not from callees.

**root cause:** `Rect::default()` returns a 4-field struct.
main's frame was 80 bytes. struct copy writes to offset
72 (struct_base), but 4 × 8 = 32 bytes → 72 + 32 = 104
> 80. the struct overflowed into the saved FP/LR area,
corrupting the return address. segfault at exit.

**fix (zo-register-allocation):** for each Call to a
user-defined function, scan the SIR for the callee's
`FunDef ... StructConstruct ... Return` pattern. if found,
add the field count to `struct_slots` so the caller's
frame reserves space for the copy.

| file | change |
|------|--------|
| `zo-executor/src/executor.rs` | capture defaults, emit synthetic ::default() FunDef |
| `zo-register-allocation/src/allocator.rs` | struct-return Call frame slot reservation |
