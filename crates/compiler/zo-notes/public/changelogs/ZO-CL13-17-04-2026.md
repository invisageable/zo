# ZO-CL13 ‚Äî 17-04-2026.

two bugs from CL12 landed with silent failure modes ‚Äî
tests passed, compilation succeeded, binaries ran exit 0
‚Äî but user-facing behavior was broken. both rooted in
the same kind of oversight: a piece of SIR (or tree
span) was treated as safely disposable when it wasn't.
owning them here so the contract is explicit: "pre-
existing on main" doesn't excuse a broken path.

---

## bugfix ‚Äî DCE dropped `Insn::Template` / `Insn::StyleSheet`.

`zo run <file>` on 063/064/067 how-to programs printed
the compile banner and exited cleanly ‚Äî no UI runtime,
no window. the compiler produced a valid binary; the
driver looked for a template to match the `#dom`
directive, found nothing, and fell through to the
programming-path (compile-to-binary-then-exit).

**root cause.** DCE's `is_impure` list (which instructions
can never be dropped) omitted `Insn::Template` and
`Insn::StyleSheet`. Template defines a `ValueId` consumed
by the enclosing `VarDef { init }` and the downstream
`Directive { value }`. once a separate DCE pass prunes
the VarDef (or its liveness chain breaks for any other
reason), the Template's id looks transitively dead and
DCE drops it. but Template IS the observable side effect ‚Äî
it carries the UI command stream the runtime renders.
StyleSheet is the same story for scoped CSS.

**fix.** `zo-dce/src/dce.rs` ‚Äî add `Insn::Template` and
`Insn::StyleSheet` to `is_impure`. neither instruction
can ever be safely eliminated regardless of liveness.

### files changed.

| file | change |
|------|--------|
| `zo-dce/src/dce.rs` | `is_impure` includes Template and StyleSheet |

### verified with.

- `zo run 063-zsx-styling.zo` launches runtime with 4 UI
  commands. ‚úì
- `zo run 064-zsx-html-directive.zo` launches with 6. ‚úì
- `zo run 067-zsx-events.zo` launches with 9. ‚úì

## bugfix ‚Äî generic-outer-pass skip swallowed file siblings.

`std/math.zo` declares `pub fun abs<$T>` as the first
function, followed by 13 `pub ext` intrinsics (`sqrt`,
`floor`, `ceil`, `trunc`, `round`, `pow`, `sin`, `cos`,
`tan`, `log`, `log2`, `log10`, `exp`, `is_nan`,
`is_finite`). after preload, user code like
`showln(sqrt(9.0));` reported `Undefined variable: sqrt` ‚Äî
every math intrinsic was missing from `imported_funs`.

**root cause.** CL12's phase 2.b (skip initial generic
body execution) used the caller's `children_end` to set
`skip_until` after processing a generic function:

```rust
// BAD: _end_idx is the caller's children_end ‚Äî can reach
// to the end of file for top-level Fun nodes whose child
// span includes everything after them.
self.skip_until = _end_idx;
```

for the top-level `abs<$T>` node, `children_end` turned
out to be 343 ‚Äî the last tree index of the whole file.
setting `skip_until = 343` told the outer loop to skip
everything AFTER abs, so the 13 subsequent ext
declarations and the rest of the generics never ran
through `execute_ext` / `execute_fun`. `extract_exports`
then saw 0 public FunDefs in math's SIR.

worth pausing on: `children_end` as computed by the
parser is not always the function's own span. for top-
level siblings it can cover the enclosing scope (or the
file). phase 2.b's assumption was wrong.

**fix.** `zo-executor/src/executor.rs` ‚Äî compute
`end_of_block` explicitly by walking from the body's
`LBrace` with a depth counter to the matching `RBrace`,
and use that position for both:
- `skip_until = end_of_block` (so only this function's
  body is skipped, not its siblings).
- `generic_tree_ranges.insert(name, (start_idx,
  end_of_block))` (so the re-execution pass replays the
  correct span ‚Äî the previous broken range also meant
  re-execution for any generic in a multi-item file
  would replay trailing siblings).

factored the scan out so both generic and non-generic
functions get the correct tree range.

### files changed.

| file | change |
|------|--------|
| `zo-executor/src/executor.rs` | `end_of_block` scan, generic-skip uses matching RBrace, fixed `generic_tree_ranges` insert |

### verified with.

- `math-intrinsics.zo`: sqrt(9.0)=3.0, floor(3.7)=3.0,
  is_nan(1.0)=false, is_finite(1.0)=true ‚Äî all correct. ‚úì
- `libm-all.zo`: pow/sin/cos/tan/log/log2/log10/exp all
  resolve and run. ‚úì
- every generics test still passes (tree range stored is
  now narrower, not wider, so re-execution still sees
  the same function span ‚Äî invariant preserved). ‚úì

## lessons baked in.

1. **`children_end` is a hint, not a promise.** the
   parser's child span reflects the tree's flat layout,
   not necessarily the node's lexical extent. anywhere
   we want "skip past THIS function's body" we need an
   explicit depth-matched scan to the closing `}`.
2. **DCE's `is_impure` is load-bearing.** it's the only
   thing standing between the optimizer and observable
   side effects. any instruction whose presence the
   runtime depends on ‚Äî Template, StyleSheet, Directive,
   Call, Store, Return ‚Äî belongs there.
3. **"pre-existing on main" is not an exit condition.**
   if a test is red on main, it stays red until someone
   decides to fix it OR marks it explicitly as known-
   broken. silently accepting a green column that isn't
   actually green is how real regressions hide.
