# ZO-CL13 — 17-04-2026.

two bugs from CL12 landed with silent failure modes —
tests passed, compilation succeeded, binaries ran exit 0
— but user-facing behavior was broken. both rooted in
the same kind of oversight: a piece of SIR (or tree
span) was treated as safely disposable when it wasn't.
owning them here so the contract is explicit: "pre-
existing on main" doesn't excuse a broken path.

---

## bugfix — DCE dropped `Insn::Template` / `Insn::StyleSheet`.

`zo run <file>` on 063/064/067 how-to programs printed
the compile banner and exited cleanly — no UI runtime,
no window. the compiler produced a valid binary; the
driver looked for a template to match the `#dom`
directive, found nothing, and fell through to the
programming-path (compile-to-binary-then-exit).

**root cause.** DCE's `is_impure` list (which instructions
can never be dropped) omitted `Insn::Template` and
`Insn::StyleSheet`. Template defines a `ValueId` consumed
by the enclosing `VarDef { init }` and the downstream
`Directive { value }`. once a separate DCE pass prunes
the VarDef (or its liveness chain breaks for any other
reason), the Template's id looks transitively dead and
DCE drops it. but Template IS the observable side effect —
it carries the UI command stream the runtime renders.
StyleSheet is the same story for scoped CSS.

**fix.** `zo-dce/src/dce.rs` — add `Insn::Template` and
`Insn::StyleSheet` to `is_impure`. neither instruction
can ever be safely eliminated regardless of liveness.

### files changed.

| file | change |
|------|--------|
| `zo-dce/src/dce.rs` | `is_impure` includes Template and StyleSheet |

### verified with.

- `zo run 063-zsx-styling.zo` launches runtime with 4 UI
  commands. ✓
- `zo run 064-zsx-html-directive.zo` launches with 6. ✓
- `zo run 067-zsx-events.zo` launches with 9. ✓

## bugfix — generic-outer-pass skip swallowed file siblings.

`std/math.zo` declares `pub fun abs<$T>` as the first
function, followed by 13 `pub ext` intrinsics (`sqrt`,
`floor`, `ceil`, `trunc`, `round`, `pow`, `sin`, `cos`,
`tan`, `log`, `log2`, `log10`, `exp`, `is_nan`,
`is_finite`). after preload, user code like
`showln(sqrt(9.0));` reported `Undefined variable: sqrt` —
every math intrinsic was missing from `imported_funs`.

**root cause.** CL12's phase 2.b (skip initial generic
body execution) used the caller's `children_end` to set
`skip_until` after processing a generic function:

```rust
// BAD: _end_idx is the caller's children_end — can reach
// to the end of file for top-level Fun nodes whose child
// span includes everything after them.
self.skip_until = _end_idx;
```

for the top-level `abs<$T>` node, `children_end` turned
out to be 343 — the last tree index of the whole file.
setting `skip_until = 343` told the outer loop to skip
everything AFTER abs, so the 13 subsequent ext
declarations and the rest of the generics never ran
through `execute_ext` / `execute_fun`. `extract_exports`
then saw 0 public FunDefs in math's SIR.

worth pausing on: `children_end` as computed by the
parser is not always the function's own span. for top-
level siblings it can cover the enclosing scope (or the
file). phase 2.b's assumption was wrong.

**fix.** `zo-executor/src/executor.rs` — compute
`end_of_block` explicitly by walking from the body's
`LBrace` with a depth counter to the matching `RBrace`,
and use that position for both:
- `skip_until = end_of_block` (so only this function's
  body is skipped, not its siblings).
- `generic_tree_ranges.insert(name, (start_idx,
  end_of_block))` (so the re-execution pass replays the
  correct span — the previous broken range also meant
  re-execution for any generic in a multi-item file
  would replay trailing siblings).

factored the scan out so both generic and non-generic
functions get the correct tree range.

### files changed.

| file | change |
|------|--------|
| `zo-executor/src/executor.rs` | `end_of_block` scan, generic-skip uses matching RBrace, fixed `generic_tree_ranges` insert |

### verified with.

- `math-intrinsics.zo`: sqrt(9.0)=3.0, floor(3.7)=3.0,
  is_nan(1.0)=false, is_finite(1.0)=true — all correct. ✓
- `libm-all.zo`: pow/sin/cos/tan/log/log2/log10/exp all
  resolve and run. ✓
- every generics test still passes (tree range stored is
  now narrower, not wider, so re-execution still sees
  the same function span — invariant preserved). ✓

## bugfix — deferred binops survived across Semicolon.

`check((1) == 1); check(0xf0 | 0xf == 0xff);` — the second
check reported "incompatible types" at the `|`. investigation
traced it to deferred_binops leaking across statement
boundaries.

**root cause.** when an infix operator node is processed
before one of its operands has been pushed (e.g. the parser
emits `Eq` between its two operands), `execute_binop` defers
the op: pops the lhs, stores `(op, lhs, ...)` in
`deferred_binops`, returns. normally `apply_deferred_binop`
runs after the rhs is pushed and completes the op. but if
the outer expression (e.g. `check(...)`'s Call) consumes the
stack to zero BEFORE the defer resolves, `apply_deferred_binop`
runs with an empty stack and the while-loop exits immediately,
leaving the deferred entry in place. at Semicolon, the entry
is still there. when the next statement pushes its first
value, `apply_deferred_binop` (triggered by the main loop's
per-node hook) consumes it as the stale op's rhs — the next
statement's first operand is silently eaten.

**fix.** `zo-executor/src/executor.rs` — at Semicolon, after
calling `apply_deferred_binop`, `deferred_binops.clear()`.
statement boundaries are hard barriers for the defer queue;
any entry left over can't legitimately belong to the next
statement.

### files changed.

| file | change |
|------|--------|
| `zo-executor/src/executor.rs` | `deferred_binops.clear()` at Semicolon after flush |

### verified with.

- 4 commented-out checks in `binop-bitwise.zo` now
  compile and pass.
- `binop-bitwise.zo` promoted from `wip/wip-pass/`.
- 268/268 zo_test, 255/255 executor unit tests.

## bugfix — unary operator after bitwise operator.

`-4 & -1` was miscompiling to a wrong value. the parser's
`is_unary_context` list omitted the bitwise operator tokens,
so after `&`/`|`/`^`/`<<`/`>>` the next `-` was parsed as
*binary* minus rather than the unary negation that the
grammar requires. the resulting tree had `-4 - 1` (or similar
nonsense) folded in place of `-4 & -1`, producing `-5`.

**fix.** `zo-parser/src/parser.rs` — add `Token::Amp`,
`Pipe`, `Caret`, `LShift`, `RShift` to the list of tokens
after which a following `Minus` / `Star` / `Amp` should be
treated as unary.

## bugfix — signed bitwise const-fold stripped sign extension.

`showln(-4 | 0)` printed `4294967292` (the unsigned u32
interpretation of `-4`) instead of `-4`. same issue across
`BitAnd`, `BitOr`, `BitXor`, `Shl` and (separately) `Shr`.

**root cause.** `mask_to_width` ANDs the result with
`(1 << bits) - 1`. for a signed 32-bit `-4`
(`0xFFFFFFFFFFFFFFFC`), masking produces `0x00000000FFFFFFFC`
— the same low 32 bits but zero-extended in the high bits.
the codegen's `itoa` path reads the full 64-bit register via
`cmp x0, #0 / b.ge` — a 64-bit signed comparison — to decide
whether to print a leading `-`. a zero-extended value reads
as positive regardless of whether the low 32 bits encoded a
negative number in the declared type.

**fix.** `zo-constant-folding/src/constfolding.rs` — new
`mask_and_signext(value, signed, width)` helper that masks
then, if the type is signed and the sign bit of the masked
value is set, ORs in the high-bits-all-one mask so the full
64-bit word encodes the same signed number. applied to
`BitAnd`, `BitOr`, `BitXor`, `Shl`. for signed `Shr`, the
masking step is skipped entirely — `(lhs as i64) >> rhs`
already sign-extends, and the result can't exceed the
declared width's range. test
`s16_arithmetic_shr` updated to match the new semantics.

### files changed.

| file | change |
|------|--------|
| `zo-parser/src/parser.rs` | unary-context list includes `&`, `\|`, `^`, `<<`, `>>` |
| `zo-constant-folding/src/constfolding.rs` | `mask_and_signext`, used by signed bitwise fold paths |
| `zo-constant-folding/src/tests/width.rs` | `s16_arithmetic_shr` expects full-width sign-extended result |

### verified with.

- `showln(-4 & -1)`, `-4 | 0`, `-4 ^ 0`, `-4 << 1`,
  `-16 >> 2` all print as signed. ✓
- `check(-16 >> 2 == -4)` added to `binop-bitwise.zo`
  and passes. ✓
- 268/268 zo_test, 115/115 const-folding, zero clippy.

## lessons baked in.

1. **`children_end` is a hint, not a promise.** the
   parser's child span reflects the tree's flat layout,
   not necessarily the node's lexical extent. anywhere
   we want "skip past THIS function's body" we need an
   explicit depth-matched scan to the closing `}`.
2. **DCE's `is_impure` is load-bearing.** it's the only
   thing standing between the optimizer and observable
   side effects. any instruction whose presence the
   runtime depends on — Template, StyleSheet, Directive,
   Call, Store, Return — belongs there.
3. **"pre-existing on main" is not an exit condition.**
   if a test is red on main, it stays red until someone
   decides to fix it OR marks it explicitly as known-
   broken. silently accepting a green column that isn't
   actually green is how real regressions hide.
4. **statement boundaries are hard barriers.** any
   transient executor state (deferred binops, tuple_ctx,
   branch_stack…) that legitimately spans only ONE
   expression must be cleared — or asserted empty — at
   Semicolon. carrying state across statements is a silent
   data corruption channel.
5. **`is_unary_context` is a whitelist, not a blacklist.**
   every token that can precede a unary operator needs to
   be explicitly listed. forgetting one (e.g. `Amp`) means
   `a & -b` compiles but parses as nonsense. the test
   matrix should cover every infix-operator × unary-operator
   product.
6. **mask-to-width erases the sign bit.** for signed types,
   after any op that might mask (`BitAnd`/`BitOr`/`BitXor`/
   `Shl`), re-extend the sign bit to the full storage
   width — otherwise runtime paths that read the full
   register misinterpret the sign.

## bugfix — `match` on literal/expression scrutinee segfaulted.

```zo
match "zo" { "ivs" => ..., "zo" => ... }  // SIGSEGV
imu s: str = "zo"; match s { "ivs" => ..., "zo" => ... } // OK
```

matching a literal or expression (anything without a
source `Ident`) crashed at runtime with SIGSEGV 139.

**root cause.** `execute_match` locates the scrutinee's
source symbol by scanning the scrutinee range for a
`Token::Ident`. when the scrutinee is a literal (`"zo"`)
or an expression (`"z" ++ "o"`, `f()`), no Ident exists
and `scrutinee_sym` is `None`. the per-arm reload path
then skipped emitting a `Load` and returned a fresh,
never-emitted `ValueId` as the scrutinee register for
that arm. the register allocator gave it a register that
nothing ever populated; the ARM str-compare path read it
as a pointer and dereferenced garbage → segfault.

parameters-without-a-Store already had a fallback (emit
a synthetic `Store { name: __match_scrut__ }`) but it
only fired when an `Ident` existed — just one the
executor hadn't stored yet.

**fix.** `zo-executor/src/executor.rs` — extend the
synthetic-store fallback to the no-Ident case. any
scrutinee that leaves a SIR value on the stack gets
persisted under `__match_scrut__` before arm walking, so
per-arm `Load { src: Local(__match_scrut__) }` has a real
value to pull. covers `match "literal"`, `match a ++ b`,
`match f()`, etc.

### files changed.

| file | change |
|------|--------|
| `zo-executor/src/executor.rs` | synthetic-store fallback in `execute_match` extended to literal/expression scrutinees |

### verified with.

- `match "zo" { "zo" => check(true) };` ✓
- `match "z" ++ "o" { ... };` ✓ (the original wip program)
- `branch-match-lit-str.zo` promoted from
  `wip/wip-pass/`.
- 269/269 zo_test.

### lesson baked in.

7. **every scrutinee must have a load source.** per-arm
   reload in `match` is only meaningful if the load
   target exists. a `None` scrutinee_sym means no Ident
   was found — that's not a "skip the load" signal, it's
   a "synthesize the local first" signal. same reasoning
   as "statement boundaries are hard barriers": state
   that downstream code relies on must be materialized
   before downstream runs, not inferred from absence.

## feature — φ-like merge for branching expressions.

`when cond ? a : b` in expression position returned
garbage when the condition was true. same bug in `if
cond { a } else { b }`. zo's SIR emits arm bodies
linearly, and the consumer at `;` or the function
epilogue picked up whichever arm pushed last in
emission order (the false arm). when the true arm
executed at runtime, the false arm's register was
never written — the consumer read whatever the
register allocator left there.

**fix.** emulate an SSA φ via a synthetic stack slot.
each branch-expr allocates `__branch_result_N__` at
entry. each arm `Store`s into it at its exit (before
the merge label). after the merge label, a single
`Load` pulls the result and pushes it onto the
stacks. zo's SIR is not SSA-form, but `Store` +
`Load` through a stack slot is semantically
equivalent: whichever predecessor ran wrote the slot;
the merge's `Load` reads whatever that predecessor
wrote. no new `Insn` kind, no SSA rewrite.

**scope.** `when ? :` (this fix). `if { } else { }`
and `match` in expression position use the same
mechanism — the scaffolding (`BranchCtx.value_sink`,
`mint_branch_sink_if_expr_position`,
`emit_branch_sink_store`, `emit_branch_sink_load`) is
shared, and extending to `if`/`match` is follow-up
without further structural changes.

### files changed.

| file | change |
|------|--------|
| `zo-executor/src/executor.rs` | `value_sink` / `value_sink_ty` on `BranchCtx`, `branch_result_counter`, 3 helper methods, Colon emits Store, Semicolon emits Store+Label+Load, function-body RBrace closes sink-ternaries |

### verified with.

- `when true ? 100 : 200` prints `100` (was `1`). ✓
- `if true { 100 } else { 200 }` prints `100` (was `1`). ✓
- `when x > 3 ? x * 2 : x + 1` with `x = 5` prints
  `10` (was `5`). ✓
- `fun fact(x) { when x <= 1 ? 1 : x * fact(x - 1) }`
  still correct — tail-ternary in return position
  closes at the function's `}` and the merge Load
  flows into the implicit return. ✓
- `branch-ternary-arith.zo` promoted from wip.
- 270/270 zo_test pass (up from 268).

### plan.

`PLAN_BRANCH_EXPR_PHI.md` documents the seven-phase
design: detection, BranchCtx extension, Store at arm
exits, Load at merge, return/break handling,
register-allocator verification, and cleanup.
phase 1, 2, 3 (ternary), 4, 5, 6, 7 (ternary
only) shipped here. if-expr and match-expr reuse the
same mechanism — follow-up.

## feature — two-pass function signature pre-scan.

```zo
fun even(n: int) -> bool {
  if n == 0 { return true; } else { return odd(n - 1); }
}
fun odd(n: int) -> bool {
  if n == 0 { return false; } else { return even(n - 1); }
}
```

mutual recursion didn't compile — `odd(...)` inside
`even`'s body hit `UndefinedVariable`, because the
executor processes Tree nodes in lexical order and
`odd` wasn't yet in `self.funs` when `even`'s body
was executed. same failure for any forward call
(calling a function defined lexically below the
caller) between non-generic functions.

**root cause.** the main execution loop is strictly
single-pass: parse Token::Fun → set pending_function
→ LBrace pushes FunDef → body executes. callers
resolved against `self.funs` that only ever contained
functions whose headers had been visited. no forward
view.

**fix.** `zo-executor/src/executor.rs` — add a
signature-only pre-scan that runs before the main
loop. `prescan_fun_signatures` walks the Tree at
brace-depth 0, calls `execute_fun` with a new
`prescan_only` flag set, and the early-return inside
`execute_fun` pushes the `FunDef` (signature fields
only, `body_start: 0`) via a new `push_or_replace_fun`
helper. no scope is pushed, no params become locals,
no pending_function is set. the main pass then runs
normally and the LBrace handler upserts each function
(real `body_start` from the body emission) via the
same helper, so the pre-scanned stub is replaced in
place without duplication.

**scope.** top-level free functions. methods inside
`apply Type { ... }` and closures are not pre-scanned
— mutual recursion across an apply block or between
closures is a follow-up that requires honoring
`apply_context` during the pre-scan walk.
generic-outer-pass `self.funs.push` was also switched
to `push_or_replace_fun` so re-execution's
monomorphization pass continues to upsert correctly.

### files changed.

| file | change |
|------|--------|
| `zo-executor/src/executor.rs` | `prescan_only: bool` field, `push_or_replace_fun` helper, `prescan_fun_signatures` walker, `fun_body_end` helper, prescan early-return in `execute_fun`, LBrace and generic-outer-pass use upsert |

### verified with.

- `fun-mutual-recursion.zo` — `check(even(42))`,
  `check(odd(45))` both pass. promoted from
  `wip/wip-pass/`.
- 270/270 zo_test (no regression; same number as pre-
  fix because the new file replaces its wip slot).
- 1040/1040 Rust unit tests.

### lesson baked in.

8. **single-pass execution ≠ single lookup pass.**
   declarations and their uses can appear in any
   order in source; the data they produce
   (signatures, type decls) must be available to the
   whole pass, not just "whatever came before this
   node". pre-scan the lightweight piece (sig) so the
   heavyweight pass (body) can resolve forward.

## feature — `showln(array)` element-wise formatting.

```zo
imu xs: []int = [1, 2, 3];
showln(xs);                -- was: 6088255304  (raw ptr)
                           -- now: [1, 2, 3]
```

`showln(array)` leaked the array's heap pointer to
`itoa` and printed the numeric pointer value. Every
existing test worked around this with
`showln(arr.len)` or `showln(arr[i])`; no test ever
printed a whole array. Function-returned arrays like
`shuffle`'s result showed the same symptom.

**root cause.** `emit_typed_write` in the ARM64 codegen
dispatches on the argument's type: float →
`emit_ftoa`, bool → `emit_bool`, char → `emit_char`,
str → direct write, enum → `emit_enum_write`; **every
other type falls through to `emit_itoa`**. Array
pointers fell there. No code path formatted them
structurally, because:
  1. the codegen holds only a `TyId` per `ValueId` and
     no access to the `ty_checker`'s tables, so
     resolving `Ty::Array(aid) → ArrayTy { elem_ty }`
     was impossible.
  2. no SIR instruction surfaced the element type to
     codegen — unlike `Insn::EnumDef` which already
     does this for enum pretty-printing.

**fix.** Mirror the enum-pretty-printer pipeline:

1. **New SIR instruction**
   `Insn::ArrayTyDef { array_ty, elem_ty }` — emitted
   once per unique array type by the executor.
2. **Executor** adds `emit_array_ty_defs` that runs at
   the tail of `execute()`: walks every `ty_id` in the
   emitted SIR, calls `ty_checker.resolve_id` to follow
   the inference chain, and if the canonical type is
   `Ty::Array(aid)`, appends one `ArrayTyDef` per
   unique array `TyId`. Dedup via `HashSet<u32>` on
   `ty_id.0`.
3. **DCE** preserves `ArrayTyDef` as impure AND exempts
   it from the "dead zone" walker — otherwise any
   `ArrayTyDef` sitting past the last `Return` gets
   wiped by `eliminate_unreachable_after_return`.
4. **Codegen** pre-scans for `ArrayTyDef` into
   `array_metas: HashMap<u32, TyId>` (array TyId →
   elem TyId) before the main translation loop so
   `is_array_value(vid)` can look up elem_ty for any
   array value regardless of SIR order.
5. **`emit_array_write`** emits an inline ARM64 loop:
   ```
   MOV  X19, arr_ptr      ; base
   LDR  X20, [X19]        ; len
   MOV  X21, #0           ; i
   write "["
   loop:
     CMP  X21, X20
     B.GE end
     CBZ  X21, skip       ; no separator before first
     write ", "
   skip:
     LSL  X22, X21, #3    ; i*8
     ADD  X22, X22, #16   ; + header
     ADD  X22, X19, X22
     LDR  X0, [X22]       ; element
     <emit_field_write elem_ty>
     ADD  X21, X21, #1
     B    loop
   end:
     write "]"
   ```
   Uses callee-saved X19/X20/X21/X22 so write syscalls
   can clobber caller-saves freely. Element formatting
   reuses the existing `emit_field_write` dispatch
   (same switch the enum payload printer uses), so
   `[]int`, `[]float`, `[]bool`, `[]char`, `[]str` all
   print correctly without per-type copy-paste.
6. **Dispatch branch** in `emit_typed_write` between
   enum and the int fallback: if the value's type is a
   registered array, call `emit_array_write` instead of
   `emit_itoa`.

**scope.**
- in: `[]int`, `[]float`, `[]bool`, `[]char`, `[]str`.
- out (follow-up): nested arrays (`[][]int`), arrays of
  struct/tuple, arrays of enum — these fall through to
  `emit_field_write`'s int path and still print
  pointers for the inner elements. Adding them needs
  recursive dispatch (for nested arrays) or repeating
  the enum pretty-printer's payload walk (for structs/
  enums inside arrays). Deferred, same deferral style
  as "enum field payload printing" in CL12.

| file | change |
|------|--------|
| `zo-sir/src/sir.rs` | new `Insn::ArrayTyDef { array_ty, elem_ty }`, `visit_ty_ids_mut` + `visit_value_ids_mut` arms |

### lesson baked in.

9. **codegen shouldn't need the ty_table to format a
   value.** Every structural type that needs
   pretty-printing (enum, array, tuple, struct) should
   surface its shape through a dedicated SIR
   instruction (`EnumDef`, `ArrayTyDef`, etc.), so
   codegen can drive a uniform `value_types + meta`
   lookup. Threading the ty_checker into codegen would
   have worked once; the side-table pattern scales.
10. **DCE's "dead zone after Return" is a whitelist.**
    Every new "metadata" instruction that lives at the
    top/tail of SIR needs an entry — otherwise a run
    that happens to produce it after a function's last
    `Return` will silently drop it, and the failure
    surfaces only at runtime as mis-formatted output
    (or worse, a segfault if the metadata is load-
    bearing).

## bugfix — UTF-8 char literals rejected as "unterminated".

```zo
imu yen: char = '¥';   -- error: unterminated character literal
imu pi: char = 'Π';    -- error: unterminated character literal
```

Any non-ASCII codepoint inside `'...'` produced
`UnterminatedChar`. On top of that, the error reporter
itself panicked while rendering the error because its
`span_to_range` assumed byte-offsets always landed on
UTF-8 char boundaries — so the actual error was hidden
behind a thread panic.

**root cause — tokenizer.** `scan_char` advances by
ONE byte between the opening and closing quote, which
is fine for ASCII but leaves the cursor inside a
multi-byte codepoint (`¥` = `0xC2 0xA5`, `Π` = `0xCE
0xA0`). The next `current() == b'\''` check sees a
continuation byte instead of the closing quote →
"unterminated". Advance logic was per-byte where it
should have been per-codepoint.

**root cause — reporter.** `span_to_range` called
`source[..byte_pos].chars().count()`. When an upstream
span lands mid-codepoint (same class of bug, or
recovery state after one), `source[..byte_pos]` panics
with "byte index N is not a char boundary". The
reporter should never crash — it's the last line of
defense.

**fix.**
- `zo-tokenizer/src/tokenizer.rs` — new `utf8_cp_len`
  helper that returns 1/2/3/4 based on the lead byte.
  `scan_char` advances by `utf8_cp_len(current())`
  after the opening quote instead of a fixed 1. The
  existing content-decoding path already handled
  multi-byte (`std::str::from_utf8(content)`), so
  downstream codegen/runtime sees a correct codepoint
  once the tokenizer finds the closing quote.
- `zo-reporter/src/render.rs` — `span_to_range`

## bugfix cluster — tuple/struct interaction.

```zo
struct Point { x: int, y: int }
apply Point { fun new(x: int, y: int) -> Self { Self { x, y } } }
type Rect = (Point, Point);

fun f(rect: Rect, x: int, y: int, w: int, h: int) { ... }
fun main() {
  imu r1: Rect = (Point::new(10, 20), Point::new(11, 22));
  f(r1, 10, 20, 11, 22);
}
```

`lit-tuple-rect.zo` was red in five different ways at
once. Five separate bugs — three in the executor
front-end, two in the ARM64 codegen — all surfaced
here because the test happens to compose tuple types,
tuple parameters, call-nested tuple literals,
struct-returning factories, and back-to-back struct
returns in a single program.

Owning them as a cluster because each is narrow but
their composition is what made the test impossible to
pass before.

### 1. `Ident` elements silently dropped in tuple type.

`resolve_tuple_type` gated element-collection on
`tok.is_ty()`. `is_ty()` is deliberately keyword-only
(`int`, `bool`, `str`, …) and excludes `Token::Ident`
— so `(Point, Point)` produced an empty-elem tuple.
Indexing into it then returned `unit` and downstream
code emitted `TypeMismatch` everywhere. Added an
`|| tok == Token::Ident` arm so struct/enum/alias
names land in `resolve_type_token`'s `Ident` branch.

### 2. Tuple parameter types resolved to `unit`.

`execute_fun`'s parameter-type parser had dedicated
arms for `[]T` and `Fn(…) -> T` but not `(T, U)`.
`resolve_type_token` saw `(` as non-type and returned
`unit`, so `fun f(t: (int, int))` typed `t` as unit
and every `.N` access mismatched. Added a
`Token::LParen` branch that calls `resolve_tuple_type`
— the same helper begin_decl already uses for tuple
annotations on locals.

### 3. Call `)` inside tuple literal popped the outer `tuple_ctx`.

`(f(x), g(y))` dispatched the FIRST `)` (which closes
the call `f(x)`) as a tuple-close because
`Token::RParen`'s handler checked `tuple_ctx.pop()`
before testing whether the paren pair was a call. The
pop drained the outer tuple's depth, the call emission
was skipped, the second call ran against an empty
`tuple_ctx` and the outer `)` fell through to
`execute_potential_call` — emitting ONLY the second
call and dropping the tuple literal entirely. Added
`rparen_closes_call(rparen_idx)` — walks back to the
matching `(`, reuses `resolve_call_target` to decide.
Call path now runs first, tuple-close is the
fallback.

### 4. Codegen silently emitted no `LDR` for immutable param reads.

The executor sometimes lowers an immutable parameter
read as `Insn::Load { src: LoadSource::Local(sym) }`
rather than `LoadSource::Param(idx)` (e.g. inside
`Self { x, y }` in a factory method). The codegen's
`LoadSource::Local` branch only recognized `sym`s
that lived in `mutable_slots`. Non-mutable param syms
hit the `else if` chain with no match — **no
instruction emitted at all**. The destination
register inherited whatever the caller left behind,
so `Self { x, y }` was effectively reading stale
caller state. Added `param_sym_slots: HashMap<u32,
(u32, bool)>` populated in the function prologue
alongside the index-keyed `param_slots`; the
`LoadSource::Local` branch now falls back to it and
emits a proper `LDR` / `LDR Dt` from the param spill
slot.

### 5. Struct-return call spilled the CALLEE's frame pointer.

The register allocator's spill-around-next-call logic
captures the call result's home register (`X0`, the
ABI return) at allocation time. Scalar calls leave
the return value in X0 — spilling from X0 is correct.
Struct-return calls use X0 as a scratch source for
the field-copy loop that materializes the caller's
own copy, then write the caller's pointer to
`result_reg` via `add result_reg, SP, #dst_base`. X0
was NEVER updated. So the allocator's pre-next-call
spill captured a pointer into the previous call's
stale callee frame — and when the tuple literal
later read the spilled pointer, it dereferenced into
the second call's struct area, aliasing `r.0` onto
`r.1`. Fix: after emitting the result_reg update,
also emit `add x0, SP, #dst_base` so the
spill-from-X0 invariant stays valid.

### lessons baked in.

13. **`is_ty()` is a whitelist, not a type classifier.**
    Wherever the grammar allows "a type expression", the
    code must accept both keyword-types and user idents
    — the latter resolve via `resolve_type_token`'s
    `Ident` branch. Every site that filters tokens with
    `is_ty()` alone has the same class of bug.
14. **Param-parse arms must cover every shape the type
    grammar produces.** `execute_fun`'s parameter loop
    had explicit arms for `[]T` and `Fn(...) -> R` but
    not `(T, U)` — the exact same omission as
    `resolve_tuple_type` in point 13, at a different
    site. Any grammar-production that has its own
    sub-parser needs an arm at every parameter-type
    entry point. An audit-level fix would be a single
    `resolve_param_type` helper, deferred.
15. **`tuple_ctx` and the call path must not both fight
    for `RParen`.** Two orthogonal consumers of the
    same syntactic boundary is a precedence bug waiting
    to happen. The fix is always "identify the consumer
    by looking at what opened the paren pair", not
    "pop one stack and hope".
16. **Codegen must have a path for every SIR
    producer-consumer shape.** The executor lowered a
    param read as `LoadSource::Local(sym)` and the
    codegen had no matching consumer — so the branch
    silently emitted nothing. Any `Insn::Foo` field
    that has more than one variant (`LoadSource`,
    `BinOp`, `CallTarget`, …) needs a codegen arm
    for every variant OR an `unreachable!()` that
    would panic in tests. Silent "do nothing"
    fallbacks are where heisenbugs live.
17. **Calling-convention invariants span both sides
    of a Call.** Regalloc's spill logic assumed X0 is
    the call result regardless of return kind, and
    that's true for scalar calls. Struct returns
    broke that invariant locally without restoring
    it. Every non-scalar return ABI has to either
    honor the scalar register-holds-result contract
    or the register allocator needs a different
    spill strategy for non-scalar call results.

## feature — short-circuit `&&` / `||`.

```zo
fun incr(mut x: int) -> bool { check(false); false }
fun main() {
  imu x := 1 == 2 || 3 == 3;  -- x = true
  imu y: int = 10;
  showln(x || incr(y));       -- must NOT call incr
  check@eq(y, 10);            -- y untouched
}
```

`&&` / `||` eagerly evaluated both operands and codegen'd
to bitwise `AND` / `ORR` — no branch, no short-circuit.
Any RHS with side effects ran unconditionally. Worse, the
ABI spill for live-across-call bool LHS was unreliable:
`false || g()` (both sides returning false) came back as
`true` because the LHS register was clobbered when `g`
returned — the register allocator hadn't seen it as
live-across-call for the bool binop path.

Short-circuit lowering fixes both at once: when the LHS
is enough to decide the result, the RHS call never runs,
and the liveness problem becomes moot.

**fix.** `zo-executor/src/executor.rs` — mirror the
branch-expr φ-sink pattern (see
`PLAN_BRANCH_EXPR_PHI.md`) for logical binops:

```
-- lhs && rhs
[emit lhs -> %l]
allocate sink S, emit Store S, %l
BranchIfNot %l, done_label
[emit rhs -> %r]
emit Store S, %r
Label done_label
Load %result <- S  -- push onto stacks

-- lhs || rhs
same, but synth UnOp::Not on %l and BranchIfNot on that.
```

`&&`/`||` tokens now dispatch to
`execute_logical_binop(op, idx)` instead of the generic
`execute_binop`. When both operands are already on the
stacks (pure truth tables — const LHS + const RHS), the
handler falls back to `execute_binop` so constant folding
still collapses them. Otherwise it pops the LHS, mints a
fresh `__branch_result_N__` sink (counter shared with the
ternary sink), emits the Store + BranchIfNot + (synth Not
for `||`) scaffolding, and queues a `DeferredShortCircuit`
with three depth marks (`pre_rhs_depth`,
`pre_tuple_ctx_len`, `pre_direct_call_depth`).

The finalize step emits the RHS Store + merge Label +
Load, and runs only when all three depths reach the right
"end of RHS" condition. This is necessary because the RHS
can itself be a call (`a || f(y)` — finalize AT the call-
closing `)`), can be a call ARG of an outer call
(`outer(a || f(y))` — finalize at the inner `)`, not the
outer's arg push), can be nested in a tuple
(`(a, b || g())` — finalize at the inner `)`, not the
tuple's `)`), or a compound with a short-circuit inside
another (`a || (b && c)` — the inner `&&` must finalize
before the outer `||`).

The finalize hook fires at RParen (post-call-close), at
tuple-close RParen, and has a safety-net at Semicolon
that emits any dangling labels for short-circuits whose
RHS never produced a stack value (e.g. the enclosing
expression trapped into an error path).

### lessons baked in.

18. **eager logical operators lie about side effects AND
    about liveness.** Short-circuit isn't just a
    semantic nicety — it's the difference between a live-
    across-call bool surviving or being clobbered. Any
    binop whose operand can have side effects (or whose
    evaluation straddles a call) needs control flow, not
    a bitwise opcode.
19. **deferred-op finalization needs depth marks, not
    "first new value".** Tuple contexts, nested calls,
    and call args all push intermediate values onto the
    SIR stacks. A deferred finalizer that fires on "one
    new value landed" gets confused by them. The right
    guard is: finalize when stack depths equal the pre-
    RHS marks + exactly 1 new value AND the nesting
    state (tuple_ctx, direct_call_depth) has unwound
    back to the marks. Three orthogonal depth counters
    wide enough to distinguish every nesting shape.
