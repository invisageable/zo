# ZO-CL08 — 12-04-2026.

## feature — unified style layer across native and web.

Introduce a shared style layer so a zo program renders consistently
on both the native (egui) and web (wry) targets. The layer sits
below PLAN_STYLER (user-authored `$: {}` rules) and above the
renderers themselves. See `PLAN_STYLE_UNIFICATION.md` for the full
architectural narrative.

### cascade model.

Three layers, evaluated in CSS order, with a per-target reality:

```
1. UA sheet       — default look of unstyled HTML
2. Author sheet   — user's `$: {}` rules (PLAN_STYLER, upcoming)
3. Inline style   — `style="..."` on elements (upcoming)
```

- **Native** resolves all three layers — egui ships zero HTML
  defaults, so without the UA layer `<h1>` and `<p>` would look
  identical.
- **Web** skips the UA layer — the browser already applies its
  own UA stylesheet, and duplicating our table as a `<style>` tag
  would just drift from real browser defaults.

### ui protocol — new `zo-ui-protocol::style` module.

Shared, target-agnostic style data. No egui dep, no wasm dep,
pure data + pure functions. Module laid out with Rust 2018 style
(no `mod.rs`):

```
crates/compiler/zo-ui-protocol/src/
  style.rs              — module entry + re-exports
  style/
    computed.rs         — ComputedStyle, StylePatch, enums
    ua.rs               — UA_SHEET + lookup
    cascade.rs          — resolve(tag, author, inline)
```

- `ComputedStyle` — fully resolved style: box model (display,
  margin, padding, width, height, min_width, min_height),
  typography (font_family, font_size, font_weight, font_style,
  text_align, text_decoration, line_height), and color
  (foreground, background). `ComputedStyle::ROOT` = browser-like
  defaults (white bg, black text, 16px sans, 400).
- `StylePatch` — sparse override with `Option<T>` per field.
  `StylePatch::apply(&mut base)` folds set fields in, leaves
  unset fields alone.
- `UA_SHEET` — user-agent stylesheet as a `&[(&str, StylePatch)]`
  table keyed by lowercase tag name. Covers body, h1–h6, p, a,
  code, pre, ul, ol, blockquote, strong, b. Values transcribed
  from the WHATWG rendering spec and WebKit's `html.css`.
  `lookup(tag)` returns the patch or `None`.
- `cascade::resolve(tag, author, inline) -> ComputedStyle` —
  starts from `ComputedStyle::ROOT`, folds UA → author → inline
  patches in order, returns the result.
- `Rgba` — premultiplied 8-bit RGBA with `BLACK`, `WHITE`,
  `LINK_BLUE`, `TRANSPARENT` constants and `rgb(r, g, b)` ctor.
- `Edges` — four-sided length in CSS order with `ZERO`, `all(v)`,
  and `v(top, bottom)` helpers.

### ui protocol — `ElementTag` extended to h4/h5/h6.

`<h4>`, `<h5>`, `<h6>` join `<h1>/<h2>/<h3>` in the enum.
Renumbered `as_u32` / `from_u32` so all six headings stay
contiguous (13–18) and `Custom(_)` moves to 24. `is_text_tag`
now matches the full `h1..h6` range.

### native runtime — new `theme` module.

Extracted every egui-specific styling knob out of `runtime.rs`
into `crates/compiler/zo-runtime-native/src/theme.rs`. This is
the lighter precursor to PLAN_STYLER's full `zo-styler` crate —
cascade machinery lives in `zo-ui-protocol::style`, egui chrome
lives here.

- `theme::style_default(ctx)` — installs the default zo theme on
  an egui context. Called once from the `eframe::CreationContext`
  closure. Sets:
  - Pure white surface (`panel_fill`, `window_fill`,
    `extreme_bg_color`) + black text override, built from
    `Visuals::light()` and then forced.
  - Chromium-like button chrome across all egui interaction
    states — `#efefef` resting, `#e5e5e5` hovered, `#dcdcdc`
    pressed, `#767676` border, `1.0` stroke width.
  - Browser-like `Spacing::button_padding = (6, 2)` so every
    button gets the same gutter regardless of glyph advance.
- `theme::body_frame() -> egui::Frame` — builds the root
  CentralPanel frame at each `App::ui` call. Reads the 8px body
  gutter from `UA_SHEET["body"]` so the table stays the single
  source of truth; no magic numbers in the renderer.

### native runtime — `App::ui` owns the root panel.

eframe 0.34 hands `App::ui` a raw `egui::Ui`, not a CentralPanel.
The previous code rendered directly into that `Ui`, which made
the viewport fall through to a black default. Now `App::ui`
wraps its content in an explicit `egui::CentralPanel::default()
.frame(theme::body_frame()).show_inside(ui, ...)` so the page
background is an owned, deterministic choice (white + body
gutter).

### native renderer — UA cascade drives text styling.

`render_styled_text_for_tag` now calls `cascade::resolve(tag, None,
None)` and applies the resulting `ComputedStyle` through a new
`apply_computed(rt, style)` helper instead of hardcoded H1/H2/H3
sizes. egui's `RichText` API is narrow, so the helper collapses:

- `font_size` → `RichText::size(s)`
- `font_weight >= 600` → `RichText::strong()`
- `font_style == Italic` → `RichText::italics()`
- `font_family == Mono` → `RichText::monospace()`
- `text_decoration == Underline` → `RichText::underline()`
- `color` → `RichText::color(rgba_to_color32)`

Block-flow vertical margin is honored via `ui.add_space(top)`
before and `ui.add_space(bottom)` after the label. Horizontal
margin and the rest of the box model are ignored at v1 — Taffy
integration in Phase 3 will handle those.

### native renderer — inline flow for block parents.

Pre-Taffy hack for CSS inline-formatting: a block container
whose direct children are all inline-ish (span, button, input,
textarea, img, or non-blank text) now lays out horizontally
instead of stacking vertically. Fixes the counter demo
`<><button>-</button>{count}<button>+</button></>` which was
rendering as three stacked rows.

- new `children_are_inline_flow(commands, start) -> bool` — walks
  direct children up to the matching `EndElement`, returns true
  iff every child is an inline-flow tag or non-blank `Text`.
- new `is_inline_flow_tag(tag) -> bool` — mirrors CSS `inline |
  inline-block` for the small subset zo currently models.
- `render_element`'s block-container branch picks
  `ui.horizontal(...)` when the scan returns true, else falls
  back to `ui.vertical(...)`.
- `render_with_ui` runs the same scan on the top-level command
  stream — fragment templates (`<></>`) emit their children as
  siblings at the root, so the per-element scan never fires for
  them. The top-level wrap is what makes the counter flow
  horizontally.

### web renderer — drop the CSS reset.

`crates/compiler/zo-runtime-web/assets/default.css` used to
ship a universal `* { margin: 0; padding: 0; box-sizing: border-box }`
reset plus `body { padding: 1rem }`. The reset killed the
browser's real UA stylesheet, and the `1rem` body padding (16pt)
was twice the native-side `UA_SHEET["body"]` gutter (8pt) — the
exact 2× drift a user spotted in side-by-side screenshots.

Fix: delete the reset and the explicit body padding. Let the
browser apply its own UA stylesheet (8px body margin, h* sizes,
p margins, etc.) so native and web both draw against the same
canonical defaults. `box-sizing: border-box` and the flex
container classes are kept — genuinely useful, no drift risk.

### tokenizer bugfix — `peek()` OOB read.

Long-standing fuzz panic on inputs like `"b:0"`:

```
range start index 4 out of range for slice of length 3
  at tokenizer.rs:1121 (0b/0o/0x number prefix branch)
```

Root cause was the branchless OOB guard in `Tokenizer::peek`:

```rust
fn peek(&self, offset: usize) -> u8 {
  let pos = self.cursor + offset;
  let in_bounds = (pos < self.source.len()) as usize;
  unsafe { *self.source.as_ptr().add(pos * in_bounds) }  // buggy
}
```

When `pos` was out of bounds, `in_bounds = 0`, and `pos * 0 = 0`
— so `peek` returned `source[0]` instead of a sentinel zero.
For `"b:0"`, `peek(1)` at the trailing `0` returned the leading
`'b'`, spoofing a binary-literal prefix. The tokenizer then
advanced the cursor past the end, and the slice
`&source[start + 2..cursor]` panicked.

Fixed by switching to the same `[ZERO_BYTE, source]` jump-table
trick `Tokenizer::current` already uses — reads from a static
sentinel byte when out of bounds, never from `source[0]`.

All `token_spans_within_bounds` and `tokenizer_never_panics`
proptests green post-fix. A tokenizer must never panic on any
`&str` input; proptests exist exactly to catch this class of
contract violation before users hit it.

## feature — tuple-variant enums + pretty-printing (PLAN_SUM_TYPES Phase 1, partial).

First real consumer of the `PLAN_SUM_TYPES` dependency chain.
Land a working `enum Loot { Gold(int), Potion(int), Nothing }`
declaration *and* make `show(Loot::Gold(50))` produce a
human-readable `Loot::Gold(...)` instead of leaking a raw stack
pointer. Unblocks PLAN_MATCH Phase 2 (which will reuse the same
cmp-chain-on-discriminant machinery for `match` arms) and
PLAN_IO Phase 5 (`Result<str, io::Error>` without seeing
`0x10a2f8680` on failure).

### root cause of the original bug.

`Insn::EnumConstruct` was already wired end-to-end in the
executor for tuple variants — the construction path allocates
`[tag, f0, f1, ...]` in the struct area and leaves a pointer
to the base in the destination register. The gap was in
`emit_typed_write`: it recognized `is_string_value`,
`is_float_value`, `is_bool_value`, and fell through to `itoa`
for everything else. Enum pointers hit the fallthrough, so
`show(Loot::Gold(50))` printed the pointer's numeric value
(`6094348664`). Unit variants accidentally worked because
`EnumConstruct` with empty fields emits `mov imm, discriminant`
— a plain integer that itoa prints correctly.

### native codegen (zo-codegen-arm).

- new `enum_metas: HashMap<u32, EnumMeta>` keyed by `TyId.0`,
  populated once per `Insn::EnumDef` via the new
  `register_enum_meta` helper. Each `VariantMeta` records the
  discriminant, field count, and a synthetic `display_sym`
  pointing at a pre-baked `"EnumName::VariantName"` zo-string
  in `string_data`.
- new `next_enum_sym` counter starting at `0xE000_0000` hands
  out synthetic symbol ids for the display strings. Mirrors
  the `emit_bool_and_write` pattern that already uses synthetic
  `Symbol(0xFFFD)` / `Symbol(0xFFFC)` for `"true"` / `"false"`.
  Zero changes to the interner — codegen owns its own symbol
  namespace far above the interner's dynamic range.
- new `ENUM_TUPLE_ELLIPSIS_SYM = Symbol(0xE000_FFFF)` — a
  shared synthetic symbol holding the `"(...)"` suffix
  appended to tuple-variant prints so users can tell
  `Loot::Gold(...)` apart from `Loot::Nothing` without
  inspecting the payload.
- new `is_enum_value(vid, all_insns) -> Option<TyId>` helper
  sitting next to `is_string_value` / `is_float_value` /
  `is_bool_value`. Returns the enum's `TyId` when the producing
  instruction is an `Insn::EnumConstruct`, else `None`.
- `emit_typed_write` now checks the enum case between the bool
  and the itoa branches. When hit, it dispatches to
  `emit_enum_write` instead of falling through.
- new `emit_enum_write(vid, ty_id, all_insns, fd)` — the heart
  of the change:
  - Loads the discriminant into `X17`. Unit variants store the
    integer directly in the source register, so we `mov X17,
    src`. Tuple variants store a pointer, so we `LDR X17, [src,
    #0]`. The shape is discriminated by peeking at the
    producing `EnumConstruct`'s `fields` length.
  - Emits a cmp-chain — one arm per variant — `cmp X17,
    #disc; b.ne past_this_variant; <write display string>; [if
    tuple: write "(...)"]; b done`.
  - Forward branches (`b.ne past`, `b done`) are patched after
    the body via two new emitter primitives, same pattern the
    libm / BL fixups already use.
- `X17` chosen deliberately — it's outside the register
  allocator's pool (allocator uses `X0-X7` + `X9-X15`). An
  earlier attempt with `X15` crashed the binary because
  clobbering a live allocated value silently corrupted the
  surrounding showln scope.

### emitter (zo-emitter-arm).

Two small additions, both mirrors of existing primitives:

- new `patch_bcond_at(pos, offset)` — byte-offset patcher for
  a previously emitted `B.cond`. Preserves the original
  condition code via a mask, only rewrites the `imm19` field.
  Mirrors `patch_cbz_at`.
- new `patch_b_at(pos, offset)` — byte-offset patcher for a
  previously emitted unconditional `B`. Mirrors `patch_bl` but
  for plain `B`.

Both are used by `emit_enum_write`'s cmp-chain to fix up
forward jumps once each arm body's length is known, and by the
shared `.done` label after all arms are emitted.

### known gap — enum values via local variables.

`imu x: Loot = Loot::Gold(50); showln(x);` still hits the
itoa fallthrough because `find_producing_insn` for the `Load`
returns the load instruction, not the original `EnumConstruct`.
`is_enum_value` intentionally rejects `Load` / `ArrayIndex`
sources: a loaded slot could hold either an integer (unit
variant discriminant) or a pointer (tuple variant struct),
and dispatching through the wrong shape segfaults. The
conservative v1 leaves stored enums on the itoa path.

The proper fix is **uniform pointer representation** — change
`Insn::EnumConstruct` for unit variants to allocate a single
`[tag]` slot on the stack and return a pointer too, so every
loaded enum value is statically known to be a pointer. That's
a real refactor of `emit_enum_construct` plus every enum load
site; it lives in its own follow-up.

## feature — uniform pointer representation for enum values (PLAN_SUM_TYPES Phase 1, finished).

Close out the Phase 1 gap the first section of this changelog
left open: enum values passed through a local variable
(`imu x: Loot = Loot::Gold(50); showln(x);`) still hit `itoa`
and printed a raw pointer, because `is_enum_value` conserva-
tively rejected `Insn::Load` sources to avoid segfaulting on
the shape-ambiguous register. This section makes every enum
value in a register a **pointer** — unit or tuple, construc-
tion or load — so `emit_enum_write` can always dereference
for the discriminant and `BinOp::Eq`/`Neq` can always compare
tags.

### what "uniform pointer representation" means.

Before: unit variants emitted `mov imm, discriminant` and
returned a small integer; tuple variants allocated `[tag, f0,
...]` on the stack and returned a pointer. Loads of an
enum-typed slot were shape-ambiguous — you couldn't tell at
codegen time whether the register held an integer or a
pointer, so `is_enum_value` had to reject loads and keep them
on the `itoa` path.

After: every enum variant allocates `[tag, f0, f1, ...]` on
the stack. Unit variants get a single `[tag]` slot; tuple
variants get `[tag]` + one slot per field. Every register
holding an enum value is a pointer to one of these blocks,
full stop. The shape ambiguity is gone.

Cost: unit variants now consume one 8-byte stack slot + one
store per instance, instead of living in a register. For
programs that use enums in hot loops, this matters; for the
current `zo-how-zo` / `zo-tests` / demos, it's noise next to
the `show` syscall cost. Revisit when a benchmark shows the
extra store is hot.

### codegen (zo-codegen-arm).

- `Insn::EnumConstruct` — unit and tuple branches unified. One
  path allocates `1 + fields.len()` stack slots at
  `struct_base + next_struct_slot`, stores the tag via
  `mov X16 + str`, stores fields (empty for unit variants),
  and returns `add_imm(dst, SP, base)`. ~25 lines of
  duplication collapsed to one linear flow.
- `is_enum_value` — extended with `Insn::Load { ty_id, .. }`
  and `Insn::ArrayIndex { ty_id, .. }` arms. Both check the
  codegen-owned `enum_metas` table as the "is this ty_id an
  enum?" oracle. Thanks to uniform representation, loaded
  enum values are statically known to be pointers, so the
  dispatch is now safe.
- `emit_enum_write` — the unit-vs-tuple branch on the
  discriminant load is gone. Every caller loads `LDR X17,
  [src, #0]`. Signature shrunk from four parameters to three
  (`all_insns` no longer needed — `find_producing_insn` is
  only called by `is_enum_value` at the dispatch site).
- **`BinOp::Eq` / `BinOp::Neq` on enum operands** — new
  enum-aware branch in the integer-binop lowering. When
  `ty_id.0` is in `enum_metas`, codegen loads both
  discriminants (`LDR X16, [l, 0]` + `LDR X17, [r, 0]`) and
  compares via `emit_cmp_csel`. Without this, `check@eq(c,
  Color::Red)` would compare raw pointers to distinct
  allocations and always return `false`, even when both
  operands hold the same variant. Other comparison ops (`<`,
  `>`, `<=`, `>=`) are undefined on enum types and fall
  through to the integer path as a documented no-op.

### register allocation (zo-register-allocation).

Root cause of the test-runner crash that surfaced during the
refactor: `allocator::assign_registers`'s `struct_size`
computation excluded unit enum variants via
`EnumConstruct { fields, .. } if !fields.is_empty()`. With
uniform representation, unit variants DO need stack slots,
and the old guard caused the prologue to reserve too little
frame — the unit-variant store walked past the end of the
reserved region into the caller's frame, producing a silent
corruption that surfaced as SIGBUS (exit 138) on function
return.

Fix: drop the `if !fields.is_empty()` guard so every
`EnumConstruct` contributes `1 + fields.len()` slots to
`struct_slots`, matching what the codegen now actually
allocates. One-line change with a long explanation comment
because the failure mode was non-obvious (the interactive
`zo run` path executed in-process and didn't trip the bug;
only the built binary via `zo-test-runner` caught it).

  to explicitly project the tag via `match`.

## feature — `match` expression, Slice 1 (PLAN_SUM_TYPES Phase 2, partial).

Land the first vertical slice of `match`: literal int patterns,
wildcard `_`, side-effecting arm bodies. Enough to dispatch
on integers and exercise the full pipeline (parser → executor →
SIR → codegen). Enum patterns and match-as-expression are
follow-up slices.

### parser (zo-parser).

- new `handle_match_keyword` — emits `Token::Match` as an
  introducer, pushes onto the introducer stack, parses the
  scrutinee as an expression, then lets `handle_lbrace_
  introducer` open the arm block. Mirrors the `if` / `while`
  / `when` shape exactly.
- `handle_rbrace_closer` — `Token::Match` added to the parent
  auto-close list so the match introducer closes when its arm
  block `}` is reached.
- No new token variants needed — `Token::Match`,
  `Token::FatArrow`, `Token::Comma` already exist. Patterns
  are flat children separated by `FatArrow` + `Comma` markers;
  the executor groups them at lowering time.

### executor (zo-executor).

- new `execute_match(start_idx, end_idx)` — batch-mode handler
  that takes ownership of the match's children via
  `skip_until = end_idx`. Processes the scrutinee by streaming
  its nodes through `execute_node`, then walks the LBrace
  children as flat `[pat, FatArrow, body..., Comma]*` groups.
- For each arm: emits a fresh `Insn::Load` of the scrutinee
  variable, a `Insn::ConstInt` for the pattern literal, a
  `Insn::BinOp::Eq` comparison, and `Insn::BranchIfNot` to
  the next arm's label. Wildcard arms skip the test. Arm
  bodies are executed by streaming their nodes through
  `execute_node` (same pattern `execute_closure` uses for its
  body range). Each arm's body is bracketed by
  `Insn::Jump { target: end_label }` + `Insn::Label { id:
  next_arm_label }`.
- **Multi-match liveness fix**: early version reused a single
  scrutinee `ValueId` across all arms, which broke the register

  `Insn::Load` per arm so each comparison gets its own
  `ValueId` with correct local liveness.

  correctly. Match arms compose from existing SIR primitives.

### executor (zo-executor).

  `Ident :: Ident` (unit variant) and
  `Ident :: Ident ( Ident )` (tuple variant with binding).

## feature — stdlib `Result<$T, $E>` and `Option<$T>` (PLAN_SUM_TYPES Phase 3 + 4).

Phase 3 (generic enums) was already working — zero changes
needed. Phase 4 lands the stdlib preload module and the
import pipeline to make `Result` and `Option` available in
every zo program without a `load` statement.

### preload module (crates/compiler-lib/std).

- new `preload.zo` — declares `pub enum Option<$T> { Some($T),
  None }` and `pub enum Result<$T, $E> { Ok($T), Err($E) }`.
- `lib.zo` — `pack preload;` added so the module resolver
  finds it.
- compiler prelude list — `"preload"` prepended to the
  `["io", "assert", "math"]` auto-import array.

### enum export pipeline (zo-module-resolver).

- `ExportedEnum` struct — carries `name: Symbol` + raw
  `variants: Vec<(Symbol, u32, Vec<TyId>)>`. Raw variant data
  instead of TyChecker-internal IDs so the importing executor
  can re-intern into its own TyChecker.
- `extract_exports` — new `Insn::EnumDef` arm extracts pub
  enums with `translate_symbol` + `translate_ty_id` for variant
  names and field types.
- `ModuleExports` — new `enums: Vec<ExportedEnum>` field.
- Module SIR filter — `Insn::EnumDef` is now filtered from the
  merged SIR alongside `Insn::PackDecl`. EnumDef's ty_ids
  reference the module's throwaway TyChecker; leaving them in
  the merged SIR causes ty_id collisions in the codegen's
  `enum_metas` HashMap.

### import pipeline (zo-analyzer + zo-executor).

- `ImportedSymbols` — new `enums` field.
- `Executor::with_imports` — stores imported enums as
  `pending_imported_enums` for **lazy interning**. Eagerly
  adding enum types shifts the TyId counter and pollutes the
  HM unification state, breaking ternary/if-else type checking.
- `execute_enum_access` — on first reference to an unknown enum
  name, checks `pending_imported_enums`, removes the entry,
  interns it in the executor's own TyChecker with fresh
  inference variables (inside a push/pop scope to isolate from
  function-body generalization), and pushes to `enum_defs`.
- Ident handler — `is_enum` check extended to also peek at
  `pending_imported_enums` so the "Undefined variable" error
  doesn't fire before the `::` triggers lazy import.

### codegen fix — `is_enum_value` Store-tracing.

The uniform pointer representation's `is_enum_value` for
`Load`/`ArrayIndex` caused a SIGSEGV when generic type-variable
TyIds (`$T` in `Result<$T, $E>`) numerically collided with enum
TyIds in `enum_metas`. Routing an integer through
`emit_enum_write` dereferences it as a pointer → crash at
address `0x63` (99, the payload value).

Fix: for `Load` sources, trace through the variable's `Store`
instruction to find the initializing `EnumConstruct` and use
ITS ty_id instead of the Load's potentially-generic ty_id.

### type annotation workaround.

Generic enum type annotations (`imu a: Option<int> = ...`) are
not yet parsed as monomorphized types — the parser captures
only the last type token (`int`), not the full `Option<int>`.
Workaround: `finalize_pending_decl` skips unification when the
init type is a known enum, using the inferred type as ground
truth. Proper generic type annotations are a follow-up.

## feature — file I/O syscall implementation (PLAN_IO, PLAN_SUM_TYPES Phase 5).

Land the actual ARM64 syscall-level implementation of
`read_file`, `write_file`, `append_file`, and `exists` so
zo programs can perform file I/O with `Result` error handling.

### codegen (zo-codegen-arm).

- `emit_io_exists` — `access(path, F_OK)` syscall, CSET CC
  for bool result.
- `emit_io_read_file` — `open → read → close` sequence,
  constructs `Result::Ok(str)` or `Result::Err(errno)` on
  stack. macOS ARM64 error convention: carry flag set + X0 =
  positive errno (B.CS branch).
- `emit_io_write_impl` — shared for `write_file`/`append_file`,
  `open(flags) → write → close → Result`. Write uses
  `O_WRONLY|O_CREAT|O_TRUNC` (0x601), append uses
  `O_WRONLY|O_CREAT|O_APPEND` (0x209).
- `emit_add_sp_offset`, `emit_str_sp`, `emit_ldr_sp` — helpers
  for SP-relative addressing when offsets exceed ARM64's 12-bit
  ADD immediate or 9-bit STR/LDR immediate limits. IO functions
  allocate ~4KB read buffers that push struct offsets past these
  limits.

### emitter (zo-emitter-arm).

- `COND_CS` constant (0x2) and `emit_bcs` method for branch
  on carry set (macOS syscall error detection).
- **Critical fix**: `emit_add_ext` / `emit_sub_ext` — extended
  register form ADD/SUB where register 31 = SP. The shifted
  register form (`emit_add`/`emit_sub`) treats register 31 as
  XZR (zero register), so `SUB SP, SP, X16` was encoding as
  `SUB XZR, XZR, X16` — a no-op. Every function whose frame
  exceeded 4095 bytes had a broken prologue (SP never adjusted).
  This was a pre-existing bug that also caused the
  `unary-negation` and `037-closures` test failures on main.

### register allocation.

- `zo-interner` threaded through `RegAlloc::allocate` so the
  allocator can identify IO intrinsics by name and reserve
  extra struct slots: `read_file` → 520 slots (4KB buffer),
  `write_file`/`append_file` → 5 slots.

### executor (zo-executor).

- **Match lazy import**: the match handler now triggers lazy
  import of pending imported enums (e.g. `Result`) when an
  enum variant pattern is the first reference. Previously,
  `Result::Ok(n)` in a match pattern silently skipped the arm
  if Result wasn't already imported via a prior construction.
- **EnumDef emission from lazy import**: lazy-imported enums
  now emit `Insn::EnumDef` to the SIR so the codegen
  registers `enum_metas` for match discriminant handling.
- **Fresh inference var for parameterized ext return types**:
  `execute_ext` now uses `Token::Lt`/`Token::Gt` (not
  `Token::LAngle`) to detect generic type args in return
  types. When type args are present, the return type is
  replaced with a fresh inference variable to prevent multiple
  ext functions (e.g. `Result<str,int>` vs `Result<int,int>`)
  from fighting over shared enum field inference variables.
- **Return type args propagation**: ext function `FunDef`s
  carry `return_type_args` (concrete TyIds like `[str, int]`).
  These flow through `SemanticResult.funs` → `extract_exports`
  → the main program's imported functions. The match handler
  uses them to type arm bindings correctly instead of the
  enum's unresolved generic field types.

### tutorial (zo-how-zo).

- new `048-io.zo` — exercises `write_file`, `exists`,
  `read_file`, `append_file` with `Result` matching. Err arms
  use `eshowln` for stderr. Expected output:
  ```
  8
  true
  hello zo
  6
  hello zo rocks
  ```

### register allocation — dynamic mutable slot sizing.

- `FunctionInfo::mutable_size` — computed from the actual
  number of unique `Store` targets per function, replacing
  the hardcoded `MUTABLE_VAR_RESERVE = 64` (8 slots). Functions
  with many locals no longer silently corrupt the stack frame.

### executor — keyword rejection in match bindings.

- Match arm pattern bindings now reject type keywords (`bytes`,
  `str`, `int`, etc.) with an `ExpectedIdentifier` error. The
  tokenizer produces `BytesType` / `StrType` instead of `Ident`
  for these words, making them unusable as variable references
  in the arm body. Previously, using a keyword as a binding
  silently produced a broken binary.

### executor — Err variant type arg offset.

- The `var_return_type_args` mapping now correctly offsets by
  variant position when assigning field types. For
  `Result<str, int>`: Ok field 0 → `rta[0]` (str), Err field 0
  → `rta[1]` (int). Previously both used `rta[0]`.

## feature — enum payload pretty-printing.

### codegen (zo-codegen-arm).
