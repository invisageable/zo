# ZO-CL07 — 11-04-2026.

## refactor — unified Element model for templates (R1–R4).

Land the HTML-parity `UiCommand::Element { tag, attrs, self_closing }` model
so every template attribute flows through a generic `Vec<Attr>` bag. Supersedes
the per-tag flat-field variants (`Text { style }`, `Button { id, content }`,
`TextInput { id, placeholder, value }`, `Image { id, src, width, height }`,
`BeginContainer { direction }`, `EndContainer`) that forced a four-crate
cascade every time an HTML attribute or tag was added. See
`PLAN_DYNAMIC_ATTRIBUTES.md` for the full architectural narrative.

### ui protocol (zo-ui-protocol).

- new `UiCommand` shape — `Element { tag, attrs, self_closing } | EndElement | Text(String) | Event { .. } | StyleSheet { .. }`
- new `ElementTag` enum — 22 HTML tags + `Custom(String)` escape hatch, with `as_str`, `is_self_closing_default`, `is_inline`, `is_text_tag`, `as_u32`, `from_u32` helpers
- `Attr::Dynamic { name, var: u32, initial: PropValue }` extended from the previous post-MVP stub to carry the reactive binding source and eager initial value
- `PropValue::parse(raw)` + `PropValue::to_display()` helpers power the shared eager-stringify path
- new `UiCommand::set_attr(name, value)` — generic attribute setter that finds-and-updates any `Prop` or `Dynamic` on an `Element` command, no-op on unknown attrs or non-element variants
- new `Ui::resolve_image_paths` walks `Element { tag: Img, attrs }` (previously `UiCommand::Image.src`)
- deleted: `UiCommand::{BeginContainer, EndContainer, Text { content, style }, Button, TextInput, Image}`, `TextStyle`, `ContainerDirection`

### sir (zo-sir).

- new `TemplateBindings { text: Vec<(usize, Symbol)>, attrs: Vec<(usize, Attr)> }` struct replaces the opaque `Vec<(usize, Symbol)>` in `Insn::Template.bindings`
- text bindings target `UiCommand::Text(_)` content patches; attribute bindings target a named attribute on a `UiCommand::Element` via `UiCommand::set_attr`

### executor (zo-executor).

- `handle_template` rewritten — one code path produces `UiCommand::Element` uniformly, `TagKind` / `classify_tag` / `resolve_direction` / `close_template_tag` sentinel hack all deleted
- `tag_to_element` replaces `classify_tag` — maps HTML tag names directly to `ElementTag` variants, unknown tags fall through to `ElementTag::Custom(name)`
- `close_template_tag` reduced to `commands.push(UiCommand::EndElement)` — no more `__text_{style}__` / `__button_{wid}__` sentinel id rewriting
- text interpolation (`TemplateText` + brace-expression) emits `UiCommand::Text(_)` directly
- shorthand attribute `<tag {ident} />` — sugar for `<tag ident={ident}>`, resolves the local and pushes `Attr::Prop` (imu source) or `Attr::Dynamic { name, var, initial }` (mut source)
- interpolated attribute strings `alt="a picture of '{name}'"` — walks the tokenizer's pre-parsed `InterpSegment` list, concatenates literal + variable segments into an eager `Attr::Prop`
- eager `attr={ident}` form fixed — the single-ident fast path resolves directly to the local's compile-time value (the previous path called `execute_node` which emitted a `Load` insn whose value_id had no compile-time value, silently producing empty strings)
- new helpers: `resolve_local_for_template`, `local_is_mut`, `make_attr_from_local`, `resolve_interp_string_attr`, `try_resolve_template_component`
- `pretty_print_commands` tracks an open-element stack so `EndElement` can recover the matching tag name — `check@eq(view, "<h1>hello</h1>")` assertions keep working
- `emit_opening_tag` records `Attr::Dynamic` entries into `template_bindings.attrs` with `cmd_idx = commands.len()` (the Element's position in the command stream)

### template optimizer (zo-template-optimizer).

- `classify_command` rewritten for Element model — dynamic iff an attr is `Attr::Dynamic`, else static; `flatten_containers` + `find_matching_end` deleted (direction-based, no longer applies)
- `merge_adjacent_text` simplified — single pending buffer for `Text(_)` PCDATA, no more style-discriminator branching

### driver (zo-driver).

- `ReactiveContext` split into `text_bindings: &[(usize, Symbol)]` + `attr_bindings: &[(usize, Attr)]` fields
- `build_reactive_handlers` dedupes state cells across both binding kinds (text and attrs can reference the same variable), registers closure handlers that patch text via direct `UiCommand::Text(_)` mutation and attributes via `UiCommand::set_attr`
- widget id attribute renamed from `id` to `data-id` in the element attrs vec — matches what `bridge.js` reads via `e.target.dataset.id`

### native renderer (zo-runtime-native).

- generic Element dispatcher `render_element(tag, attrs, self_closing)` handles `Img` via the `ImageLoader`, `Button` via `peek_text_children` + `egui::button`, `Input` / `Textarea` via `egui::TextEdit`, text tags (`H1`/`H2`/`H3`/`P`/`Span`) via `render_styled_text_for_tag`, inline via `ui.horizontal`, block via `ui.vertical`
- helpers `attr_str`, `attr_num`, `peek_text_children`, `skip_to_end_element`
- fixed widget id lookup to read `data-id` instead of the removed `id` attr — both + and - buttons in the reactive counter were dispatching to the same handler

### web renderer (zo-runtime-web).

- generic Element walker emits HTML with a uniform `data-zo-cmd="{idx}"` id on every element, enabling granular reactive DOM patches via `document.querySelector('[data-zo-cmd="N"]')`
- `emit_attr` stamps `Attr::Prop` / `Attr::Dynamic` / `Attr::Style` with correct HTML syntax, `Attr::Event` routes through the bridge JS
- `TextNode` wraps in an inline `<span data-zo-cmd="{idx}">` so reactive text updates can target bare PCDATA uniformly
- image src rewrite extracted to `rewrite_attr_value` + `is_absolute_fs_path` + `normalize_uri_path` helpers — Unix-rooted paths and Windows drive-letter paths both detected cross-platform and normalized to `zo://localhost/…`
- `bridge.js` click / input / focus handlers use `e.target.closest('button')` / `'input, textarea'` / `'[data-id]'` instead of `e.target` directly — clicks on child nodes of a button (e.g., the `<span>` wrapping its text) now route to the button correctly

### web runtime (zo-runtime-web).

- granular DOM patch loop diffs command pairs in parallel — `(Text, Text)` → `textContent`, `(Element, Element)` → per-attr `setAttribute` dispatched through the uniform `[data-zo-cmd]` selector
- `zo://` custom protocol server — `with_url("zo://localhost/")` + `with_custom_protocol("zo", serve_asset)` replace the previous `with_html` approach so image assets flow through a same-origin protocol handler; `uri_path_to_fs` strips leading `/` from Windows drive paths on decode
- `attr_display_value` + `mime_from_path` helpers

### codegen (zo-codegen-arm).

- `handle_template` stubbed for the unified model — emits header + type_code + null_data_pointer per command, no attribute payload yet. Interactive `zo run` path is unaffected (bypasses the dylib loader entirely). Full binary encoder/decoder for the Element model is a follow-up.

- `parse_command` stubbed to return `None` for every type code — the legacy per-variant decoder with its `ContainerData` / `TextData` / `ButtonData` / `TextInputData` / `ImageData` structs is gone, full Element decoder pending.

## template comments — context-sensitive syntax.

Fix the long-standing problem that `--` zo line comments collide with literal template text like `<>Welcome -- i'm here!</>`. Adopt Svelte's context-sensitive model where comment syntax follows the enclosing language.

### tokenizer (zo-tokenizer).

- `scan_template_token` now handles `<!--` at the start of a tag position: consumes forward through the closing `-->` and emits zero tokens. Comments are stripped at compile time, matching React/Svelte/Vue behavior. Unterminated comments consume to EOF without panicking. ~20-line addition to the `b'<'` match arm.
- `--` line comments and `-* ... *-` block comments already work inside tag attribute lists (between `<tag` and `>`) and inside `{expr}` interpolation blocks because both contexts fall through to `scan_code_token` via the default arm / brace-depth delegation. Verified by tests.

### context rules.

| Context | Syntax |
|---|---|
| Template text (between elements) | `<!-- -->` only — `--` is literal text |
| Inside tag attribute list `<tag ... >` | `--` line, `-* *-` block (zo-style) |
| Inside `{...}` expressions | `--` line, `-* *-` block (zo-style) |

### tests (zo-tokenizer).

- `html_comment_multiline_stripped` — multi-line comment consumes across newlines, resumes template text correctly

- `unterminated_html_comment_does_not_panic` — malformed input consumes to EOF gracefully
- `zo_line_comment_inside_tag_attribute_list` — `--` line comment between attrs
- `zo_block_comment_inside_tag_attribute_list` — `-* ... *-` block comment between attrs (multi-line)

## raw html directive — `{#html expr}`.

Introduce a compile-time `{#html expr}` directive that splices
a string's HTML content as real structural commands into the
enclosing template. Equivalent to Svelte's `{@html expr}`,
Vue's `v-html`, React's `dangerouslySetInnerHTML`. Designed
for rendering pre-rendered markdown blobs and other static
HTML content. See `PLAN_HTML_DIRECTIVE.md` for the full
architectural narrative.

### approach — reuse zo's existing template pipeline.

The cleanest implementation writes **zero new HTML parser
code**. The raw string is wrapped in a synthetic zo function
body (`fun __zo_html_inline__() { imu __v__: </> ::= <>STRING</>; }`),
run through the full `Tokenizer` + `Parser` + `Executor`
pipeline, and the resulting `Insn::Template.commands` are
extracted and spliced into the enclosing template's command
vector. A fresh `Executor::new` instance isolates the sub-parse
from the main compilation state.

This pattern — spin up a scratch compiler on a sub-program to
reuse the existing pipeline — could apply to future features
like compile-time macros or const-fn evaluation.

### executor (zo-executor).

- new module `zo-executor/src/html_inline.rs` (~130 LoC) with
  `parse_raw_html(input) -> Vec<UiCommand>`. Wraps input,
  runs the full pipeline, extracts commands from the first
  `Insn::Template` it finds. Empty input yields empty vec;
  malformed input yields whatever partial commands the
  sub-parse produced before bailing.
- new `try_handle_html_directive` method on `Executor` (~90
  LoC) in the `Token::LBrace` template interpolation handler.
  Detects the `Hash + Ident("html") + Ident(src)` shape,
  resolves the source to a compile-time string via the
  existing `value_to_string` helper, calls `parse_raw_html`,
  and splices the result into `commands`. Rejects `mut`
  sources with a diagnostic (dynamic `#html` is follow-up).
- `zo-executor/Cargo.toml` — promoted `zo-tokenizer` and
  `zo-parser` from `[dev-dependencies]` to `[dependencies]`
  so the sub-executor can run the full pipeline. No
  dependency cycles introduced.

### tokenizer / parser / sir / protocol / runtimes.

**Zero changes.** The feature is entirely contained in
`zo-executor`. The spliced commands are ordinary
`UiCommand::Element` / `Text` / `EndElement` values that
downstream crates (SIR, driver, renderers) already know how to
handle.

### supported subset (MVP).

- Open / close / self-closing tags
- Quoted and boolean attributes
- Nested elements
- Inline text between tags
- HTML comments `<!-- -->` (stripped by the tokenizer —
  automatic win from ZO-CL07's earlier work)
- Enumerated tags (`p`, `div`, `section`, `article`, `h1`,
  etc.) map to `ElementTag::*` variants
- Unknown tags fall through to `ElementTag::Custom(name)`

### deferred limitations.

- **HTML entities (`&amp;`, `&lt;`, `&gt;`, `&quot;`, `&apos;`,
  `&nbsp;`)** — zo's template-text scanner exits on `;`, so
  entities terminate the sub-parse prematurely. Users must
  pre-decode entities before passing to `#html` for MVP. Fix
  queued as a follow-up: mask `;` inside `&...;` patterns with
  a private-use codepoint, restore + decode after the sub-parse
  completes. ~50 LoC of string walking.
- **Numeric entities** (`&#65;`, `&#x41;`) — same root cause
  as named entities.
- **CDATA / DOCTYPE / processing instructions** — not in zo's
  template grammar, passed through as stray text which the
  sub-parser may reject.
- **Dynamic `mut` sources** — reactive HTML splicing needs a
  new binding kind that points at a range of spliced commands
  rather than a fixed cmd_idx, plus runtime re-parsing on each
  patch. Out of scope for MVP.
- **Error position translation** — parse errors in the HTML
  blob surface with offsets into the wrapped source, not the
  original. Subtract the wrap prefix length when emitting
  diagnostics in the follow-up.

## parser bugfix — `#html` directive leaks `Hash` introducer.

When the `{#html strong}` smoke test landed, the outer
`</p>` close tag was silently being consumed as code-mode
`Lt`/`Gt` operators instead of template-mode tag closers,
dropping the final `EndElement` from the command stream.

### root cause.

`handle_directive` pushes a `Hash` introducer onto the parser
stack. `Hash` is normally closed by `;`, but template
interpolations like `{#html strong}` have no `;` — the
directive spans only until the matching `}`. When the `}`
arrived, the `Hash` introducer was still on top, so
`handle_rbrace_closer` couldn't find its matching `LBrace`,
parser state stayed in `Expression` mode, and the subsequent
`</p>` tokens got reordered by the shunting-yard as if they
were comparison operators.

### fix (zo-parser).

`handle_rbrace_closer` now pops a pending `Hash` introducer
off the top of the stack before searching for the matching
`LBrace` — exactly the same pattern already used for the
inline-closure boundary (`@click={fn() => expr}`). 4-line
addition.

### tests.

- `test_html_directive_smoke` now passes end-to-end — 6
  commands emitted (`<p>` open + 4 spliced + `</p>` close)
- all 72 parser tests, 219 executor tests, 79 tokenizer
  tests green

## tokenizer bugfix — zo comments inside `$: { ... }` style blocks.

`063-zsx-styling.zo` tripped the tokenizer when `--` line
comments appeared inside the `$:` style block. Comment bodies
leaked as stray `Ident` tokens, corrupting delimiter balance
and surfacing as `UnmatchedClosingDelimiter` diagnostics.

### root cause.

`scan_style_token` dispatched `b'-'` into the CSS-ident range
because CSS allows hyphens in names (`font-size`,
`-webkit-foo`). zo's `--` line comment and `-!` doc comment
had no dedicated branch — they were tokenized letter-by-letter
as part of an unwanted identifier.

### fix (zo-tokenizer).

New `b'-'` branch in `scan_style_token` (~10 LoC). Peek the
next byte: `-` or `!` → `skip_line_comment()` (zo `-- ...` /
`-! ...`); anything else → fall through to `scan_style_ident`
so CSS identifiers starting with `-` still work. The CSS
ident range trimmed to `a-zA-Z_` accordingly.

### tests.

- new `zo_line_comment_inside_style_block` test in
  `zo-tokenizer/tests/snapshots.rs` — asserts that comment
  words (`custom`, `tailwind`, `width`, `height`) don't leak
  into `Ident`/`StyleValue` tokens and no delimiter errors
  are emitted
