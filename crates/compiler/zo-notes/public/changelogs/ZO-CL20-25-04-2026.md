# ZO-CL20 — 2026-04-25

## str stdlib + parser + preload fixes.

CL20 lands the first `str` stdlib pack (`split`,
`lines`, `trim`, `parse_int`) plus the runtime / SIR
plumbing that makes it work, and closes two
pre-existing parser / compiler bugs that surfaced
along the way.

---

## str stdlib (`std/str.zo`).

Four byte-oriented inherent methods on `str`:

- `split(self, delim: str) -> []str` — split on
  every exact occurrence of `delim`. Empty
  delimiter returns `[]`. Empty input returns
  `[""]`.
- `lines(self) -> []str` — split on `'\n'`,
  strip a trailing `'\r'` per line for CRLF
  inputs. A single trailing `'\n'` does NOT
  produce an empty final line (matches Python /
  Rust behaviour).
- `trim(self) -> str` — strip ASCII whitespace
  (`' '`, `'\t'`, `'\n'`, `'\r'`) from both ends.
  Internal whitespace preserved.
- `parse_int(self) -> Option<int>` — decimal int
  parse with optional leading `'-'`. Returns
  `Option::None` on empty input or any non-digit
  byte. Overflow wraps.

`pack str;` now ships in `std/lib.zo` and
preloads after `arr`, so user code reaches str
methods with no explicit `load`.

### runtime + SIR.

`str` is a length-prefixed pointer
(`[len: u64 LE][bytes][null]`). `split` /
`lines` / `trim` build their results via one new
runtime export + one new SIR insn:

- `_zo_str_slice(src, lo, hi)` — copy
  `src[lo..hi]` into a fresh allocation matching
  the `str` ABI. Returns the new pointer.
- `Insn::StrSlice { dst, src, lo, hi, ty_id }`
  — wired through liveness, DCE, register
  allocator, validator, pretty-printer.

The executor's `execute_str_slice_const`
previously hard-erred on non-const bounds. It now
falls back to `Insn::StrSlice` when bounds are
runtime values; the const-fold path stays for
literal-receiver + literal-bounds slices.

Codegen lands the AArch64 ABI: receiver / lo / hi
in `X0` / `X1` / `X2` for `StrSlice`. Result in
`X0`.

### `==` on `str` routes through `Eq`.

`apply Eq for str { fun eq(self, other: str)
-> bool { ... } }` lands in `std/str.zo` —
length check + byte-by-byte loop in pure zo.
The executor's existing abstract-Eq dispatch
(previously struct-only) now also covers
primitives via `primitive_ty_name_str`, so
`==` / `!=` on str operands lower to a regular
`Insn::Call` to `str::eq`.

No `Insn::StrEq` SIR opcode, no `_zo_str_eq`
runtime export, no codegen special case — same
trait path as user-defined `apply Eq for Foo`
impls. Matches Rust / Swift / Carbon's approach
where equality is a trait, not a primitive
operator with hard-coded codegen.

## Parser fix — operator placement around `[`.

`while lo < hi && a[lo] == 0 { ... }` failed to
type-check pre-CL20: the parser emitted

```
lo, 3, a, [, lo, ], <, 0, ==, &&
```

— `<` landed AFTER the subscript instead of
BEFORE the receiver, so `<` popped `a[lo]`/`3`
and `==` saw `bool` vs `int`.

`handle_lbracket` now distinguishes COMMITTED
operators in `expr_buffer` (last occurrence is
NOT in `operator_stack`) from PENDING ones. When
flushing buffered operands ahead of `[`,
committed operators emit inline; pending ones
stay buffered for `flush_expr` to place at the
end. Same treatment used by `handle_lparen` for
short-circuit / / `||` `&&`.

Reproduces in any subscript-on-RHS shape:
`<int_cmp> && a[i] == lit`,
`x + b[j] && y < z`, etc. All compile and
type-check.

## Preload symbol propagation.

Each preload pack used to be analysed in
isolation: its analyzer's executor saw none of
the funs / enums / abstract defs registered by
earlier packs. So `str.zo` referencing
`Option<int>` (defined in `preload.zo`) silently
produced broken SIR — the first user file to hit
it (`hello.zo`) would then E0572 / E0575 at
`}`.

Each preload analyzer is now seeded with the
imports accumulated so far via
`Analyzer::with_imports` — exactly the path the
user-file analyzer uses. `ExportedEnum`,
`AbstractDef` and `AbstractMethod` gained
`#[derive(Clone)]` for the per-iteration
forward.

This is one-time startup work; the cloned
collections are small (12 packs, tens of FunDef
/ ExportedEnum entries each), and the alternate
designs (shared session storage, single
mega-tree) require disproportionate refactoring
for no perceptible win.

## direct match on FFI struct return — fixed.

Pre-CL20:

```zo
match read_file(path) {
  Result::Ok(text) => showln(text),
  ...
}
```

printed a stack address instead of the file
contents. The bound form

```zo
imu r := read_file(path);
match r { Result::Ok(text) => showln(text), ... }
```

worked. User functions returning `Result<str,
int>` worked directly too. The bug was FFI-
specific.

### root cause.

`var_return_type_args` is the side map that
carries an FFI's concrete return-type args
(e.g. `Result<str, int>` → `[Ty::Str, Ty::Int]`)
into the match arm's payload-type lookup.
Insertion happened on two paths: under the
declared local's name when `pending_decl` was
set (bound case), and under the enum's type
name when `kind_of(func.return_ty) == Ty::Enum
(_)`. The enum-name path never fired for FFI
calls because parameterized FFIs intentionally
overwrite `return_ty` with a fresh inference
var (executor.rs:6909-6911) to avoid signature
clashes — `kind_of(fresh_var)` is `Ty::Infer`,
not `Ty::Enum`. With no entry under either key,
the arm's `text` binding fell back to the
enum's generic field var; codegen's
`emit_typed_write` saw a non-`STR_TYPE_ID`
and routed to the int decimal-print path. The
"stack address" output was the str pointer
formatted as `i64`.

### fix.

`execute_match` synthesises a `__match_scrut__`
local for non-bare-ident scrutinees. It now
inspects the producing SIR `Insn::Call` (when
the scrutinee resolves to one) and copies that
function's `return_type_args` into
`var_return_type_args` keyed by the synthetic
symbol. The arm's existing lookup at
`scrutinee_sym = __match_scrut__` then finds
the concrete payload types, identical to the
bound path. About fifteen lines in
`zo-executor/src/executor.rs`; no codegen
change.

### regression coverage.

- `programming/io-direct-match.zo` — `match
  read_file(...) { Result::Ok(text) => ... }`
  with str payload.
- `programming/io-str-lines-loop.zo` — the
  `io + str.lines() + for` example from the
  user request, now using the natural direct-
  match form (no intermediate `imu`).

## method dispatch on non-ident receivers — fixed.

`"a\nb\nc".lines()`, `s[0].is_whitespace()`,
`'7'.to_digit()` — every method call whose
receiver wasn't a bare ident — silently dropped
the call and yielded the receiver. Bound
receivers (`imu s := "..."; s.lines()`) worked.

### root cause.

`resolve_dot_call` derived the receiver type by
reading the receiver tree node's symbol and
looking it up in `self.locals`. Three preconds
have to hold for that path to work: the node
has a `NodeValue::Symbol`, the symbol matches
a registered local, and the local's `ty_id`
covers the actual runtime type. Literal nodes
yield their interned content (not a local
name), `Token::RBracket` / `Token::RParen`
yield no symbol at all. In every non-ident
case the function returned the unmangled
method name; `execute_call` then found no
`FunDef`, took the early-return branch, and
left the receiver on the stack as the
"result". Codegen consumed it as if the call
had returned it.

### fix.

`is_dot_method_call` already peeks
`ty_stack[len-2]` correctly while the member
ident is still on top of the stack — that's
the only point where the receiver type is
unambiguously available, because by the time
RParen fires, args have been pushed in between.
The Dot handler now stashes the receiver type
into `dot_method_recv_ty` keyed by the dot's
tree index when it classifies a method call.
`resolve_dot_call` and
`resolve_dot_call_with_receiver` now look up
that stash by dot index instead of walking
locals — the call-site signature gained a
`dot_idx: usize` parameter and dropped the
old tree-symbol lookup.

### regression coverage.

The four `str-*.zo` tests use the natural
literal-receiver form throughout (`"a,b,c"
.split(",")`, `"  hello  ".trim()`, etc.).
They exercised the bug pre-fix; they exercise
the fix now.

## known gaps surfaced (NOT fixed in CL20).

- `Option<int>` carrying a NEGATIVE payload
  loses the value when passed through a function
  parameter. Direct `match parse_int()` returns
  the right `-N`; `unwrap_or(parse_int(), 0)`
  returns garbage. parse_int's negative branch
  is therefore tested with a direct `match`,
  not through `unwrap_or`.

Both are pre-existing and warrant their own
followups.
