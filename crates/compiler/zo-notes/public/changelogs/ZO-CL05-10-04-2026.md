# ZO-CL05 — 10-04-2026.

## reactive state.

- implement reactive state system for interactive UI templates
- the counter program `@click={fn() => count += 1}` works end-to-end on both native (egui) and web (wry) runtimes

## parser (zo-parser).

- fix `Fn` node scoping inside template attributes — inline closures `fn() => expr` inside `@click={...}` left the Fn introducer open, consuming closing tag tokens
- close Fn introducer before LBrace in `handle_rbrace_closer` when top of introducer stack is Fn (inline closure boundary)

## executor (zo-executor).

- intern closure names as `"__closure_N"` strings instead of synthetic `Symbol(0x80000000 | N)` — makes closures accessible via `interner.get()`
- handle inline closure expressions in event attributes `@click={fn() => expr}` — detect `Token::Fn` after `{`, call `execute_closure`, extract generated handler name
- track `is_mutable` flag in `identify_captures` — captured `mut` variables retain `Mutability::Yes` inside the closure
- record dynamic template bindings — `{count}` where `count` is `mut` pushes to `self.template_bindings` with the command index
- finalize pending compound assignments at inline closure body boundary — call `finalize_pending_compound()`, `finalize_pending_assign()`, `apply_deferred_binop()` before implicit Return so `fn() => count += 1` emits `BinOp + Store`
- flush remaining deferred closures at end of `execute()` as safety net

## sir (zo-sir).

- add `bindings: Vec<(usize, Symbol)>` field to `Insn::Template` — maps command indices to reactive variable names

## value (zo-value).

- add `CaptureInfo` struct with `name`, `sir_value`, `is_mutable` fields
- change `ClosureValue.captures` from `Vec<(Symbol, ValueId)>` to `Vec<CaptureInfo>`

## dce (zo-dce).

- treat `FunctionKind::Closure` as reachable roots in dead function elimination — closures are called by the runtime via `EventRegistry`, not by static code

## runtime-render (zo-runtime-render).

- add `StateCell` — `Arc<Mutex<StateValue>>` with `get()`, `set()`, `mutate()` for shared reactive state
- add `StateValue` enum (Int, Float, Bool, Str) with `display()` and `Display` impl
- add `HandlerEvaluator` — minimal SIR stack machine that executes closure bodies at runtime, handles `ConstInt`, `ConstFloat`, `ConstBool`, `Load`, `BinOp` (17 operators), `UnOp`, `Store`, `Return`

## driver (zo-driver).

- detect reactive templates — `template_bindings` non-empty + closure handler names
- create `StateCell` for each bound `mut` variable, initialize from `VarDef` in SIR
- build handler closures that execute closure body via `HandlerEvaluator`, patch dynamic `Text` commands, push to shared command buffer
- add `ReactiveContext` struct for clean parameter grouping
- add `shared_commands()` getter on `Runtime` for handler closure wiring

## runtime-native (zo-runtime-native).

- use `Arc<Mutex<Vec<UiCommand>>>` shared command buffer — handler closures push updated commands, egui reads current commands each frame
- add `set_shared_commands()` method to accept driver's shared buffer
- build `event_map` from shared command buffer

## runtime-web (zo-runtime-web).

- rewrite with wry IPC — `EventLoop<String>` with user events, `with_ipc_handler` forwards JS messages to main loop
- `App::user_event` dispatches to `EventRegistry`, reads updated commands from shared buffer, calls `evaluate_script` for DOM updates
- add `set_shared_commands()` method for reactive state wiring
- rewrite bridge.js — replace WebSocket-based `ZoRuntime` class (174 lines) with event delegation via `window.ipc.postMessage` (33 lines)
- remove inline `onclick`/`oninput` handlers from HTML, use `data-id` delegation

## codegen (zo-codegen-arm).

- track reactive bindings and event handlers during template compilation
- generate `_zo_handle_event` dispatch function — CMP + B.NE + BL chain by widget_id
- export `_zo_handle_event` symbol in Mach-O
