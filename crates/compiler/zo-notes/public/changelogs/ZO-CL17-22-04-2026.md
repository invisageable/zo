## consolidate ULEB128 in `zo-writer-macho`.

Three ULEB128 encoders lived side by side in
`macho.rs`:

- `DebugFrameEntry::add_uleb128(&mut self, u64)`
  ‚Äî pushed to `self.cfa_instructions` (7 callers
  in the CFA builders).
- `MachO::encode_uleb128(&self, u32) -> Vec<u8>`
  ‚Äî allocated a fresh `Vec` per call, 45 DWARF
  emission sites.
- `MachO::push_uleb128(buf, u64)` ‚Äî already
  static-style, 1 caller.

All three implemented the same LEB128 loop; a
TODO at the top of the file flagged the
duplication. This changelog entry closes it.

### what landed.

- One module-level canonical function:

  ```rust
  #[inline(always)]
  fn push_uleb128(buf: &mut Vec<u8>, mut value: u64) {
    loop {
      let mut byte = (value & 0x7f) as u8;
      value >>= 7;
      if value != 0 {
        byte |= 0x80;
        buf.push(byte);
      } else {
        buf.push(byte);
        break;
      }
    }
  }
  ```

- All 53 callers rewritten to the free function:
  - 7 √ó `self.add_uleb128(X)` ‚Üí
    `push_uleb128(&mut self.cfa_instructions, X)`.
  - 45 √ó `data.extend(&self.encode_uleb128(X))` ‚Üí
    `push_uleb128(&mut data, X as u64)`. Every
    DWARF site now streams into the buffer
    instead of allocating a fresh `Vec<u8>`.
  - 1 √ó `Self::push_uleb128(&mut data, offset)` ‚Üí
    `push_uleb128(&mut data, offset)`.
- Three method definitions deleted.
- TODO comment at the top of the file removed.
- `encode_sleb128` left alone ‚Äî signed LEB128 is
  a separate encoding with different termination
  logic.

### why it matters.

- One source of truth ‚Äî future changes don't
  have to be kept in sync across three copies.
- Zero-allocation DWARF emission. The old
  `encode_uleb128` allocated a small `Vec<u8>`
  per field; DWARF entries contain dozens of
  ULEB128 fields each. For a compile that emits
  debug info for N functions, that's O(N √ó fields)
  pointless allocations per binary.
- Consistent with zo's data-oriented ethos:
  stream into an existing buffer rather than
  return-by-value.

### tests added.

- `push_uleb128_encodes_canonical_boundaries` ‚Äî
  asserts the canonical encoding at 0, 1, 127,
  128, 255, 16384, 624485 (DWARF 5 Appendix C
  worked example), `u32::MAX`, and `u64::MAX`
  (the 10-byte worst case).
- `push_uleb128_appends_to_existing_buffer` ‚Äî
  pins the invariant every DWARF / CFA call
  site relies on: streaming must append to the
  passed buffer, never overwrite.

The primitive had been exercised indirectly by
the 59-test macho suite, but not pinned at the
function level. Direct coverage guards against
future regressions in the canonical encoder.

## stale blocker audit.

Three entries in auto-memory pointed at
blockers that had already been closed by prior
work. Verified against current code and
removed:

### `project_sir_width_invariants` ‚Äî closed.

Memory claim: `coerce_int_width` in CLIF
backend is a downstream shim; root fix is
executor + tychecker producing width-
consistent SIR.

Current state: closed by CL16's post-plan
"generic mono narrow" work. `grep
coerce_int_width crates/compiler/zo-codegen-clif`
returns zero results. `zo-executor/src/executor.rs:884`
now runs a linear `resolve_id` pass over every
SIR insn, with a comment explicitly stating
this is what lets the CLIF backend drop the
shim.

### `project_module_system_blocker` ‚Äî closed.

Memory claim: user multi-file projects
blocked by multiple interner/tychecker ‚Äî
needs shared instances.

Current state: `zo-module-resolver` crate is
wired into `zo-compiler` (uses shared
`session.interner`), and the `zo-how-zo/`
catalog includes multi-feature programs
(generics, results, IO) that work end-to-end.

### `project_frame_overflow` ‚Äî could not reproduce.

Memory claim: codegen segfaults with many
locals + showln calls (~13+) in one function.

Repro attempts with 14 imu ints + showln,
20 mut ints + showln, and 20 mixed-type
locals (int / str / f64 / bool) + 20 showln
‚Äî all compile and run cleanly on
`arm64-apple-darwin`. No segfault. Likely
fixed as a side effect of the SIR-invariants
work in CL16 (width-consistent literal
emission downstream of Phases 1‚Äì6). Memory
entry removed; will re-add with a concrete
repro if the bug resurfaces.
