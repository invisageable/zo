# ZO-CL25 — 2026-05-01

## `.map` for arrays — closure-inlined loop in SIR.

```zo
imu nums: []int = [1, 2, 3, 4, 5];
imu doubled: []int = nums.map(fn(x) => x * 2);

showln(doubled);   -- [2, 4, 6, 8, 10]
```

Step 3a of `PLAN_ZSX_TODOLIST.md`. `.map` lowers to a
synthesized `for i := 0..arr.len { __res.push(closure(arr[i])) }`
loop. The closure invocation is by-name through `Insn::Call`;
captures (when present) flow through the same shape
`execute_call` already uses for closure values — no new
function-pointer machinery needed yet.

### shape.

```text
mut __map_res_<n>: []U = [];
mut __map_i_<n>:   int = 0;

while __map_i_<n> < arr.len {
  imu __t_<n>: T = arr[__map_i_<n>];
  __map_res_<n>.push(closure(__t_<n>));
  __map_i_<n> = __map_i_<n> + 1;
}
```

Result type `[]U` is inferred from the closure's
`return_ty` after the body has unified its inference vars.
A new pass at the end of `execute_closure` promotes the
FunDef record's `return_ty` to the actual body-expression
type so `.map` (and any future caller) sees the concrete
result type.

### approach.

- **`zo-executor`** — `execute_array_map()` emits the loop:
  ArrayLiteral → VarDef + Store for `__map_res_<n>`;
  ConstInt 0 → VarDef + Store for `__map_i_<n>`; Label;
  Load + ArrayLen + BinOp Lt → BranchIfNot end; ArrayIndex;
  Call closure_name with `[captures..., elem]`; ArrayPush;
  BinOp Add 1 + Store; Jump → loop_start; Label end; final
  Load __map_res_<n> as the expression value.
- **`zo-executor`** — dot-method dispatch (line 15634)
  routes `.map` to `execute_array_map` alongside `.push`
  / `.pop`. `is_dot_method_call` (line 9970) recognizes
  `.map` so the executor defers Dot processing to the
  matching RParen. Non-array receiver or non-closure arg
  reports a hard `TypeMismatch` instead of bailing
  silently.
- **`zo-executor`** — `execute_closure` (line 6210) updates
  the FunDef record's `return_ty` to the body's actual
  return type after the implicit return is emitted.
  Without this, the FunDef carries `unit` (the initial
  default) and `.map` produces `[]Infer` instead of
  `[]int`.

### the load-bearing parser bug.

`fn(x) => x` inside `arr.map(fn(x) => x)` was leaving the
`Fn` introducer open all the way to the next `;`. The
outer call's `)` and the trailing `;` were getting absorbed
into Fn's children, so:

- The closure's `children_end` extended past the outer
  RParen.
- `execute_closure` then ran AFTER the outer RParen had
  already triggered `execute_potential_call` — with the
  closure value still missing from the stack.
- `.map` dispatch saw `ty_stack=[Infer(_)]` (the call's
  return placeholder) instead of `[arr_ty, closure_ty]`
  and bailed.

Fix in `zo-parser::handle_rparen_closer` — cascade-close
any `Token::Fn` introducers on top of the stack before
falling through to the regular RParen close path. Mirrors
the existing `handle_semicolon` cascade
(parser.rs:1212-1214).

## `.map` over user-defined types — element-type propagation.

```zo
struct User { name: str, age: int }

imu users: []User = [
  User { name: "alice", age: 30 },
  User { name: "bob",   age: 25 },
];

imu names: []str = users.map(fn(u) => u.name);
showln(names);   -- ["alice", "bob"]
```

Step 3b of `PLAN_ZSX_TODOLIST.md`. Without explicit
annotation, `fn(u) => u.name` has `u` as a fresh inference
var; `u.name` couldn't resolve to a struct field. Same
pattern as Step 2's `@event` auto-typing — synthesize a
`Fn(elem_ty) -> ?fresh` annotation through `pending_decl`
so the existing param-propagation block (executor.rs:5897)
unifies `u` with the receiver's element type.

### approach.

`execute_closure` peeks at the surrounding context at
entry: if `start_idx - 2` is a `Token::Dot` AND
`dot_method_recv_ty[dot_idx]` is `Ty::Array(_)`, it stashes
a synthetic `pending_decl` whose `annotated_ty = Fn(elem_ty)`.
Saved outer pending_decl is restored after the propagation
block runs but before body execution, so the outer let's
finalize sees its original shape unchanged. Hot-path
discipline: the synthesis is gated behind the cheap
`Token::Dot` check before any HashMap or interner work, so
non-`.map` closures pay one branch.

The .map lowering itself is reused from Step 3a — same
`execute_array_map`, same Insn::Call to the closure. Only
the type-inference scaffolding is new.

Both this site and Step 2's `@event` auto-typing now go
through one shared
`Executor::install_synthetic_fn_pending_decl(name, param_ty,
span)` helper — the synthetic `PendingDecl` literal lived
in two places and they were drifting.

## `@event` payload restore — clobbered `pending_decl`.

Surfaced while extracting the shared
`install_synthetic_fn_pending_decl` helper. Step 2's
`@input`/`@change` path always restored
`saved_pending_decl` after `execute_closure`, even when
the synthetic install never ran (e.g. `Event` symbol not
resolvable at that scope). The restore would overwrite
the live outer `pending_decl` with `None`, silently
dropping the let-binding's annotation for the
surrounding `imu view: </> ::= …`.

Fix: track install via `Option<Option<PendingDecl>>` —
outer `Some` means "synthesis fired, restore saved",
outer `None` means "left pending_decl alone". Same shape
applied to the new `.map` site so they can't drift.
