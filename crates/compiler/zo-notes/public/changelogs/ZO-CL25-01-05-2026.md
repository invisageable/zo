# ZO-CL25 — 2026-05-01

## `.map` for arrays — closure-inlined loop in SIR.

```zo
imu nums: []int = [1, 2, 3, 4, 5];
imu doubled: []int = nums.map(fn(x) => x * 2);

showln(doubled);   -- [2, 4, 6, 8, 10]
```

Step 3a of `PLAN_ZSX_TODOLIST.md`. `.map` lowers to a
synthesized `for i := 0..arr.len { __res.push(closure(arr[i])) }`
loop. The closure invocation is by-name through `Insn::Call`;
captures (when present) flow through the same shape
`execute_call` already uses for closure values — no new
function-pointer machinery needed yet.

### shape.

```text
mut __map_res_<n>: []U = [];
mut __map_i_<n>:   int = 0;

while __map_i_<n> < arr.len {
  imu __t_<n>: T = arr[__map_i_<n>];
  __map_res_<n>.push(closure(__t_<n>));
  __map_i_<n> = __map_i_<n> + 1;
}
```

Result type `[]U` is inferred from the closure's
`return_ty` after the body has unified its inference vars.
A new pass at the end of `execute_closure` promotes the
FunDef record's `return_ty` to the actual body-expression
type so `.map` (and any future caller) sees the concrete
result type.

### approach.

- **`zo-executor`** — `execute_array_map()` emits the loop:
  ArrayLiteral → VarDef + Store for `__map_res_<n>`;
  ConstInt 0 → VarDef + Store for `__map_i_<n>`; Label;
  Load + ArrayLen + BinOp Lt → BranchIfNot end; ArrayIndex;
  Call closure_name with `[captures..., elem]`; ArrayPush;
  BinOp Add 1 + Store; Jump → loop_start; Label end; final
  Load __map_res_<n> as the expression value.
- **`zo-executor`** — dot-method dispatch (line 15634)
  routes `.map` to `execute_array_map` alongside `.push`
  / `.pop`. `is_dot_method_call` (line 9970) recognizes
  `.map` so the executor defers Dot processing to the
  matching RParen. Non-array receiver or non-closure arg
  reports a hard `TypeMismatch` instead of bailing
  silently.
- **`zo-executor`** — `execute_closure` (line 6210) updates
  the FunDef record's `return_ty` to the body's actual
  return type after the implicit return is emitted.
  Without this, the FunDef carries `unit` (the initial
  default) and `.map` produces `[]Infer` instead of
  `[]int`.

### the load-bearing parser bug.

`fn(x) => x` inside `arr.map(fn(x) => x)` was leaving the
`Fn` introducer open all the way to the next `;`. The
outer call's `)` and the trailing `;` were getting absorbed
into Fn's children, so:

- The closure's `children_end` extended past the outer
  RParen.
- `execute_closure` then ran AFTER the outer RParen had
  already triggered `execute_potential_call` — with the
  closure value still missing from the stack.
- `.map` dispatch saw `ty_stack=[Infer(_)]` (the call's
  return placeholder) instead of `[arr_ty, closure_ty]`
  and bailed.

Fix in `zo-parser::handle_rparen_closer` — cascade-close
any `Token::Fn` introducers on top of the stack before
falling through to the regular RParen close path. Mirrors
the existing `handle_semicolon` cascade
(parser.rs:1212-1214).

## `.map` over user-defined types — element-type propagation.

```zo
struct User { name: str, age: int }

imu users: []User = [
  User { name: "alice", age: 30 },
  User { name: "bob",   age: 25 },
];

imu names: []str = users.map(fn(u) => u.name);
showln(names);   -- ["alice", "bob"]
```

Step 3b of `PLAN_ZSX_TODOLIST.md`. Without explicit
annotation, `fn(u) => u.name` has `u` as a fresh inference
var; `u.name` couldn't resolve to a struct field. Same
pattern as Step 2's `@event` auto-typing — synthesize a
`Fn(elem_ty) -> ?fresh` annotation through `pending_decl`
so the existing param-propagation block (executor.rs:5897)
unifies `u` with the receiver's element type.

### approach.

`execute_closure` peeks at the surrounding context at
entry: if `start_idx - 2` is a `Token::Dot` AND
`dot_method_recv_ty[dot_idx]` is `Ty::Array(_)`, it stashes
a synthetic `pending_decl` whose `annotated_ty = Fn(elem_ty)`.
Saved outer pending_decl is restored after the propagation
block runs but before body execution, so the outer let's
finalize sees its original shape unchanged. Hot-path
discipline: the synthesis is gated behind the cheap
`Token::Dot` check before any HashMap or interner work, so
non-`.map` closures pay one branch.

The .map lowering itself is reused from Step 3a — same
`execute_array_map`, same Insn::Call to the closure. Only
the type-inference scaffolding is new.

Both this site and Step 2's `@event` auto-typing now go
through one shared
`Executor::install_synthetic_fn_pending_decl(name, param_ty,
span)` helper — the synthetic `PendingDecl` literal lived
in two places and they were drifting.

## `@event` payload restore — clobbered `pending_decl`.

Surfaced while extracting the shared
`install_synthetic_fn_pending_decl` helper. Step 2's
`@input`/`@change` path always restored
`saved_pending_decl` after `execute_closure`, even when
the synthetic install never ran (e.g. `Event` symbol not
resolvable at that scope). The restore would overwrite
the live outer `pending_decl` with `None`, silently
dropping the let-binding's annotation for the
surrounding `imu view: </> ::= …`.

Fix: track install via `Option<Option<PendingDecl>>` —
outer `Some` means "synthesis fired, restore saved",
outer `None` means "left pending_decl alone". Same shape
applied to the new `.map` site so they can't drift.

## `.map` returning `[]</>` — list-rendering scaffold.

```zo
imu items: []str = ["a", "b"];
imu view: </> ::= <ul>{items.map(fn(t) =:> <li>{t}</li>)}</ul>;
#dom view;
```

Step 3c of `PLAN_ZSX_TODOLIST.md`. The closure body returns
`</>`, the call's result is `[]</>`, and the template
interp recognizes the shape and registers a `ListBinding`
in `TemplateBindings`. The wip todolist program now builds
and launches the runtime end-to-end (window opens with the
static frame; reactive growth on `Add` requires array
state-cell support — separate follow-up).

### shape.

```rust
// zo-sir
pub struct ListBinding {
  pub items_var: Symbol,
  pub item_template: Vec<ListItemCmd>,
}

pub enum ListItemCmd {
  Element { tag: ElementTag, attrs: Vec<Attr> },
  EndElement,
  Text(String),
  TextFromItem,
}
```

`item_template` is a per-item recipe. The runtime walks it
once per element of `items_var`, substituting
`TextFromItem` with the element's stringified value, and
splices the resulting commands into the parent buffer at
the placeholder slot.

### the load-bearing lexer bug.

`<ul>{arr.map(fn(t) =:> <li>{t}</li>)}</ul>` exposed two
problems with `=:>` firing inside an `{...}` interp:

- The lexer's TEMPLATE-vs-CODE gate keyed off `brace_depth >
  0`, forcing CODE for every byte inside the interp. So
  `<li>` tokenized as `Lt`/`Ident`/`Gt` rather than
  `LAngle`/`Ident`/`RAngle` — the closure body looked
  syntactically like a comparison.
- The `>` arm unconditionally re-enabled `template_text`
  mode, so after `</li>` the trailing `)` of `.map(...)`
  got eaten as template text instead of re-tokenizing as
  `RParen`.

### approach.

- **`zo-tokenizer`** — three new fields on `Tokenizer`:
  - `template_frame_base: u32` — `brace_depth` snapshot
    taken on `=:>` from inside an interp. The
    TEMPLATE-vs-CODE gate becomes `brace_depth >
    template_frame_base` so closure-body markup tokens
    survive the gate even at `brace_depth > 0`.
  - `template_element_depth: u32` + `template_element_depth_at_frame_entry: u32`
    — element nesting is tracked via the `>` of open vs
    close tags. When the depth returns to the entry
    snapshot inside a frame, the closure body's outermost
    element has just closed — surrender the frame and
    suppress `template_text` so the trailing `)` re-enters
    code-token scanning.
  - `in_close_tag: bool` — set on `</…` (the `<` peek
    immediately sees `/`), consumed by the matching `>`.
    Drives the open/close discrimination above.
- **`zo-parser`** — `handle_rparen_closer` cascade now
  also closes a synthetic `TemplateFragmentStart` sitting
  directly above an `Fn` — that's the closure body's
  wrapper from `handle_template_fat_arrow`. Without this,
  the wrapper stayed open until the next `;`, absorbing
  the surrounding `}`, `</ul>`, `;`, AND the following
  `#dom view;` directive as Fn children, so neither the
  template nor the directive made it into the SIR. The
  rule is conservative: only fires when the
  `TemplateFragmentStart` immediately follows an `Fn` on
  the introducer stack — `imu view ::= <body>`'s
  top-level synthetic fragment is left alone (it must
  survive until `;`).
- **`zo-sir`** — new `bindings.list: Vec<(usize,
  ListBinding)>` next to `text` / `attrs` / `computed`.
  `ListBinding { items_var, item_template }` and
  `ListItemCmd { Element / EndElement / Text /
  TextFromItem }`.
- **`zo-executor`** — `execute_template_fragment`'s
  `Token::LBrace` interp arm grew a third fast path
  (after single-ident and before general-expression).
  `try_extract_list_binding` matches the `Ident Ident
  Dot LParen Fn ... RParen` token shape; on hit,
  `build_item_recipe` walks the closure body and emits
  `ListItemCmd`s. The interp slot gets a placeholder
  `Text(String::new())` plus a `ListBinding` registration.
- **`zo-driver`** — drains `bindings.list` into a
  per-template-rebased `Vec<(usize, ListBinding)>` and
  registers a `StateCell` per `items_var` so events can
  re-render the list (see "reactive list growth" below).
- **`zo-ui-protocol`** — new `ElementTag::from_name(&str)
  -> Option<Self>` next to `EventKind::from_name`. Maps
  HTML tag names to the typed enum, falls through to
  `Custom(String)` for tags the compiler hasn't
  enumerated.

## reactive list growth — clicking Add appends a `<li>`.

Closes the wip todolist loop on both native and web:
typing in the input, clicking Add, the new `<li>` appears,
the input clears, and you can keep going. Built on the
Step 3c scaffold above plus six unified pieces across the
runtime stack.

### approach.

- **`zo-runtime-render::StateValue`** — new
  `Strs(Vec<String>)` variant + `as_strs() -> Option<&[String]>`
  accessor. Backs `mut []str` reactive state. `display()`
  uses the established `["a", "b"]` array-stringification
  shape (matches `array_map_struct.zo`'s expected output).
- **`zo-runtime-render::StateCell`** — new
  `with_strs<R>(|&[String]| -> R) -> Option<R>` borrow API
  + `is_strs()` cheap peek. Both let consumers walk the
  array under the cell's lock without cloning the whole
  `Vec<String>` per event.
- **`zo-runtime-render::HandlerEvaluator`** —
  `Val::StrArrRef(usize)` sentinel materialized by
  `Load Param(N)` when the captured slot holds `Strs(_)`.
  New `Insn::ArrayPush` arm mutates the underlying state
  cell via `state[slot].mutate(...)`. `is_strs()` peek
  on the param-binding decision avoids the upfront clone.
- **`zo-driver::run`** — `find_initial_value` recognizes
  `Insn::ArrayLiteral` and seeds `StateValue::Strs(...)`
  (empty for `mut todos: []str = []`, populated for
  `["a", "b"]`). New `apply_list_bindings(new_cmds,
  list_binds, cells)` walks each binding's recipe once
  per element — `ListItemCmd::TextFromItem` substitutes
  the element value — and `Vec::splice`s the rendered
  commands into the placeholder slot. Runs both in the
  per-event closure (after computed bindings) and on
  initial render.
- **`zo-runtime-native::renderer`** — new
  `last_value_attr: HashMap<u32, String>` per input id.
  Each frame compares the current `value` attribute
  against the snapshot; on mismatch (program-side
  `input_val = ""`) the renderer's `text_inputs[id]` is
  re-synced from the attribute. Bidirectional binding
  stays intact: user typing fires `@input` which updates
  `input_val`, the next frame's attribute now matches the
  typed text → no resync, no clobber.
- **`zo-runtime-web::runtime`** — `build_patch_js` gains
  two unified rules. (1) Length mismatch → full body
  re-render via `HtmlRenderer::render_body_inner` (cached
  on `App` so no per-event allocation; bridge.js delegates
  from `document` so handlers survive the innerHTML swap).
  (2) `<input>` / `<textarea>` `value` attribute change
  also sets the live `.value` JS property — `setAttribute`
  alone updates only the *default* value, leaving typed
  text in place. The wip's `input_val = ""` now actually
  empties the field on web.
- **`zo-runtime-web::renderer`** — `render_to_html` now
  delegates to `render_body_inner` for the body content,
  killing ~50 lines of duplicated event-map / scope-class /
  command-walk setup. The two paths can no longer drift.

### known deficits.

- The web length-mismatch fallback replaces
  `document.body.innerHTML` wholesale. Loses input focus,
  scroll position, and IME composition state on every list
  growth. Acceptable for the wip's empty-list-grows-by-one
  shape; revisit when keyed reconciliation lands.
- `apply_list_bindings` clones `tag` / `attrs` per item
  per event. Fine through ~100 items; for larger lists,
  move recipe data behind `Arc<[Attr]>` so per-item
  materialization is a pointer bump instead of a Vec
  clone.

## `@submit` — Enter key in text inputs.

```zo
<input
  value={input_val}
  @input={fn(e) => input_val = e.value}
  @submit={fn(e) => {
    todos.push(e.value);

    input_val = "";
  }}
/>
```

`EventKind::Submit` joins `Click | Hover | Change | Input |
Focus | Blur` in `zo-ui-protocol`. Payload-bearing like
`@input` / `@change` — the closure receives the input's
current text via `e.value`. Native binding fires on egui's
`lost_focus() && enter_pressed`, refocuses the field so the
user can keep typing. Web bridge attaches a `keydown`
listener that posts `submit:id:value` on Enter and
preventDefaults only when an input claims the keystroke.

### dispatch unification.

The pre-existing `event_map` was keyed widget-only — a
single element with both `@input` and `@submit` would lose
one handler on the second insert. Both runtimes now key on
`(widget_id, EventKind)` via a shared
`build_event_map(commands) -> HashMap<(String, EventKind),
String>` helper in `zo-runtime-render`.

The native runtime previously built the map once at
startup; reactive re-renders that emit new `Event`
commands (list items via `apply_list_bindings`,
conditional branches) silently dropped clicks. Web
previously did an O(N) linear `find_map` per IPC frame.
Both now rebuild from the freshly-locked command buffer
on each dispatch — single source of truth, single
implementation.
