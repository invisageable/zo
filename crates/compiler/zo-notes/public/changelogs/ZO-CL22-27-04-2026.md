# ZO-CL22 — 2026-04-27

## `while cond => expr;` — line form parity with `for`.

```zo
mut i: int = 0;
while i < 5 => i += 1;
showln(i);  // expected: 5
```

printed `1`. Same shape as the recently-fixed
`loop { ... }` bug: the line form parsed but never
looped. `execute_while` only emitted a label and
pushed a branch ctx — the main walk drives the cond
+ LBrace `BranchIfNot` + RBrace close for the **block**
form, but there's no `=>`/Semicolon hook in the main
walk, so the line form ran the body once and fell
through with no back-edge.

### approach.

Mirror `execute_for`'s line-form inline finalize. The
block form keeps the existing main-walk-driven path
(works fine, identical for both for and while).
`execute_while` now scans for `LBrace` vs `FatArrow` at
header depth 0 up front; if it finds a `FatArrow`, it
inline-finalizes:

```text
loop_label:
  <sub-walk cond from idx+1..fat_arrow_idx>
  cond_sir = pop sir_values
  BranchIfNot cond_sir, end_label
  push_scope
  <sub-walk body from fat_arrow_idx+1..=semicolon_idx>
  drain leftover stack
  Jump loop_label
end_label:
  pop_scope
  branch_stack.pop
  skip_until = semicolon_idx + 1
```

The block form is unchanged: scan returns `None`, the
function pushes the branch ctx and returns; the main
walk continues with the cond expression and the LBrace
fires the existing branch-emission path.

### executor.

`zo-executor/src/executor.rs` —
`execute_while(start_idx, end_idx)` now takes the real
indices (was `_start_idx`, `_end_idx`). New
`scan_while_body_marker` walks the header at depth 0,
returns `Some((fat_arrow_idx, semicolon_idx))` for line
form or `None` for block form. Paren/bracket depth
tracking matches the for-loop scanner so a `=>` inside
e.g. `f(a => b)` (hypothetical) wouldn't be mistaken
for the loop-body marker.

### regression.

`programming/99-bottles-of-beer-on-the-wall.zo` golden
re-baked to its full 397-line output (was an `EXPECTED
OUTPUT: DONE` stub from the WIP move). The program
itself uses the **block** form `while count > 0 { ... }`
and was always correct — only the test's golden was
incomplete. Test now exercises the full song through
shipping.

### suite.

1005 / 1005 unit tests, 338 / 338 integration green;
clippy clean.

## `readln()` / `read()` — wired end-to-end.

Both were declared as FFI in `std/io.zo` but had no
runtime export and no codegen wiring. Programs that
called them would have dyld-failed at link time.
This is item §1.2 of the stabilization quest;
**shipped**.

### runtime.

New `zo-runtime/src/io.rs`:
- `_zo_io_readln(buf, max_len) -> isize` — single
  `libc::read(0, …)` then truncate the returned len
  at the first `\n`.
- `_zo_io_read(buf, max_len) -> isize` — single
  `libc::read(0, …)`, no truncation. Multi-read
  accumulation deferred (covers piped + line-
  buffered tty input under the buffer size, which
  is the 99% case).
- Negative return is `-errno` so the caller can
  build `Result::Err`.
- 3 Rust unit tests for `truncate_at_newline`
  (no-newline / one-newline / first-newline-wins).

### codegen.

`emit_io_readln` / `emit_io_read` in
`zo-codegen-arm/src/codegen.rs` mirror the
`emit_io_read_file` shape minus the open/close
syscalls. Both share an `emit_io_read_stdin`
helper that:

1. Sets `X0 = buf_ptr`, `X1 = max_len`.
2. `BL` to the runtime helper.
3. Branches on sign — `n >= 0` → `Result::Ok(str)`
   built on stack with `n` as length prefix; `n < 0`
   → `Result::Err(errno)` with `errno = -n`.
4. Returns the `Result` pointer in `dst`.

Buffer size is the existing `READ_FILE_BUF_SIZE`
(4096 bytes). Stack layout matches `read_file`
exactly — `result_base` carries `[tag][field]`,
`scratch_off` saves the signed return, then the
str header + buffer follows.

### compiler.

`RuntimeNeeds::from_sir` in
`zo-compiler/src/compiler.rs` extends its name
filter to flag `concurrency = true` for `readln`
and `read` so `libzo_runtime.dylib` stages next
to the user binary. The `_zo_io_*` exports route
through the existing `ZO_RUNTIME_SYMBOL_PREFIX`
machinery — no new dylib, no new linker
plumbing.

### regression.

End-to-end verified:
- `echo "hello world" | bin` → `hello world` from
  `readln()`.
- `echo "alpha\nbeta\ngamma" | bin` → all 3 lines
  back from `read()`.

The integration test runner has no stdin-pipe
support today, so no `programming/*.zo` golden —
the Rust unit tests cover the truncation logic and
the codegen path is exercised at every build that
uses the symbols.

### suite.

1008 / 1008 unit tests, 338 / 338 integration green;
clippy clean.

## `args() -> []str` — process arguments + FFI `[]T` parsing.

**`args()`.** `std/io.zo` declares `pub ffi args() ->
[]str;`. `_zo_args` (`zo-runtime/src/io.rs`) walks
`std::env::args_os().skip(1)`, allocates a zo str per
arg via `alloc_str` (made `pub(crate)`), packs the
pointers into a heap-leaked `[len:u64][cap:u64]
[ptr0]...[ptrN]`, memoizes via `OnceLock<usize>` so
loop-calls return the same pointer.
`emit_io_args` is `BL _zo_args; mov dst, X0`.
`RuntimeNeeds::from_sir` flags `args` for dylib
staging.

**FFI `[]T` parsing fix.** `execute_ffi`'s return- and
param-type parsers both called `resolve_type_token`
(single-token), silently typing `[]T` as `unit`. So
`pub ffi nope() -> []int` failed analyzer binding,
and `pub ffi nope(xs: []int)` mis-typed the param.
Both arms now check `Token::LBracket` and dispatch
to the existing `resolve_array_type` (handles `[]T`,
`[N]T`, multi-dim).

### regressions.

- `bin one two three` → `3` + each arg.
- `bin` (no args) → `0`.
- `pub ffi nope() -> []int; imu xs: []int = nope();
  showln(xs.len);` compiles and runs.
- Two consecutive `args()` calls share the same
  memoized pointer.

### deferred (stabilization §5b).

- `alloc_str_array` + shared `RUNTIME_ARRAY_HEADER_
  SIZE` / `RUNTIME_SLOT_SIZE` constants in
  `zo-runtime/src/arr.rs`. Every `[]str` runtime
  helper currently re-derives the `16 + n*8` math.
- Unify the four type-resolve call sites
  (`execute_fun_def` × 2, `execute_ffi` × 2) behind
  one helper branching on `LBracket` / `LParen` /
  `FnType` / scalar. Three instances of the same
  bug class is the signal.

### suite.

1005 / 1005 unit, 338 / 338 integration green;
clippy clean.

## `[N]T` static arrays stack-allocate (stabilization §2.1).

Yesterday's `ArrayLiteral` heap-unification fixed the
`realloc`-on-stack UB by paying one `_malloc` per
literal — including statically-sized `[N]T = [...]`
that never grow. End-to-end fix across three layers:

**SIR.** `Insn::ArrayTyDef` gains `size: Option<u32>`
(`Some(N)` for `[N]T`, `None` for `[]T`). Validator,
pp, visitors updated. Codegen merges with `array_metas`
into a single `HashMap<u32, ArrayMeta { elem_ty, size
}>`.

**Type checker.** `finalize_pending_decl` rewrites the
SIR `Insn::ArrayLiteral.ty_id` to the binding's
annotation when shapes differ. Without it,
`mut v: []int = [10, 20, 30]` left the literal typed
`[3]int` (static) while the binding was `[]int`
(dynamic) — codegen would stack-allocate, then
`v.push(40)` would `realloc` the stack pointer (UB).
The rewrite scan is bounded by `PendingDecl
.init_start_idx`, snapshotted in `begin_decl` — never
misses a deeply-nested literal. Element compat is
already enforced by the surrounding `unify`; the
shape check is just `kind_of(_) == Ty::Array`.

**Codegen.** `Insn::ArrayLiteral` consults
`array_metas[ty_id].size`: `Some` → stack-allocate
the `[len:8][cap:8][e0:8]...[eN:8]` block in the
function frame (zero `_malloc`); `None` → heap path.

### regressions.

- `imu a: [3]int = [10, 20, 30]` disassembles with
  zero `bl _malloc` in main (pure stack stores).
- `mut v: []int = [10, 20, 30]; v.push(40)` heap-
  allocates, grows.
- `mut v: []bool = []; v.push(true); v.push(false)` —
  empty literals with `Infer($T)` element types are
  rewritten correctly (no elem-equality gate; the
  prior `unify` is the source of truth).

### suite.

1005 / 1005 unit, 338 / 338 integration green;
clippy clean.

## `HashMap<(int, int), V>` tuple keys (stabilization §2.2).

Tuple-keyed maps compiled and ran but every `get`
returned `MISS`. Three layers, three bugs:

**Generic-args parser.** `begin_decl`'s annotation
walker only handled single-token type args.
`HashMap<(int, int), int>` unwrapped to `[int, int,
int]` instead of `[(int,int), int]`. Now dispatches
to `resolve_tuple_type` on `Token::LParen` (mirrors
the FFI parser's `LBracket` dispatch).

**`collection_new_type_args`.** Hardcoded `key_sz =
8`, `key_kind = 0|1`. Now detects `Ty::Tuple(_)`,
returns `(KeyKind::Tuple, n * 8)`.

**Runtime.** `KeyKind::Tuple` previously read the
slot as inline payload (like `Prim`) but the codegen
spills a pointer to the tuple's stack-allocated
payload. Now dereferences (like `Str`) and reads
`key_sz` bytes from the tuple's actual address. Hash,
eq, and to_vec unified through one
`key_payload(key_ptr) -> &[u8]` helper.

### regressions.

- `m.insert((1,2), 100); m.get((1,2))` → `100`.
- `m.get((3,4))` → `200` after insert.
- `m.get((9,9))` → `MISS`.

### suite.

1005 / 1005 unit, 338 / 338 integration green;
clippy clean.

## UnOp executor fallthrough + dead variants (stabilization §2.3).

`execute_unop` had a "TODO: Handle these properly"
ladder that short-circuited `UnOp::Neg` to
`rhs_ty` — so `-true` typechecked silently — and
`UnOp::BitNot` had a type-checker rule that the
executor never invoked. The unop dispatch table
double-assigned `Token::Star` (`Deref` then `BitNot`,
the second overwriting the first). And `UnOp::Ref` /
`UnOp::Deref` were dead variants: zo's grammar
exposes only `!`, `-`, `+` as prefix unary, so no
parser path emitted them, and codegen-arm never
handled them.

Routed the executor through `ty_checker.infer_unop`
(single source of truth: `Neg` rejects bools, `Not`
unifies with bool, `BitNot` rejects floats). Dropped
`Ref` / `Deref` from the SIR `UnOp` enum, from pp,
from codegen-clif's `translate_unop`. Dropped the
`Token::Amp` / `Token::Star` table entries. Removed
unreachable catch-all arms in tychecker /
constfolding now that `UnOp` exhausts on three
variants.

### suite.

1005 / 1005 unit, 338 / 338 integration green;
clippy clean.

## enum-typed struct fields work end-to-end (stabilization §3.1).

Repro `Box { c: Color::Green, n: 42 }` failed at
parse, then at execute, then at pretty-print. Fixed
all three:

**Struct decl parser.** `execute_struct`'s field-type
arm matched `Token::Dollar` (generics) and
`is_ty()` (builtin keywords) only. User-defined
idents (`Color`) fell through, leaving the type token
unconsumed — the next loop iteration parsed it as a
NEW field name, producing a phantom `Color` field.
Fix: add `Token::Ident` to the arm.

**Struct-construct walker.** `try_struct_construct`'s
field-value walker called `execute_node` blindly per
index. `Color::Green` triggers `execute_enum_access`
which sets `skip_until = idx + 2` — but the walker
ignored it, so the `Green` ident node ran through the
plain Ident handler and reported `UndefinedVariable`.
Fix: respect `skip_until` (`if idx < self.skip_until
{ idx += 1; continue; }`), matching every other
walker in the executor.

**Pretty-printer fall-through.** `emit_field_write`
in `zo-codegen-arm/src/codegen.rs` dispatched
arrays / structs / str / float / bool / char but
never enums; enum fields fell through to
`emit_itoa_and_write` and printed the enum pointer
as an integer. Factored `emit_enum_walk_from_x19`
out of `emit_enum_write` (mirrors the existing
`emit_array_walk_from_x19` / `emit_struct_walk_from_x19`
shape — X19 holds the value, recursion is
`emit_pp_state_push` → mov X0→X19 → walk → pop).
Added the dispatch in `emit_field_write`. Generic
substitution (`value_enum_field_tys`) only applies
at top-level entry, so the nested case uses the
template's `enum_metas` field types — fine for the
non-generic case targeted here.

### regressions.

- `Box { c: Color::Green, n: 42 }; showln(b);` →
  `Box { c: Color::Green, n: 42 }`.
- `showln(b.c)` → `Color::Green`.
- New `programming/struct-with-enum-field.zo` golden
  test locks the shape end-to-end.

### suite.

1005 / 1005 unit, 339 / 339 integration green;
clippy clean.
