# ZO-CL18 — 2026-04-23

## channels plan — Phase 0 semantic decisions.

`PLAN_CHANNELS.md` had six open semantic questions
that gated Phases 4-6 (executor, codegen, runtime).
Resolved in a new "Phase 0 — Semantic Decisions"
section so the ABI doesn't churn across phases.

### the six calls.

1. **Panic propagation** — stored `Result` at task
   boundary (`Completed(T)` / `Panicked(p)` /
   `Cancelled`), not DWARF unwinding. Identical
   mental model to Go's `recover`. Phase 6 runtime
   owns the `_zo_panic` hook.
2. **`Tx` clone, `Rx` move-only** — API-enforced
   via exposed method surface (`Tx::clone` yes,
   `Rx::clone` no). No affine/linear type system
   required. Hazard: struct-field copy smuggling,
   documented.
3. **`spawn` outside a nursery** — hard compile
   error. No implicit main-nursery. Checked in
   Phase 4 executor via `nursery_depth == 0`.
4. **`spawn fn()` unit return** — produces
   `Task<unit>`, no special case. Fire-and-forget
   is just unbound handle.
5. **`channel(N)` capacity** — integer literal
   only in MVP. Variables / `val` constants post-
   MVP.
6. **Threading model** — `pthread_create` per
   task (real multi-core parallelism), NOT
   `pthread_cancel`. Cancellation cooperative,
   polled at send/recv/await entry on shared
   `nursery.cancelled`. Supersedes the plan's
   earlier "single-threaded cooperative MVP"
   framing, which contradicted its own Phase 6.

## channels plan — Phase 1: Ty variants.

Per `PLAN_CHANNELS.md` Phase 1. Types-only layer,
purely additive — zero dependencies on decisions
1-6 above.

### what landed.

- `Ty::Channel(TyId)` and `Ty::Task(TyId)` variants
  added to `zo-ty/src/ty.rs`. Follow the plan's
  literal shape (wrap inner `TyId` directly rather
  than interning via side table) because there's
  no extra structural data and `Ty` already dedups
  via the tychecker's `intern_map`.
- Constructor helpers on `TyChecker`:
  - `channel_type(elem_ty: TyId) -> TyId`
  - `task_type(return_ty: TyId) -> TyId`
- Unification arms for `(Ty::Channel, Ty::Channel)`
  and `(Ty::Task, Ty::Task)` — both unify their
  single inner `TyId` and return the canonical
  representative. Matches the existing single-
  inner-type pattern (mirror of `Ty::Ref`).
- 7 smoke tests:
  - `channel_ty_interning_dedups_equal_elem`
  - `channel_ty_distinct_on_different_elem`
  - `channel_ty_unifies_through_element`
  - `task_ty_interning_dedups_equal_return`
  - `task_ty_distinct_on_different_return`
  - `task_ty_unifies_through_return`
  - `channel_and_task_are_distinct_variants`

### why direct-wrap vs interned side table.

Every other compound `Ty` variant (`Array`, `Ref`,
`Fun`, `Tuple`, `Struct`, `Enum`) uses a dedicated
`*TyId` newtype + `TyTable` side array. That
pattern earns its keep when the variant carries
more than one field (array has size, ref has
mutability, fun has params). `Channel` and `Task`
each carry exactly one `TyId` — a second level
of indirection would just add a hashmap lookup
per intern with zero semantic benefit. The plan
itself called out `Ty::Channel(TyId)` / `Ty::Task(TyId)`
and this matches.

### exhaustiveness check.

`cargo check --workspace` reports clean after the
two new variants. No downstream code uses an
exhaustive `match` on `Ty` that would have
silently broken — every consumer already handles
unknown variants via a wildcard arm or specific-
variant dispatch.

## channels plan — Phase 2: parser.

Per `PLAN_CHANNELS.md` Phase 2. Tokens
(`Token::Nursery`/`Spawn`/`Await`) were already
recognized by the tokenizer (see CL18 Phase 1
audit); the parser had zero handlers for them.
This phase lands the three handlers plus the
closing cascades.

### tree-shape contracts.

Three different ways to land concurrency keywords
in the tree, chosen to match what the executor
will consume in Phase 4:

- **`nursery { body }`** — introducer. Emitted
  BEFORE its body block. The `{` opens via the
  normal LBrace introducer path; when that block's
  `}` closes, a new arm in the
  `handle_rbrace_closer` cascade also closes the
  Nursery. Mirrors `While` / `For`.
- **`spawn callee(args)`** — introducer, closed on
  `;` like `Return`. Emitted before the call
  subtree so the executor sees `Spawn` FIRST and
  can route the upcoming call to `TaskSpawn` SIR
  (not `Call`). Post-hoc rewriting was rejected —
  the executor can't undo a `Call` insn after
  emitting it.
- **`await expr`** — prefix-unary via the existing
  `unary_spans` stash (same mechanism as `!`,
  `-`, `&`). Lands postfix after the operand it
  applies to. Works mid-expression naturally
  (`showln(await t)` → `t Await` inside the arg
  buffer), which matches the plan's example
  syntax.

### why `await` as unary-stash, not introducer.

The plan shows `await` in mid-expression positions
(`showln(await result)`). An introducer would
have to close on `,` / `)` as well as `;`, which
means extending three cascade sites. The stash
mechanism already handles the
"unary-applies-to-next-operand-or-call-chain"
case — `!f(x)` and `await f(x)` want the same
lifting semantics. Reusing it is strictly
simpler.

### why `spawn` as introducer, not unary-stash.

Unary-stash emits postfix: for `spawn compute(42)`
the tree would read `compute LParen 42 RParen
Spawn`. The executor's call pipeline emits a
`Call` SIR insn at the matching RParen — BEFORE
it sees the trailing `Spawn`. Since SIR
instructions are append-only, there's no clean
way to rewrite that into a `TaskSpawn`. Putting
`Spawn` as a prefix introducer flips the order:
the executor sees it first and can install a
"next call is a spawn" state before descending
into the body.

### closing cascades.

- `handle_rbrace_closer`: after a block's `}`
  closes, if the parent introducer is `Nursery`,
  auto-close it. One new arm added next to
  `While` / `For`.
- `handle_semicolon`: `Token::Spawn` added to the
  list of introducers that close on `;`, next to
  `Return`.

No cascade change for `Await` — unary-stash is
drained by `handle_operand` when the operand
lands, not by an introducer boundary.

### decision 5 deferred to executor.

The plan said Phase 2 enforces "`channel(N)`
capacity must be `Token::Int`". In practice the
parser has no identifier-aware logic — it doesn't
know that the callee `channel` is a built-in.
Adding that knowledge would bleed semantic
analysis into the parser, which the architecture
forbids. Enforcement moves to Phase 4 (executor),
where `channel` is resolved as a built-in and the
arg-literal check fits naturally next to the
existing call-arg type checks.

### tests added.

New module `zo-parser/src/tests/concurrency.rs`
with 6 tests pinning the tree-shape contracts:

- `nursery_empty_block_auto_closes` — bare
  `nursery {}` ends with two `RBrace`s and
  nothing else.
- `nursery_with_spawn_body` — basic
  `nursery { spawn foo(); }` shape.
- `spawn_call_closes_on_semicolon` —
  `spawn compute(42);` — Spawn terminates at `;`,
  enclosing block stays flat.
- `await_ident_emits_postfix_unary` —
  `imu x := await task;` — Await lands AFTER
  `task` in the tree (postfix drain).
- `await_inside_function_arg` —
  `showln(await t);` — Await inside call args,
  RParen still closes call cleanly.
- `nursery_multiple_spawns_flat` — two
  back-to-back `spawn` statements — each Spawn
  closes on its own `;`, stack stays clean
  between them.

## channels plan — Phase 3: SIR instructions.

Per `PLAN_CHANNELS.md` Phase 3. Seven new
`Insn` variants on `zo-sir/src/sir.rs` — still
purely additive; no code currently emits them.
The executor (Phase 4) will be their first
producer, the codegen (Phase 5) their first
consumer.

### variants landed.

```rust
Insn::ChannelCreate {
  tx: ValueId,
  rx: ValueId,
  elem_ty: TyId,
  capacity: u32,
}
Insn::ChannelSend { channel, value, ty_id }
Insn::ChannelRecv { dst, channel, ty_id }
Insn::TaskSpawn { dst, callee, args, ty_id }
Insn::TaskAwait { dst, task, ty_id }
Insn::NurseryBegin { label: u32 }
Insn::NurseryEnd { label: u32 }
```

Two deviations from the plan's literal sketch,
both documented in code comments:

- **`ChannelCreate.capacity: u32`** — plan didn't
  mention a capacity field; added inline (not as
  a separate insn) since channels need to know
  their bound at construction. `0` is unbuffered
  rendezvous; any positive value is a bounded
  buffer. Phase 0 decision 5 still stands —
  capacity must originate from an integer literal,
  enforced by the executor at the built-in
  `channel()` call site.
- **`ChannelSend.ty_id`** — plan had no ty_id
  on Send; added for validator hygiene so the
  sent-value's type can be checked against the
  declared element type without cross-referencing
  the Channel's definition insn.

### visitor coverage.

Both SIR walkers extended:

- `Insn::visit_value_ids_mut` — covers all new
  `ValueId` fields (`tx`, `rx`, `channel`,
  `value`, `dst`, `task`, `args`). Nursery
  begin/end carry no values → pass through.
- `Insn::visit_ty_ids_mut` — covers `elem_ty` on
  ChannelCreate and `ty_id` on the four others.
  This is what lets the executor's post-execute
  `resolve_id` sweep (see CL16 generic-mono work)
  rewrite channel / task types to their resolved
  representatives before codegen reads them.

### validator extensions.

- `collect_value_types` — Phase 3 insns that
  produce typed values (`ChannelCreate`,
  `ChannelRecv`, `TaskSpawn`, `TaskAwait`)
  register their `dst`s so downstream `BinOp`s
  using those values get width-parity checks.
  ChannelCreate registers both `tx` and `rx`
  under the same `elem_ty` — the `Tx` / `Rx`
  asymmetry is enforced at method resolution,
  not the type layer.
- `check_insn` — new arms for every Phase 3
  variant, each running the standard placeholder
  check on its `ty_id` / `elem_ty`. Cross-insn
  invariants (send/recv elem-ty match,
  spawn arg-type parity against callee) are
  **deferred to Phase 4** where the executor is
  the emission point and has the channel / task
  definition on hand.

### pretty-printer.

`zo-pp/src/pp.rs` gets printable forms for the
new insns, matching the existing single-line
`%dst = op args : ty` style:

```
%0, %1 = chan.new cap=0 : TyId(8)
chan.send %0, %v : TyId(8)
%r = chan.recv %1 : TyId(8)
%t = task.spawn producer(%0) : TyId(30)
%u = task.await %t : TyId(1)
nursery.begin L1
nursery.end L1
```

### tests added.

Three tests in `zo-sir/src/validator.rs#tests`:

- `nursery_scoped_channel_spawn_await_is_clean` —
  a full nursery { channel + spawn + recv +
  await } stream validates with zero violations.
  End-to-end shape check.
- `channel_send_with_placeholder_ty_is_flagged` —
  `ChannelSend { ty_id: TyId(0) }` trips the
  standard placeholder check. Regression guard
  against a future copy-paste that forgets to
  run the check on new carriers.
- `channel_create_registers_both_halves_in_value_types` —
  a `BinOp` on `tx` and `rx` after a
  `ChannelCreate` passes width parity, proving
  both halves land in the value-types map with
  the same `elem_ty`.

### exhaustiveness sweep.

New variants forced three exhaustive-match sites
to handle them:

- `Insn::visit_value_ids_mut` (sir.rs)
- `Insn::visit_ty_ids_mut` (sir.rs)

## channels plan — Phase 4: executor.

Per `PLAN_CHANNELS.md` Phase 4. Four executor
surfaces wired up: nursery scoping, spawn /
await dispatch, the `channel()` built-in, and
the `tx.send` / `rx.recv` method-dispatch
arms. ~380 LoC across 1 crate + 3 new error
kinds + 6 unit tests.

### new executor state.

- `nursery_stack: Vec<(u32, usize)>` — one entry
  per lexically-active `nursery { }`. Tuple is
  `(label, rbrace_idx)`: label threads through
  `NurseryBegin` / `NurseryEnd`; rbrace_idx is
  the tree index that triggers the close.
- `pending_spawn: Option<(usize, usize)>` —
  `(spawn_node_idx, rparen_idx)` recorded at
  `execute_spawn` so the call-emit path can
  redirect `Call` → `TaskSpawn`.

### dispatch.

Three new arms in the main `execute_node` match
(after `Token::Return`):

- `Token::Nursery` → `execute_nursery` — emits
  `NurseryBegin { label }`, pushes the scope.
- `Token::Spawn` → `execute_spawn` — enforces
  Phase 0 decision 3 (hard error if no enclosing
  nursery), records pending_spawn. Scans the
  child range backwards for the RParen since
  the trailing `;` may or may not be a child
  of the Spawn introducer.
- `Token::Await` → `execute_await` — pops the
  top value, unwraps `Ty::Task(T)` → `T`, emits
  `TaskAwait`, pushes the unwrapped value back.

`Token::RBrace`'s dispatch grew one line:
`close_nursery_at_rbrace(idx)` runs before the
other block-end housekeeping so `NurseryEnd`
lands inside the block's SIR span.

### `channel()` built-in.

Intercepted in `execute_call` immediately after
the `show` / `showln` special case, before the
user-function lookup. `execute_channel_builtin`:

1. Parses the capacity via `extract_literal_capacity`:
   - `channel()` → capacity 0 (unbuffered).
   - `channel(Int)` → the integer value.
   - anything else → hard error
     `ChannelCapacityNotLiteral` per Phase 0
     decision 5, capacity defaults to 0.
2. Mints a fresh inference var `elem_ty`;
   concrete `T` flows in from the first `send`
   / `recv` unification.
3. Mints two `ValueId`s for `tx` / `rx`.
4. Emits `ChannelCreate { tx, rx, elem_ty, capacity }`.
5. Emits a `TupleLiteral` wrapping `tx, rx` so
   the surface destructure `imu (tx, rx) :=
   channel()` consumes a single tuple on the
   stack (per existing tuple-destructure path).

### `spawn` redirect.

At the main user-call emit site in `execute_call`,
a guard checks `matches!(self.pending_spawn,
Some((_, rp)) if rp == rparen_idx)`. If matched:

- `pending_spawn` cleared.
- `task_ty = task_type(resolved_ret)` —
  wraps the callee's return type in `Ty::Task`.
- `Insn::TaskSpawn { dst, callee, args, ty_id:
  task_ty }` emitted instead of `Insn::Call`.
- Task handle pushed onto stacks with
  `task_ty`.

The regular `Call` fallback is untouched, so
non-spawned calls keep their existing emit
path.

### `tx.send(v)` / `rx.recv()` method dispatch.

Two new arms in `execute_potential_call`, inserted
before the generic `resolve_dot_call` mangling
step. Logic:

- `name_str == "send"` and ty_stack[-2] resolves
  to `Ty::Channel(_)` → `execute_channel_send`.
  Pops the value arg (top) and receiver (next
  down), unifies the sent value's type with the
  channel's `elem_ty`, emits `Insn::ChannelSend
  { channel, value, ty_id: elem_ty }`.
- `name_str == "recv"` and ty_stack[-1] resolves
  to `Ty::Channel(_)` → `execute_channel_recv`.
  Pops the receiver, mints a dst, emits
  `Insn::ChannelRecv { dst, channel, ty_id:
  elem_ty }`, pushes the dst back with
  `elem_ty`.

Parallel to the existing `Ty::Array(_)` →
`push` / `pop` pattern two arms earlier. No
user-facing `apply Channel for T { ... }`
definition needed — it's a compiler intrinsic.

### new error kinds.

Added to `zo-error/src/error.rs`:

- `SpawnOutsideNursery` — `spawn` not lexically
  inside `nursery { }`.
- `AwaitOnNonTask` — `await expr` where expr
  isn't `Ty::Task(_)`.
- `ChannelCapacityNotLiteral` — `channel(N)`
  with non-literal N.

### tests added.

Six tests in `zo-executor/src/tests/concurrency.rs`,
asserting SIR shape via `assert_sir_structure`
predicates (structural counts, not byte-for-byte
equality):

- `nursery_empty_emits_paired_begin_end` — bare
  `nursery {}` emits one NurseryBegin + one
  NurseryEnd.
- `channel_builtin_emits_channel_create_and_tuple`
  — `channel()` produces ChannelCreate + the
  wrapping TupleLiteral.
- `channel_with_explicit_literal_capacity` —
  `channel(4)` lands `capacity == 4`.
- `spawn_outside_nursery_is_error` — asserts
  `ErrorKind::SpawnOutsideNursery` diagnostic.
- `spawn_inside_nursery_emits_task_spawn_not_call`
  — the spawned call lands as TaskSpawn with
  zero regular Calls emitted.
- `channel_capacity_variable_is_error` — asserts
  `ErrorKind::ChannelCapacityNotLiteral` for
  `channel(BUF)` with a val ref.

### known gap — tuple-destructure for (tx, rx) := channel().

The `imu (tx, rx) := channel();` surface form
goes through the existing tuple-destructure
finalize path in `finalize_pending_decl`. That
path currently propagates `TyId::unit` to each
binding (extracting via `TupleIndex` with a
placeholder `ValueId::MAX` as the tuple operand)
instead of the tuple's element type
`Ty::Channel(T)`.

The send / recv dispatch arms in
`execute_potential_call` ARE correctly wired —
they just can't be reached from a pure-source
program yet because `tx` lands with Unit type,
which fails the `Ty::Channel(_)` guard. Phase 4b
owns wiring the destructure so element types
flow through.

A test `channel_send_recv_emit_paired_ops` was
drafted, demonstrated the gap, and replaced with
a doc-comment tracking the follow-up work —
shipping a failing test would misrepresent the
state of the dispatch code.
