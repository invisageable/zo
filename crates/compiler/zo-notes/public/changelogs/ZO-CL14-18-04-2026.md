## feature — `match` ident-pattern binding + guard arms.

```zo
val X: int = 10;

fun main() {
  match X {
    5 => check(false),
    num if num == (3 * 3 + 1) => check(true),
    _ => check(false),
  }
}
```

`branch-match-val.zo` was red in two different ways at
once. Owning them together because the same
`execute_match` path dispatches both.

### 1. bare-ident pattern (`num =>`) didn't bind.

Before the fix, an arm whose pattern was a plain
ident (not `_`, not `EnumType::Variant(...)`, not a
tuple literal, not a scalar literal) fell through
every existing branch of the pattern dispatcher and
compiled as an unconditional match with NO binding.
The arm body then referenced `num` and got
`Undefined variable` at compile time.

**fix.** Add an `else if !is_wildcard && !is_enum_pat
&& pat_tok == Token::Ident` arm that:

1. Loads the scrutinee via the existing
   `scrutinee_sym` / `__match_scrut__` path.
2. Emits `Store { name: bind_sym, value: scrutinee,
   ty_id: scrutinee_ty }`.
3. Pushes `self.locals` with the ident's name pointing
   at the scrutinee's SIR value.
4. Increments `arm_bindings` so the local gets popped
   at arm end (already-existing enum-binding
   infrastructure).

Mirrors how enum variant payload fields bind their
captured values.

### 2. guard arms (`if expr =>`) compiled then SIGILL'd.

The parser represents `num if num == 10 => body` as
an `If` sub-node between the pattern ident and the
`FatArrow`. Its children carry the guard's postorder
expression tokens up to the `FatArrow`. The executor
had no handler for this, so guards compiled as if the
arm were always-match, then crashed at runtime.

Two compounding bugs:

**2a — guard never lowered.** No sub-walk of the `If`
node's tokens, no `BranchIfNot` emitted for the guard
condition. Even once ident-binding worked, the guard
was just ignored.

**2b — `execute_match`'s `skip_until` over-reached.**
The handler set `self.skip_until = end_idx` (the
Match node's child-span end). For guard arms the
parser inflates the child-count (the `If` sub-node
pulls its own subtree into Match's sibling listing),
so `end_idx` reached PAST the match's own `}` into
the enclosing block's `}`. Main never saw its closing
RBrace, no function-epilogue `Return` emitted,
codegen produced a `ret`-less `@main`, and the binary
SIGILL'd (exit 132). Same failure signature as the
fizzbuzz `=>` form from CL13 — "correct output, then
crash."

**fix.** After locating the match's own `}` at
`rbrace_idx`, tighten
`self.skip_until = rbrace_idx + 1`. Outer siblings
(the enclosing block's `}`, the next statement, …)
are now processed by the main loop as they should be.

**guard lowering.** In the new Ident-pattern branch,
if `tree.nodes[pat_idx + 1].token == Token::If` we
sub-walk `(if_idx + 1)..arrow_idx` through
`execute_node`, pop the resulting top-of-stack
boolean, and emit `BranchIfNot { cond: guard_sir,
target: next_arm_label }`. Drain any extras from the
value stacks so state is clean for the arm body.

### lessons baked in.

29. **Pattern-dispatch fall-through is dangerous.**
    Any match-arm dispatcher whose final branch is
    "no explicit handler → unconditional match" will
    happily accept new syntactic forms (bare idents,
    guards) as always-true arms with broken bindings.
    The guard against this is an explicit arm for
    every pattern token the parser emits, with a
    diagnostic (not a silent fallback) for anything
    unrecognized.

30. **`skip_until` must point one-past the OWN closing
    token, not the parser's child-span end.** Parser
    child-spans can inflate beyond the construct's
    lexical extent when sub-nodes (guards' `If`,
    blocks' nested `If`/`Match`, …) are listed as
    siblings. A handler that skips by child-span end
    will swallow outer tokens. Same lesson as CL13's
    fizzbuzz `=>` form, different site: always locate
    the construct's own terminator and skip to
    `terminator + 1`.
