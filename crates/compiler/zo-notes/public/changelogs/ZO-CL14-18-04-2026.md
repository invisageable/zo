## feature — `match` ident-pattern binding + guard arms.

```zo
val X: int = 10;

fun main() {
  match X {
    5 => check(false),
    num if num == (3 * 3 + 1) => check(true),
    _ => check(false),
  }
}
```

`branch-match-val.zo` was red in two different ways at
once. Owning them together because the same
`execute_match` path dispatches both.

### 1. bare-ident pattern (`num =>`) didn't bind.

Before the fix, an arm whose pattern was a plain
ident (not `_`, not `EnumType::Variant(...)`, not a
tuple literal, not a scalar literal) fell through
every existing branch of the pattern dispatcher and
compiled as an unconditional match with NO binding.
The arm body then referenced `num` and got
`Undefined variable` at compile time.

**fix.** Add an `else if !is_wildcard && !is_enum_pat
&& pat_tok == Token::Ident` arm that:

1. Loads the scrutinee via the existing
   `scrutinee_sym` / `__match_scrut__` path.
2. Emits `Store { name: bind_sym, value: scrutinee,
   ty_id: scrutinee_ty }`.
3. Pushes `self.locals` with the ident's name pointing
   at the scrutinee's SIR value.
4. Increments `arm_bindings` so the local gets popped
   at arm end (already-existing enum-binding
   infrastructure).

Mirrors how enum variant payload fields bind their
captured values.

### 2. guard arms (`if expr =>`) compiled then SIGILL'd.

The parser represents `num if num == 10 => body` as
an `If` sub-node between the pattern ident and the
`FatArrow`. Its children carry the guard's postorder
expression tokens up to the `FatArrow`. The executor
had no handler for this, so guards compiled as if the
arm were always-match, then crashed at runtime.

Two compounding bugs:

**2a — guard never lowered.** No sub-walk of the `If`
node's tokens, no `BranchIfNot` emitted for the guard
condition. Even once ident-binding worked, the guard
was just ignored.

**2b — `execute_match`'s `skip_until` over-reached.**
The handler set `self.skip_until = end_idx` (the
Match node's child-span end). For guard arms the
parser inflates the child-count (the `If` sub-node
pulls its own subtree into Match's sibling listing),
so `end_idx` reached PAST the match's own `}` into
the enclosing block's `}`. Main never saw its closing
RBrace, no function-epilogue `Return` emitted,
codegen produced a `ret`-less `@main`, and the binary
SIGILL'd (exit 132). Same failure signature as the
fizzbuzz `=>` form from CL13 — "correct output, then
crash."

**fix.** After locating the match's own `}` at
`rbrace_idx`, tighten
`self.skip_until = rbrace_idx + 1`. Outer siblings
(the enclosing block's `}`, the next statement, …)
are now processed by the main loop as they should be.

**guard lowering.** In the new Ident-pattern branch,
if `tree.nodes[pat_idx + 1].token == Token::If` we
sub-walk `(if_idx + 1)..arrow_idx` through
`execute_node`, pop the resulting top-of-stack
boolean, and emit `BranchIfNot { cond: guard_sir,
target: next_arm_label }`. Drain any extras from the
value stacks so state is clean for the arm body.

### lessons baked in.

29. **Pattern-dispatch fall-through is dangerous.**
    Any match-arm dispatcher whose final branch is
    "no explicit handler → unconditional match" will
    happily accept new syntactic forms (bare idents,
    guards) as always-true arms with broken bindings.
    The guard against this is an explicit arm for
    every pattern token the parser emits, with a
    diagnostic (not a silent fallback) for anything
    unrecognized.

30. **`skip_until` must point one-past the OWN closing
    token, not the parser's child-span end.** Parser
    child-spans can inflate beyond the construct's
    lexical extent when sub-nodes (guards' `If`,
    blocks' nested `If`/`Match`, …) are listed as
    siblings. A handler that skips by child-span end
    will swallow outer tokens. Same lesson as CL13's
    fizzbuzz `=>` form, different site: always locate
    the construct's own terminator and skip to
    `terminator + 1`.

## feature — nested `pack` + dotted call dispatch.

```zo
pack inner {
  pack inner2 {
    fun hello() {
      showln("hello, modular world");
    }
  }

  fun hello() {
    inner2.hello();
  }
}

fun main() {
  inner.hello();
  inner.inner2.hello();
}
```

`pack-inner.zo` was blocked on three separate issues.
`execute_pack` used the parser's child-span `end_idx` as
its body boundary (fine for empty/flat packs), pack-
prefix idents (`inner`, `inner2`) tripped the Ident
handler's `UndefinedVariable` path (they are namespaces,
not variables), and there was no dot-call resolver for
the pack case (`inner.hello()` fell through to the
struct/enum method path, which returned the unmangled
`hello` and silently produced a no-op).

### 1. child-span inflation swallows `fun main`.

`pack inner { ... } fun main() { ... }`: the parser
lists `fun main` among Pack's children, so `end_idx`
reaches past the pack's own `}` into the next top-level
item. `execute_pack` used to `skip_until = end_idx`,
which both (a) swallowed `fun main` and (b) — if we
instead iterate the body via `end_idx` — *ran* `fun main`
inside `pack_context`, mangling it to `inner::main` and
leaving the binary with no entry point (hang on exec).

**fix.** Depth-count `{}` from the pack's own `LBrace`
to find its matching `RBrace`; use `rbrace_idx + 1` as
both the body-iteration boundary and `skip_until`.
Same lesson class as CL13's match / for fixes:
*always locate the construct's own terminator and skip
to `terminator + 1`*.

### 2. `Ident(inner)` flagged `UndefinedVariable`.

With the body boundary fixed, `pack inner` registered
correctly — but `main`'s `inner.hello()` failed at the
Ident handler. `inner` has no local, no global, no fun
entry, no enum / struct — so it fell into the
`UndefinedVariable` branch.

**fix.** Track every declared pack's simple name in
`pack_names: HashSet<Symbol>`. In the Ident handler,
treat a hit on `pack_names` the same as `is_dot_member`:
push a placeholder onto the stacks so the `Dot`
handler / call site see balanced shapes, and let
resolution happen at RParen.

### 3. dot-chain resolution.

The existing `resolve_dot_call` paths are **type-based**:
they read the receiver's type, pull the type's name, and
mangle `Type::method`. Pack chains don't have a receiver
type (the idents are namespaces), so that path returns
`method_name` unchanged and we ended up emitting a call
to the non-existent `hello`.

**fix — `resolve_pack_dotted_call(lparen_idx)`.** Walk
backward through the postfix chain (`a b . c . d .` →
`[a, b, c, d]`) via `collect_dot_chain` / `pack_chain_root`.
If the root is in `pack_names` AND the joined `a::b::c::d`
resolves to a declared fun, return the mangled symbol.
Otherwise try prefixing with the enclosing `pack_context`
(so `inner2.hello()` inside `pack inner` resolves to
`inner::inner2::hello`). Wired into:

- `Token::LParen`: `is_call |= resolve_pack_dotted_call(idx).is_some()` so `direct_call_depth` bumps and tuple-ctx is skipped.
- `Token::Dot`: new `is_pack_chain_dot(idx)` branch BEFORE `is_dot_method_call`. Pops both placeholders (no implicit `self` for pack calls), pushes one folded placeholder for intermediate dots or zero for the leaf dot preceding `(`.
- `rparen_closes_call`: also true when pack-dotted.
- `execute_potential_call`: first branch — if `resolve_pack_dotted_call(lparen_idx)` hits, dispatch through `execute_call(mangled, ...)` and return.

### 4. `execute_pack` owns body traversal.

To mangle nested funs as `outer::inner::name` without
relying on RBrace to pop, `execute_pack` now mirrors
`execute_apply`'s shape: emit `PackDecl`, push
`pack_context`, advance past the name+`{`, iterate the
body via `execute_node`, pop, set
`skip_until = rbrace_idx + 1`.

### lessons baked in.

31. **Parser child-spans are advisory, not lexical.**
    A construct's `end_idx` from `header.child_count`
    may reach past its own `}` when the parser groups
    sibling items underneath. Any handler that iterates
    "until `end_idx`" (or sets `skip_until = end_idx`)
    will silently pull in the next top-level item. The
    fix is always the same: depth-count from the
    construct's own `{` to find the matching `}`, and
    use `rbrace_idx + 1` as the boundary. Third site
    fixed this week (match arms at CL13's `=>` form,
    match guards at CL14 start, now pack bodies).

32. **Namespace-prefix idents need their own handling
    branch.** Stock Ident handling has five exits:
    local / global / fun / enum-or-struct /
    UndefinedVariable. Adding namespace prefixes
    (`pack`, future `module`) means a sixth: push a
    placeholder, let the RParen call path do the
    resolution. Treating namespace prefixes as "almost a
    field access" (is_dot_member) is the right shape —
    they don't carry a value, and the Dot handler owns
    the chain walk anyway.

33. **Dot-call resolution has two flavors.** Type-based
    (`receiver_value . method_ident`) and namespace-based
    (`pack_ident . fun_ident` — or longer). Bolting the
    second onto the first is a distraction; adding a
    parallel `resolve_pack_dotted_call` that owns its
    own chain walk and mangle logic keeps each path
    small and readable. The Dot handler dispatches to
    whichever matches via `is_pack_chain_dot` before
    falling back to `is_dot_method_call`.

## feature — `apply <Primitive>` (PR 1: `char`).

```zo
apply char {
  fun is_zero(self) -> bool {
    return self == '0';
  }
}

fun main() {
  imu c: char = '0';
  check(c.is_zero());
}
```

First slice of method dispatch on primitive receivers. Previously
`apply char { … }` parsed past the `apply` keyword but both the
parser and executor rejected the primitive type name: the parser
errored "expected identifier", and once that was widened the
executor's `execute_apply` scan missed `Token::CharType` so
`apply_context` never got set, `self`'s type fell back to `unit`,
and `self == '0'` errored as a type mismatch. Three sites fixed
to restore the full dispatch path (definition → mangled fn entry
→ call-site resolution).

### 1. parser accepts primitive type keywords after `apply`.

`zo-parser/src/parser.rs:1260` `handle_apply_keyword` used to
reject anything that wasn't `Token::Ident` immediately after
`apply`. Widened to accept any token whose `ty_keyword_str()` is
`Some` (`char`, `str`, `int`, `float`, `bool`, `uint`, the
sized variants). The single-scan predicate keeps the grammar
change to one line and avoids a `matches!` list that would
drift from `Token::ty_keyword_str`.

### 2. executor widens type-name extraction.

`zo-executor/src/executor.rs:7238` `execute_apply` previously did
`(start_idx+1..end_idx).find(|i| tree.nodes[i].token == Ident)`.
Now the `find` predicate is `Ident || is_primitive_type_token`,
and the name is synthesized differently per branch:

- `Ident`: read `NodeValue::Symbol` as before.
- primitive keyword: `interner.intern(tok.ty_keyword_str())`
  so the interner symbol matches the one
  `TyChecker::resolve_ty_symbol` uses for the lookup — the two
  paths have to key off the same string.

Same widening applied to the `For <Target>` rescan (line 7289)
so `apply Abstract for char { … }` is shaped identically.

### 3. `self`-param typing uses the primitive-aware resolver.

`self.ty_checker.resolve_ty_name(type_name)` only checks
aliases/structs/enums — returns `None` for primitives. Swapped
to `resolve_ty_symbol(type_name, self.interner)` at
`executor.rs:4952`, which does the string match
(`"char" → Ty::Char`, etc.) and internally falls through to
`resolve_ty_name` for non-primitives, so struct/enum applies
keep working unchanged.

### 4. `is_dot_method_call` recognises primitive receivers.

`executor.rs:7651` used to extract the mangling type name only
from `Ty::Struct(sid)` / `Ty::Enum(eid)`, returning `None` for
anything else and short-circuiting to "not a method call".
Extended: when neither struct nor enum, call
`primitive_ty_name_str(&resolved)` (new helper returning
`Ty::Char → "char"`), intern it, and proceed with the normal
mangled-name lookup. `funs.iter().any(...)` at line ~7670
remains the guard — if the mangled symbol isn't a real
function, the call still bails. No regression on struct/enum.

### 5. single-source helpers.

Two new private fns on `Executor<'a>`:

```rust
fn is_primitive_type_token(tok: Token) -> bool {
  matches!(tok, Token::CharType) // PR 2 extends this list.
}

fn primitive_ty_name_str(ty: &Ty) -> Option<&'static str> {
  match ty {
    Ty::Char => Some("char"), // PR 2 extends this list.
    _ => None,
  }
}
```

Keeping both as single-match tables means PR 2 is a two-line
change on each. The alternative — scattering the primitive
list inline at every call site — would drift.

### known limitation (follow-up, not PR 1 scope).

`method() == false` type-checks as "incompatible types" when
the method returns `bool`. Same for `!method()`. Reproducible

### lessons baked in.

36. **Parser and executor token-acceptance lists must stay in
    sync.** `handle_apply_keyword`'s predicate and
    `execute_apply`'s scan filter are a matched pair — if one
    accepts a token the other rejects, you get a silent
    "feature half-works" regression. Long-term, derive both
    from `Token::ty_keyword_str` (one source of truth);
    short-term, change both together and add a test that
    exercises a shape the parser used to reject.

37. **`resolve_ty_name` vs `resolve_ty_symbol` — prefer the
    latter in any context where a user-writable name might
    refer to a primitive.** `resolve_ty_name` is narrower
    (alias + struct + enum) and exists for places where a
    primitive would be an error (e.g. struct field type
    lookup is through `resolve_ty_token` anyway).
    `resolve_ty_symbol` does the full string table AND falls
    through to `resolve_ty_name` — cost is one string
    compare, benefit is no hidden failure for primitives.

2. `zo-compiler/src/compiler.rs` `preload` whitelist — added `"char"` so symbols land in user scope without an explicit `load`.

### discovered during this PR.

- **`pub` is mandatory for apply methods to export.**
  `extract_exports` (`zo-module-resolver/src/exports.rs:86`) filters out
  any `FunDef` where `pubness != Yes`. Non-`pub` methods land in
  the std pack's own SIR but don't cross into user scope; the call
  site then falls back to `Ty::Struct/Enum` handling, which returns
  `None` for `Ty::Char`, which shows up as `Dot` emitting a `field`
  insn (tuple/struct field access) instead of a `Call`. User
  message is "type mismatch" at the call expression.
  Lesson 38 below.

- **Semicolon `pack X;` form is a file-level assertion, not a
  mangling prefix.** User correctly flagged the concern that
  `pack char { apply char }` would produce `char::char::is_digit`.
  Traced through `execute_pack`: the no-body form pushes
  `pack_context` AND synchronously pops it (no `lbrace_idx →
  pack_end = end_idx` degenerate case), so mangling stays clean as
  `char::is_digit`. Nested-body form would collide — avoid for
  primitive applies.

### known issues — tracked for follow-up, not PR 3 scope.

A. **`!receiver.method()`** — unary `!` on a bool-returning method
   errors as "incompatible types for this operation". The method's
   return bool isn't being pushed onto `ty_stack` in a shape the
   unary handler expects. Reproduces on primitive-apply AND
   struct-apply, so this is a pre-existing gap, not regression.

B. **`method() == false`** (mirror of A on the binop side).
   Same root cause — call return bool doesn't propagate into the
   binop context.

C. **`'literal'.method()`** — method call on a `char` / `str` / int
   literal errors. The literal's Ty reaches `is_dot_method_call`
   but something in the receiver-value push path leaves the stack
   in a shape that blocks mangling. Reproducible with
   `'x'.to_digit()`. Works fine through a variable: `imu c: char = 'x'; c.to_digit()`.

D. **`apply float` body with float literal comparison** — `self == 0.0`
   where `self: f32` (from `apply float`) and `0.0: f64` (literal
   default) is rejected as type mismatch. Orthogonal to
   primitive-apply; tracked here so PR 4 (other numeric inherent
   methods) picks it up.

### lessons baked in.

38. **Pack-exported methods need `pub`.** `extract_exports` only
    carries `FunDef { pubness: Yes }` across into user scope. The
    failure mode is silent: the method lives in the pack's
    internal SIR, the call site falls back to
    `is_dot_method_call`'s None branch, and the user sees a
    `Dot` field-access error instead of a method-not-found
    message. Policy: every `fun` inside an `apply` block in a
    std pack is `pub fun` unless deliberately private. Lint
    candidate.

39. **Semicolon-form packs don't nest symbols.** `pack X;`
    (no body) is a file-level assertion — the executor pushes
    and synchronously pops `pack_context`, so any `apply` /
    `fun` inside the file mangles as if there were no pack.
    `pack X { body }` (with body) DOES nest. For inherent
    primitive methods, the semicolon form is the right choice
    — no double-prefix, no `char::char::is_digit` collision
    with the canonical `char::is_digit` that the call site
    expects.

## bugfix — three compiler bugs unblocking std expansion.

Three latent bugs surfaced when trying to ship `std/int.zo`
and `std/bool.zo` alongside the already-working `std/char.zo`.
Each had a distinct root cause; each is fixed at the site.

### A. parser — unary `!` binds looser than dot-call.

```zo
check(!f.is_set());  -- `!(f.is_set())`, not `(!f).is_set()`
```

`zo-parser/src/parser.rs:1674` — the unary-stash drain fired
after any operand whose peek was not `LParen`/`LBracket`.
When peek was `Dot` (method chain to follow), `Bang` drained
immediately, landing BEFORE the method-dispatch chain in
postfix order. Executor then popped `Bang`'s operand as `f`
(struct type, not bool) and errored "incompatible types for
unary".

Fix: add `Token::Dot` to the exclusion so unary stays
stashed until the chain completes. One-line change.

### F. parser — `last_was_value` not reset across empty flushes.

```zo
fun f() -> int {
  if false {
    return (1);
  }
  return -1;              -- expected: -1; actual: 0
}
```

The trailing `)` of `return (1);` sets `last_was_value = true`.
Then `;` calls `flush_expr` with an empty buffer, which
early-returned WITHOUT resetting `last_was_value`. At the
subsequent `return -1;`, the parser's `is_unary_context()`
read the stale `last_was_value = true` and decided `-` was
BINARY. Tree captured `Int(1) Minus` (no preceding operand
to pair with) — executor produced the empty else-branch that
DCE later removed as dead. Runtime: wrong value.

Fix: `flush_expr` always resets `last_was_value = false`,
regardless of whether the buffer was empty. A statement
boundary ends the "just produced a value" context —
that's the whole point of flushing.

### G. preload — multi-pack SIR value-id + label collisions.

Each pack's SIR starts its own `next_value_id` and
`next_label_id` at 0. The preload loop originally applied
ONE shared offset (`offset_value_ids(&mut module_sir, main_next_vid)`)
to the concatenated module block — pack A's `%0` and pack
B's `%0` both mapped to `%main_next_vid`. Same class of
collision on labels (`Label { id: 0 }` → two packs' labels
merged to one, subsequent `Jump { target: 0 }` / `BranchIfNot
{ target: 0 }` landed on the earliest occurrence).

Only visible when TWO OR MORE packs with non-trivial SIR
(i.e. bodies with control flow / arithmetic) were preloaded
together. With just `char`, no collision. With `char + int +
bool`, calls silently miscomputed — `42.is_positive()`
returning `false` is the symptom that pinned it.

Fix: per-pack rebasing BEFORE extending the module block.
- New `Sir::offset_labels(&mut [Insn], offset)` helper
  parallel to the existing `offset_value_ids`.
- `ModuleExports.next_label_id` field (carried through
  `extract_exports` alongside `next_value_id`).
- In the preload loop: `offset_value_ids(pack, module_next_vid)`
  + `offset_labels(pack, module_next_lid)` before extend;
  bump both running bases by the pack's counters.
- Same in the explicit-`load` / `module_table` paths.
- Final lift of module labels by `main_next_lid` alongside
  the existing value-id lift.

|------|--------|
| `zo-parser/src/parser.rs` | (A) `Dot` excluded from unary drain; (F) `flush_expr` always resets `last_was_value` |
| `zo-parser/src/tests/unary.rs` | `test_unary_not_binds_looser_than_dot_call` |
| `zo-sir/src/sir.rs` | new `Sir::offset_labels` helper |

### lessons baked in.

40. **Every SIR counter that starts at 0 per-module needs
    its own offset primitive at merge time.** Value-ids had
    `offset_value_ids` from day one; labels didn't, and the
    resulting silent wrong-code only surfaced once enough
    packs had bodies with control flow to trigger a real
    collision. Rule of thumb: if it's a `next_X_id: u32` on
    `Sir`, it needs an `offset_X` helper AND a `next_X_id`
    field on `ModuleExports` before a pack can merge cleanly.

41. **State that means "continue binary-parse mode" must be
    reset at every statement boundary.** `last_was_value`,
    `operator_stack`, `expr_buffer`, `unary_spans` — all of
    these are "mid-expression" state that the next statement
    must start fresh from. Early-returning out of `flush_expr`
    for perf (empty buffer → nothing to flush) is fine for
    the flush itself, but the `last_was_value = false` reset
    was on the wrong side of the early-return. The reset
    belongs UNCONDITIONALLY at the top.

42. **Don't follow "probably a workaround" offers.**
    Mid-diagnosis, I suggested switching `return -1;` to
    `return 0 - 1;` as a workaround. That was wrong — the
    user correctly rejected it. Working around parser bugs
    by changing user code is how compilers accumulate
    footguns. Fix the parser, not the std.

## bugfix — if-as-expression φ-sink wired (issue F).

```zo
fun abs_expr(n: int) -> int {
  if n < 0 {
    0 - n
  } else {
    n
  }
}
-- abs_expr(0 - 7): expected 7, actual was 1.
-- abs_expr(5):      expected 5, actual was 5 (lucky).
```

The `mint_branch_sink_if_expr_position` scaffolding has
lived in `execute_if` (`executor.rs:8304`) since CL13 but
the `Store` / `Load` call sites that actually USE the sink
were never added. The Ternary path at `Token::Colon` + the
Semicolon closer has them. The If path had neither. Result:
each arm's computed value dangled on the stack above the
`Jump` / `end_label`, the register allocator reclaimed the
slot, and the merge-point `Return` grabbed whatever was
left — often a stale ConstInt from the function's prologue.

Fix: store-before-jump and load-after-merge, symmetric to
Ternary.

### edits.

| site | change |
|------|--------|
| `executor.rs` `Token::Else` (~L951) | `emit_branch_sink_store(ctx_idx)` BEFORE `Jump { target: end_label }` |
| `executor.rs` `BranchKind::If` RBrace closer (~L1572) | `emit_branch_sink_store` before `end_label`; `emit_branch_sink_load(&popped)` after. Covers both the no-else path AND the else-body RBrace (else_label already taken by `Token::Else`, the remaining emit is just `end_label`) |
| `zo-executor/src/tests/control_flow.rs` `test_if_else` | SIR stream updated to match the new Store/Load shape — previously asserted the broken form |
| `zo-executor/src/tests/control_flow.rs` `test_if_expression_emits_branch_sink_stores` | NEW — regression guard that asserts ≥2 Stores + ≥1 Load in the if-else-expression SIR |

### verified with.

- `abs_expr(-7) == 7` ✓ (was `1`).
- `abs_expr(5) == 5` ✓.
- Ternary `when cond ? a : b` still produces identical SIR (unchanged path). ✓
- 882/882 `zo_test`, 288/288 `zo_test_runner`, zero clippy, zero typos.

### lessons baked in.

43. **Scaffolding without call sites is a ticking bug.**
    `mint_branch_sink_if_expr_position` was doing half the
    work — allocating the sink, recording it on the
    BranchCtx — but the pair of operations it enables
    (`Store` on each arm, `Load` at the merge) was only
    wired for Ternary. The compile passed, the SIR emitted,
    the sink symbol was registered and never used, and the
    bug hid behind "sometimes the ConstInt in the register
    happens to be right". Policy: every sink / scratch slot
    minting helper should have an adjacent usage assertion
    in the same PR, or a unit test asserting Store/Load
    insns appear in the expected pattern. Otherwise, the
    scaffolding claims a feature it doesn't actually deliver.
