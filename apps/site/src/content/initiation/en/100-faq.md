## comptime

**How does zo achieve sub-second compilation speeds?**

zo is built using modern compiler techniques, with less compiler stages zo can focus of what's matter to make a program safe. Otherwise zo do not do less work than other compilers does.

Most of the stage are hand-writing to get the control of the speed. zo use his own Mach-O linker and assembler. The compiler generates nativela executable and cryptographically signed binaries directly. This eliminates the multi-second startup and linkage bottleneck of standard external system linkers (like `ld` or `lld`), delivering compile times of 1.8 for 500K lines of code.

> *Keep in mind that the 500K example do not reflect a real codebase. It is used to tests how many lines of code zo can compiles. For real numbers, go read our [@benchmark-results](https://github.com/invisageable/zo/tree/main/crates/compiler/zo-benches).*

## memory management

**If there is no garbage collector (GC), and no borrow checker, what is the memory management model?**

zo uses an explicit, manual resource-release model. Just like in C or Zig, resources like dynamic vectors (`Vec`), hash maps (`HashMap`), regular expressions (`Regex`), and dynamic user interfaces (`any Abstract`) must be explicitly release by the developer calling `.free()`.

> *In the future, we plan to adopt `defer`, but we are not sure yet. I still a design decision.*

## templating

**How does zo compile a single Ui codebase to native widgets?**

zsx (zo Syntax Extension) are translated by the compiler's parser into a flat, platform-agnostic stream of user-interface instructions (`UiCommand`s).

At compilation, these instructions are compiled directly into native platform view instantiations (`UIButton` on iOS, `android.widget.Button` on Android, `<button>` on web and `Button::new` on desktop).

To ensure layout consistency acress different platforms, the runtime uses `taffy` (a W3C Flexbox specification) to calculate the exact geometric coordinates of each element. You get 100% native platform fidelity (VoiceOver accessibility, native keyboards, and native text rendering) from a single codebase.

> *Gates are opens but this feature is still in development. Because unifying different platforms is not an easy part.*

**How do I compile to iOS or Android?**

Just use the `--target=platform` flag in your command line: `zo build <pathname> --target ios`. Keep in mind that you must use the `#render` directive. This is required to tell the compiler to enable a specific runtime.

**How do I choose the Simulator device?**

`zo run <pathname> --target ios` picks a device automatically: the booted one if any, otherwise the newest iPhone on your machine. To run on a specific device, pass `--device` with any device name (or UDID) from your Simulator:

```sh
zo run app.zo --target ios --device "iPhone 17 Pro"
zo run app.zo --target ios --device "iPad (A16)"
zo run app.zo --target ios --device "Apple Vision Pro"
```

iPhone, iPad, and Apple Vision Pro simulators can all run the same `ios` build — visionOS runs it through its iOS app-compatibility layer. The Apple Watch runs its own platform build: use `--target watchos` and zo compiles, bundles, and launches the same source on a watch simulator (auto-selected, or named with `--device`):

```sh
zo run app.zo --target watchos
zo run app.zo --target watchos --device "Apple Watch Ultra 3 (49mm)"
```

If a device name does not match the target platform, the error lists every device on your machine able to run the app.

## comparison

**How does zo compare to Rust?**

zo is faster at build time, if you care about your feedback loop, zo can definetely be an alternative choice. Also regarding the friction, zo adopt simplicity — no borrow checker, no lifetime, no function coloring. Instead, zo has a concurrent system à la Erlang, Go or Swift.

In terms of runtime, Rust is faster than zo. We planned to improve our runtime performance to be at least close to C (clang).

  - @SEE [@zo-benchmark](https://github.com/invisageable/zo/blob/main/crates/compiler/zo-benches)

**How does zo compare to dioxus for Ui?**

Dioxus is a fullstack app framework for web, desktop, and mobile. It uses web technologies for buildings user interfaces. Some concrete differences:

  - **WASM vs Optimized JS**: Even if Dioxus produce well optimized WASM, it can beat JS bundle size. zo is typed-safe so runtime errors are prohibed. By producing a JS bundle, you get the same performance than any JS frameworks — without the bridge gap between WASM+JS. WASM is fine to perform heavy computation or provide a for a plugin system. Regarding the web, pure JS is the way to go. 
    - @SEE [@js-framework-benchmark](https://krausest.github.io/js-framework-benchmark/current.html)

  - **Native vs Webview or Custom Renderer**: Dioxus use webview on desktop and mobile missing native features (native keyboard, ...). On the other hands, zo produce native code, your app is fully native, no bridges, no missing features on Android or iOS.
